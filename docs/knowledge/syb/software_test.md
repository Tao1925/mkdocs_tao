## 二、软件测试是什么

### 1. 软件测试的定义

软件测试，简单来说，就是通过一系列活动来检查软件是否满足预期要求、是否存在缺陷、是否能够在目标环境中稳定运行。

这里面有三个核心点：

- 测试不是随便点一遍功能，而是有目标、有方法、有结果记录的验证活动。
- 测试不只是找程序报错，还包括验证需求、检查边界、识别风险和评估质量。
- 测试既可以人工执行，也可以借助工具自动执行。

如果用更工程化的话来说，软件测试的本质是通过设计输入、执行程序、观察输出和分析结果，来判断软件是否符合预期，并尽早暴露潜在问题。

### 2. 软件测试与调试、质量保证、质量控制的关系

这些概念经常一起出现，但不能混为一谈。

#### 软件测试

软件测试关注的是发现问题、验证行为和评估风险。

#### 调试

调试是在发现问题之后，定位问题原因并修复问题的过程。

可以简单理解为：

- 测试负责发现问题。
- 调试负责修复问题。

#### 质量保证

质量保证，通常简称 QA，更强调建立流程、规范和制度，确保团队持续地产出高质量软件。

例如：

- 制定开发流程
- 规范评审机制
- 推动测试标准
- 监督质量门禁

#### 质量控制

质量控制更偏向具体执行层面，例如检查某个版本是否达到上线标准。

可以把它理解为更靠近“实际检查和判定”的活动。

### 3. 软件测试想解决什么问题

软件开发过程中，缺陷几乎不可避免。原因可能来自：

- 需求理解错误
- 设计不合理
- 编码错误
- 环境差异
- 接口依赖变化
- 并发、性能、边界条件未考虑周全

软件测试的目的，不是追求形式上的“测试过了”，而是尽可能提前发现问题，降低缺陷流入生产环境的概率，并帮助团队理解系统当前的真实质量水平。

## 三、为什么需要软件测试

### 1. 软件缺陷不可避免

软件系统往往具有复杂性。一个看似简单的功能，在真实环境中可能要面对：

- 多种输入组合
- 多个依赖系统
- 多种设备和浏览器
- 多用户并发访问
- 异常网络和不稳定数据

代码越多、依赖越多、场景越复杂，出现问题的概率就越高。因此，测试不是“可有可无的附加项”，而是软件工程中的基础活动。

### 2. 测试对质量、成本和风险控制的意义

测试的价值，不只是发现 Bug，更在于控制成本和风险。

通常来说，缺陷发现得越晚，修复成本越高：

- 需求阶段发现问题，改文档的成本较低。
- 设计阶段发现问题，改方案的成本中等。
- 开发完成后发现问题，可能要改代码、重测、重发版。
- 上线后才发现问题，可能会影响用户、业务和品牌信誉。

所以，测试越早介入，通常越有价值。

### 3. 软件测试适用于哪些项目阶段

很多人会误以为测试只发生在“开发结束之后”。实际上，测试思想应该贯穿整个软件生命周期：

- 需求阶段：检查需求是否清晰、完整、可测试。
- 设计阶段：检查架构设计和接口设计是否合理。
- 编码阶段：编写单元测试、静态检查代码。
- 集成阶段：验证模块之间协作是否正确。
- 发布阶段：确认系统功能、性能和风险是否满足上线条件。
- 运行阶段：监控生产环境，验证真实运行质量。

## 四、软件测试的核心目标

### 1. 发现缺陷

这是最直观的目标。通过测试，尽早发现功能错误、边界问题、兼容性问题、性能问题和安全问题。

### 2. 降低风险

测试不可能把所有问题都找出来，但可以显著降低高风险问题进入生产环境的概率。

例如：

- 支付金额计算错误
- 登录鉴权失效
- 高并发时系统崩溃
- 数据删除无法恢复

### 3. 验证需求

测试不是只看“程序能不能运行”，还要看“程序是不是按需求运行”。

如果需求说“未登录用户不能查看个人订单”，那么系统即使页面能打开，也不代表是正确的。

### 4. 建立质量信心

测试的结果可以帮助团队回答一个很实际的问题：这个版本现在是否值得上线？

这种“质量信心”不是凭感觉，而是建立在测试覆盖、缺陷情况、风险分析和历史稳定性之上。

### 5. 支持发布决策

测试报告和缺陷分析，常常是发布评估的重要依据。测试团队不一定决定是否上线，但会为上线决策提供关键事实。

## 五、软件测试在软件生命周期中的位置

### 1. 需求阶段的测试活动

需求阶段并不是“还没写代码，所以没法测”。这个阶段同样需要测试思维。

常见工作包括：

- 检查需求是否明确
- 检查需求是否有歧义
- 检查需求是否可验证
- 提前识别高风险业务流程

例如，一条需求写成“系统应快速响应”，这就不够可测试。因为“快速”没有量化标准。更好的写法应该是“95% 的请求在 2 秒内返回结果”。

### 2. 设计阶段的测试活动

设计阶段可以从测试角度评估：

- 模块边界是否清晰
- 接口定义是否合理
- 异常处理是否完整
- 是否便于后续编写自动化测试

设计合理的系统，通常更容易测试；测试困难的系统，往往也意味着设计存在问题。

### 3. 编码阶段的测试活动

编码阶段最典型的测试活动包括：

- 单元测试
- 代码评审
- 静态代码分析
- 本地接口验证

这个阶段越早把问题挡住，后续集成成本越低。

### 4. 上线前后的测试活动

上线前的重点通常是：

- 功能验证
- 回归测试
- 性能与安全检查
- 发布前验收

上线后也不是测试工作的终点，还需要关注：

- 生产监控
- 日志分析
- 用户反馈
- 缺陷复盘

可以用一个简化图理解测试在生命周期中的位置：

```text
需求 -> 设计 -> 编码 -> 集成 -> 系统测试 -> 发布 -> 运行维护
  |       |       |        |          |          |         |
需求评审 设计评审 单元测试 集成测试   系统测试   验收判断   线上监控
```

## 六、软件测试的常见分类

软件测试的分类很多，学习时最好按照几个最常见的维度来理解。

### 1. 按是否运行程序分类

#### 静态测试

静态测试是不运行程序本身，而是通过检查文档、设计、代码来发现问题。

常见形式包括：

- 需求评审
- 设计评审
- 代码评审
- 静态代码扫描

静态测试的优点是介入早、成本低。

#### 动态测试

动态测试是运行程序，通过输入数据和观察输出结果来验证行为。

我们平时说的大多数功能测试、接口测试、性能测试，基本都属于动态测试。

### 2. 按测试层级分类

#### 单元测试

单元测试针对最小可测试单元，通常是函数、方法或类。

目标是确认局部逻辑正确。

示例：测试一个金额求和函数。

```python
def add_amount(a, b):
	return a + b


def test_add_amount():
	assert add_amount(100, 20) == 120
	assert add_amount(0, 0) == 0
```

这类测试粒度小、执行快，适合频繁运行。

#### 集成测试

集成测试关注多个模块组合起来是否工作正常。

例如：

- 订单服务调用库存服务是否正确
- 应用写入数据库是否成功
- 接口和消息队列是否协作正常

#### 系统测试

系统测试是从整体系统角度验证软件是否满足需求。它通常在接近真实环境下，验证完整业务流程。

例如：

- 用户注册
- 用户登录
- 下单支付
- 查看订单

#### 验收测试

验收测试通常从用户或业务角度判断系统是否可交付、可上线。

它更关注“这个产品是否满足业务目标”。

### 3. 按测试目的分类

#### 功能测试

验证功能是否按需求工作。

#### 性能测试

验证系统在响应时间、吞吐量、并发量、资源占用等方面是否符合预期。

#### 安全测试

验证系统是否存在身份认证、授权、输入校验、敏感数据保护等方面的问题。

#### 兼容性测试

验证系统在不同浏览器、设备、操作系统、数据库或运行环境下是否表现一致。

#### 可用性测试

验证系统是否容易理解、容易使用，用户是否容易犯错。

#### 可靠性与恢复测试

验证系统长期运行是否稳定，以及故障后能否恢复。

### 4. 按对内部实现的了解程度分类

#### 黑盒测试

黑盒测试不关注内部代码实现，只关注输入和输出。

例如，测试登录功能时，只看输入用户名密码后结果是否正确，而不关心内部怎么校验。

#### 白盒测试

白盒测试关注程序内部结构，例如分支、条件、循环和路径是否被覆盖。

#### 灰盒测试

灰盒测试介于两者之间。测试者知道部分内部实现信息，但仍主要从外部行为验证系统。

### 5. 按执行方式分类

#### 手工测试

由测试人员按照测试步骤人工执行。它适合探索性测试、交互体验验证和需求理解初期。

#### 自动化测试

由脚本或工具自动执行。它适合重复执行频率高、结果可稳定判断的场景。

## 七、软件测试中的关键概念

### 1. 测试用例

测试用例是对一次测试活动的结构化描述，通常包括：

- 用例名称
- 前置条件
- 测试步骤
- 输入数据
- 预期结果

一个简单示例如下：

| 用例名称 | 前置条件 | 操作步骤 | 预期结果 |
|----------|----------|----------|----------|
| 正确账号密码登录 | 用户已注册 | 输入正确账号和密码并点击登录 | 登录成功并进入首页 |

### 2. 测试套件

测试套件是多个相关测试用例的集合。例如“登录模块测试套件”中可能包含：

- 正确登录
- 错误密码登录
- 空密码登录
- 账号锁定验证
- 多端重复登录验证

### 3. 缺陷、故障、失效

这几个词经常混用，但含义有差别。

#### 缺陷

缺陷是软件中的错误根源，可能存在于需求、设计、代码或配置中。

#### 故障

故障通常指系统内部出现了异常状态。

#### 失效

失效是指系统对外表现出不符合预期的结果，例如页面报错、数据错误、接口超时。

可以粗略理解为：

- 缺陷是原因。
- 故障是系统内部出了问题。
- 失效是用户看到了错误结果。

### 4. 严重程度与优先级

#### 严重程度

严重程度描述问题本身影响有多大。

例如：

- 阻塞级：系统核心功能完全不可用
- 严重级：主要业务流程受影响
- 一般级：部分功能异常但有替代方案
- 轻微级：界面显示或提示问题

#### 优先级

优先级描述这个问题应该多快处理。

严重程度和优先级不一定完全相同。例如首页一个错别字严重程度低，但如果是品牌词错误，处理优先级可能很高。

### 5. 覆盖率

覆盖率是测试是否覆盖到某些对象的度量。常见有：

- 需求覆盖率
- 用例覆盖率
- 代码覆盖率
- 分支覆盖率

需要注意，覆盖率高不代表一定没有问题。它只能说明“测到的范围较广”，不能证明“软件完全正确”。

### 6. 回归测试与冒烟测试

#### 回归测试

回归测试是为了确认修改一个功能后，没有把原来正常的功能改坏。

#### 冒烟测试

冒烟测试是快速检查一个新版本最核心功能是否基本可用。

例如，一个电商系统发布后，可能先快速验证：

- 首页能否打开
- 登录能否成功
- 商品能否下单
- 支付流程是否可进入

如果这些最核心路径都不通，就没必要进行更大范围测试。

## 八、常见测试设计方法

测试设计方法的作用，是帮助我们更系统地选取测试数据和测试场景，而不是凭感觉随意点点。

### 1. 等价类划分

等价类划分是把大量输入分成若干“等价”的类别，每个类别选少量代表值测试即可。

例如，一个年龄输入框要求范围是 1 到 120。

可以划分为：

- 有效等价类：1 到 120
- 无效等价类：小于 1
- 无效等价类：大于 120
- 无效等价类：非数字输入

这样就不用把 1 到 120 每个值都测一遍。

### 2. 边界值分析

很多缺陷容易出现在边界附近，所以边界值分析特别常用。

仍然以年龄范围 1 到 120 为例，重点测试值通常包括：

| 类型 | 测试值 |
|------|--------|
| 下边界外 | 0 |
| 下边界上 | 1 |
| 下边界内 | 2 |
| 上边界内 | 119 |
| 上边界上 | 120 |
| 上边界外 | 121 |

### 3. 判定表法

当系统行为由多个条件共同决定时，判定表法很有用。

例如，一个优惠规则如下：

- 会员用户且订单金额大于等于 100 元，可打 9 折。
- 非会员用户无折扣。

可以设计成判定表：

| 是否会员 | 金额是否 >= 100 | 预期结果 |
|----------|------------------|----------|
| 是 | 是 | 9 折 |
| 是 | 否 | 原价 |
| 否 | 是 | 原价 |
| 否 | 否 | 原价 |

### 4. 状态迁移法

对于有明确状态流转的系统，状态迁移法很重要。

例如订单状态可能是：

```text
待支付 -> 已支付 -> 已发货 -> 已完成
   |
   -> 已取消
```

这时就要测试：

- 合法状态流转是否可执行
- 非法状态流转是否被阻止

例如“已取消订单”不应该再进入“已发货”。

### 5. 场景法

场景法强调围绕完整业务流程设计测试，而不是孤立地测试单个输入框。

例如电商下单场景可以设计为：

- 浏览商品
- 加入购物车
- 提交订单
- 选择地址
- 支付成功
- 查看订单详情

它更接近用户真实使用方式。

### 6. 错误推测法

错误推测法依赖经验。测试人员根据以往项目经验，推测哪里最容易出问题。

例如：

- 空值处理
- 重复提交
- 特殊字符输入
- 网络中断
- 时间跨天、跨月、跨年

经验丰富的测试人员，往往能通过错误推测法快速发现高价值问题。

## 九、测试执行过程与常见产物

### 1. 测试计划

测试计划回答的是：测什么、怎么测、谁来测、什么时候测、风险是什么。

一个测试计划通常包含：

- 测试范围
- 测试目标
- 测试策略
- 人员安排
- 时间安排
- 风险与应对措施

### 2. 测试设计

测试设计是在理解需求和系统行为之后，编写测试用例、准备测试数据并设计测试场景。

### 3. 测试环境

测试环境会直接影响测试结果。常见关注点包括：

- 软件版本是否正确
- 数据库版本是否一致
- 配置是否与目标环境接近
- 第三方依赖是否可用

很多“测不出来、线上才出问题”的根源，其实是环境差异太大。

### 4. 测试执行

执行阶段的核心是：

- 按用例执行操作
- 记录实际结果
- 对比预期结果
- 发现问题后提交缺陷

### 5. 缺陷管理

发现缺陷后，不能只口头说“这里有 Bug”，而要进入规范的缺陷管理流程。

一个典型的缺陷生命周期可以简化理解为：

```text
新建 -> 已确认 -> 已修复 -> 待回归 -> 已关闭
		   |
		   -> 不是缺陷 / 延后处理
```

一个高质量缺陷单通常应包含：

- 缺陷标题
- 复现步骤
- 实际结果
- 预期结果
- 运行环境
- 附件信息，例如截图、日志、请求报文

### 6. 测试报告

测试报告用于总结本轮测试情况，常见内容包括：

- 执行了多少用例
- 通过率如何
- 发现了多少缺陷
- 剩余风险是什么
- 是否建议上线

测试报告不是为了“写文档交差”，而是为了让团队形成统一的质量判断依据。

## 十、自动化测试与持续集成

### 1. 为什么要自动化

有些测试需要重复执行很多次，例如：

- 每次提交代码都要跑一遍
- 每次发版都要回归核心流程
- 多个环境都要执行相同验证

如果全部靠人工，效率低、成本高，也容易漏测。因此，自动化测试非常有价值。

### 2. 测试金字塔

测试金字塔是理解自动化测试布局的经典模型。

```text
		   UI 测试
		----------------
		  接口/服务测试
	 ------------------------
			单元测试
--------------------------------
```

它表达的核心意思是：

- 单元测试数量应最多，执行最快。
- 接口测试位于中间层，验证服务协作。
- UI 测试数量相对少，因为它通常更慢、更脆弱、维护成本更高。

### 3. 自动化适用场景

自动化测试尤其适合：

- 稳定且重复执行的流程
- 核心回归路径
- 接口层验证
- 数据驱动的批量校验

自动化测试不太适合完全依赖主观判断的场景，例如早期探索性体验测试。

### 4. CI/CD 中的测试位置

在持续集成和持续交付中，测试通常作为质量门禁存在。

一个简化流程如下：

```text
代码提交 -> 自动构建 -> 单元测试 -> 接口测试 -> 部署测试环境 -> 回归测试 -> 发布
```

如果前面的测试失败，后续流程就应该被阻止。

### 5. 自动化测试代码示例

下面是一个简单的 `pytest` 示例，用来说明自动化测试的基本结构：

```python
def is_adult(age):
	return age >= 18


def test_is_adult_with_boundary_values():
	assert is_adult(18) is True
	assert is_adult(17) is False
	assert is_adult(100) is True
```

这个例子体现了两个基本思想：

- 测试代码本身也需要清晰表达预期。
- 边界值通常是高价值测试点。

如果是接口测试，思路也类似。下面给出一个简化示意：

```python
def login(username, password):
	if username == "admin" and password == "123456":
		return {"code": 200, "message": "success"}
	return {"code": 401, "message": "unauthorized"}


def test_login_success():
	result = login("admin", "123456")
	assert result["code"] == 200


def test_login_failed():
	result = login("admin", "bad-password")
	assert result["code"] == 401
```

## 十一、软件测试的收益与局限

### 1. 测试带来的主要收益

#### 质量收益

测试能够提高缺陷发现率，减少明显问题流入生产环境。

#### 风险收益

测试能够帮助团队识别高风险模块和高风险流程，把有限资源优先投入到关键位置。

#### 协作收益

测试活动会倒逼需求更清晰、设计更规范、代码更可维护，从而改善团队协作质量。

#### 交付收益

当自动化测试和质量门禁建立起来后，版本发布会更稳定，团队更有信心进行频繁迭代。

### 2. 测试不能解决的问题

测试很重要，但也有边界。

测试不能：

- 证明软件完全没有缺陷
- 替代清晰需求和合理设计
- 替代代码质量本身
- 在没有时间和资源的情况下覆盖所有可能场景

### 3. 为什么测试不能证明程序完全无错

因为软件输入空间通常非常大，运行环境和用户行为也千变万化。测试只能说明：

- 在已经覆盖到的条件下，软件表现符合预期。

而不能说明：

- 在所有可能条件下，软件永远不会出错。

这也是为什么软件测试更强调“风险控制”而不是“绝对正确”。

## 十二、如何理解软件测试的整体图景

### 1. 测试不是最后一步

测试不应该只在开发结束后才开始。真正成熟的测试体系，是从需求阶段就开始介入，一直到上线后继续通过监控和反馈不断验证质量。

### 2. 测试是质量保障体系的一部分

软件质量不是单靠测试团队保证的，而是需求、设计、开发、测试、运维共同作用的结果。

测试在其中承担的是：

- 验证
- 反馈
- 风险识别
- 质量量化

### 3. 从“找 Bug”到“控风险”的认识升级

初学者容易把测试理解为“专门找 Bug”。这个理解不算错，但不完整。

更完整的认识应该是：

- 软件测试要发现问题。
- 软件测试要验证需求。
- 软件测试要帮助团队理解风险。
- 软件测试要为交付决策提供依据。

当理解到这一层时，就会知道测试并不是开发流程中的“附属环节”，而是软件工程中的核心质量活动。

---

## 十三、总结

软件测试的本质，是通过系统化的方法验证软件是否符合预期、尽早发现缺陷并降低交付风险。

要建立完整概念，可以抓住以下主线：

- 软件测试不是只在最后执行，而是贯穿整个软件生命周期。
- 软件测试既包括静态检查，也包括动态运行验证。
- 软件测试可以按层级、目的、视角和执行方式进行分类。
- 测试设计需要方法，常见方法包括等价类、边界值、判定表、状态迁移和场景法。
- 自动化测试是现代研发流程的重要组成部分，但并不能替代全部手工测试。
- 测试的核心目标不是证明系统绝对无错，而是识别问题、评估质量和控制风险。

如果从学习路径上看，接下来可以继续深入以下主题：

- 单元测试、集成测试、系统测试的实践差异
- 常见测试用例设计方法的专项练习
- 接口测试与自动化测试框架
- 性能测试、安全测试与兼容性测试
- 测试管理、缺陷管理和质量度量

学完这篇后，应该先建立这样一个判断：软件测试不是“上线前随便检查一下”，而是一整套围绕质量和风险控制展开的工程实践。
