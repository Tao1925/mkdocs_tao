## 二、数据库系统概论是什么

### 1. 为什么这门课叫“数据库系统概论”

很多同学刚接触数据库课程时，会把它理解成“学 SQL 的课”。

这个理解不算错，但明显不完整。

数据库系统概论，研究的不只是 SQL 命令怎么写，还包括下面这些问题。

- 数据应该如何组织。
- 多个用户如何同时安全访问同一份数据。
- 系统宕机后数据如何恢复。
- 数据结构设计为什么会影响查询效率。
- 为什么数据库能在高并发下仍然保持较强的一致性。

换句话说，SQL 只是数据库系统中的一部分语言工具，而数据库系统概论研究的是一个更完整的系统。

### 2. 数据库、数据库管理系统、数据库系统的区别

这三个词非常常见，但不能混为一谈。

#### 数据库 Database

数据库，是按照一定结构组织并长期存储在计算机中的数据集合。

例如。

- 一个学校的学生信息库。
- 一个电商网站的订单库。
- 一个企业的人事管理数据。

数据库强调的是“数据本身及其组织”。

#### 数据库管理系统 DBMS

数据库管理系统，是管理数据库的软件系统。

它负责完成很多复杂任务。

- 存储数据。
- 执行查询。
- 控制权限。
- 管理事务。
- 做恢复和并发控制。

常见 DBMS 有。

- MySQL
- PostgreSQL
- Oracle
- SQL Server
- SQLite

#### 数据库系统 DBS

数据库系统是更完整的概念，通常包括。

- 数据库
- 数据库管理系统
- 应用程序
- 数据库管理员和用户
- 相关硬件和操作系统环境

所以，数据库系统不是单纯一个软件，也不是单纯一堆数据，而是一个围绕数据管理建立起来的整体系统。

### 3. 数据库在软件系统中的位置

现代软件系统里，数据库通常承担“状态持久化中心”的角色。

例如在一个电商系统中。

- 用户注册信息存放在数据库。
- 商品库存信息存放在数据库。
- 订单和支付记录存放在数据库。

应用程序负责处理业务逻辑。

数据库负责把这些状态可靠地保存下来，并支持查询与修改。

### 4. 为什么“数据库系统概论”不只是学增删改查

如果数据库只是“增删改查工具”，那它不会成为一门核心基础课。

数据库系统真正重要的地方在于，它把很多难题统一封装进了一个通用系统。

例如。

- 多人同时修改同一数据怎么办。
- 一个操作只做了一半突然断电怎么办。
- 如何保证查询快而且数据不乱。
- 如何让用户只看到自己有权限的数据。

这些问题如果都交给应用程序自己解决，代价会非常高。

数据库系统的价值，就是把这些通用问题标准化、系统化。

### 5. 本科阶段学习数据库系统概论的重点

对本科生来说，这门课最重要的目标通常不是“会调数据库底层源码”，而是建立一套完整认识。

- 理解数据库为什么存在。
- 理解关系模型为什么重要。
- 理解 SQL 如何表达数据定义和数据查询。
- 理解事务、一致性、并发控制、恢复等核心机制。
- 能做一个基础数据库设计，并写出可靠 SQL。

### 6. 一个直观例子：学生选课系统

假设学校要做一个选课系统。

系统至少要管理这些信息。

- 学生
- 教师
- 课程
- 开课班次
- 选课记录

如果没有数据库，数据可能散落在多个 Excel 或文本文件中。

这会带来很多问题。

- 容易重复记录。
- 修改困难。
- 查询麻烦。
- 多人同时操作时容易冲突。

而数据库系统正是用来把这些结构、关系和操作统一管理起来的。

### 7. 本章小结

数据库系统概论研究的是“如何高效、可靠、安全地管理数据”。

它不仅包括数据本身，还包括管理数据的软件、访问数据的人以及支撑这些活动的系统机制。

## 三、为什么需要数据库

### 1. 只用文件系统管理数据有什么问题

很多人最开始接触数据时，并不是先接触数据库，而是先接触文件。

例如。

- 文本文件
- CSV 文件
- Excel 表格
- JSON 文件

这些方式在数据量小、业务简单时确实可以工作。

但一旦数据规模和业务复杂度上升，就会暴露很多问题。

### 2. 文件系统管理数据的典型局限

#### 数据冗余高

如果多个文件分别保存相似内容，很容易出现重复存储。

例如学生基本信息可能在多个表格里重复出现。

#### 数据一致性差

某个学生改了电话号码，如果多个文件都存过这个字段，就可能出现有的更新了，有的没更新。

#### 查询困难

如果你要找“所有已选数据库课程且平均成绩大于 85 的学生”，单纯依靠多个文件拼接会非常麻烦。

#### 并发支持差

如果两个人同时修改同一个文件，冲突和覆盖问题会很明显。

#### 安全性弱

文件级权限管理通常比较粗糙，不容易做到精细授权。

### 3. 数据共享需求推动了数据库出现

现代信息系统很少只有一个人使用。

通常会有。

- 多个业务模块访问同一数据。
- 多个用户同时读写同一系统。
- 多个服务之间共享基础数据。

如果没有统一的数据管理中心，这种共享会非常混乱。

数据库的一个重要价值，就是在共享前提下保持有序管理。

### 4. 持久化需求推动了数据库出现

程序运行时的数据通常保存在内存里。

但内存中的内容断电就会丢失。

现实业务中的很多数据必须长期保存。

例如。

- 银行交易记录。
- 订单记录。
- 学生成绩。
- 医疗档案。

数据库正是用来支撑这种长期、可靠持久化的关键基础设施。

### 5. 一致性需求推动了数据库出现

很多数据并不是随便存下就行，而是必须满足逻辑约束。

例如。

- 学号不能重复。
- 订单必须对应真实用户。
- 成绩不能超出合理区间。
- 转账不能让总金额凭空增加或减少。

数据库通过主键、外键、事务和约束，帮助系统维护这些一致性要求。

### 6. 为什么数据库不是“更高级的 Excel”

Excel 更偏向个人分析工具。

数据库更偏向面向应用系统和多用户场景的数据管理平台。

它们有本质区别。

- Excel 更强调表格编辑。
- 数据库更强调结构化管理、查询优化、事务控制和并发访问。

所以把数据库理解成“能写公式的大表格”会严重低估数据库的系统性价值。

### 7. 数据库对现代软件系统的基础支撑作用

今天几乎所有业务系统都离不开数据库。

例如。

- 网站的注册登录系统。
- 电商订单系统。
- ERP、CRM、OA 系统。
- 物流、支付、库存系统。

即使一些系统不完全使用关系型数据库，也通常仍然会有某种数据库作为核心基础设施。

### 8. 本章小结

数据库产生的根本原因，不是“大家想发明一种新软件”，而是文件式数据管理无法满足现代系统在共享、持久化、一致性、安全性和查询效率方面的要求。

## 四、数据库中的基本概念

### 1. 数据与信息

数据，是对客观事物的一种符号化表示。

例如数字、文字、图像、时间、地理位置，都可以视作数据。

信息，则是对数据进行加工、组织和解释后得到的有意义结果。

例如。

- 一串成绩数字本身是数据。
- “这个班平均分为 82 分”是信息。

数据库课程更关注数据的组织和管理方式，但最终目的往往是支持信息获取。

### 2. 数据模型

数据模型，是对现实世界数据特征和关系的一种抽象描述。

它回答的问题包括。

- 数据有哪些对象。
- 对象有哪些属性。
- 对象之间是什么关系。

不同数据库系统背后，通常都有某种数据模型作为理论基础。

### 3. 模式与实例

这两个概念非常基础，也很容易混淆。

#### 模式 Schema

模式描述的是数据库的结构定义。

例如某张表有哪些字段、每个字段是什么类型、有什么约束。

#### 实例 Instance

实例描述的是某一时刻数据库中的实际数据内容。

例如今天学生表里到底有哪些学生记录。

可以简单理解为。

- 模式像“表格模板”。
- 实例像“模板里当前填的内容”。

### 4. 元数据

元数据，就是“描述数据的数据”。

例如。

- 表结构定义。
- 列类型。
- 索引信息。
- 视图定义。

数据库管理系统要靠这些元数据理解数据库结构。

### 5. DBMS 的作用

数据库管理系统不仅负责存数据，还负责提供完整管理能力。

它通常要完成下面几件事情。

- 定义数据结构。
- 存储和读取数据。
- 执行查询。
- 管理事务。
- 控制并发和恢复。
- 做安全权限控制。

### 6. DBA 与数据库用户

数据库系统中通常有不同角色。

#### DBA

数据库管理员，负责数据库安装、维护、备份、权限、安全和性能管理。

#### 应用开发者

负责设计表结构、编写 SQL、通过程序访问数据库。

#### 最终用户

通过业务系统间接使用数据库，不一定直接接触 SQL。

### 7. 数据库系统的组成

完整数据库系统通常包含。

- 硬件平台。
- 操作系统。
- DBMS。
- 数据库。
- 应用程序。
- 用户与管理员。

这说明数据库不是一个孤立概念，而是运行在具体计算环境中的系统。

### 8. 本章小结

这一章最重要的是建立共同语言。

后面学习时，你需要清楚地区分。

- 数据和信息。
- 模式和实例。
- 数据库、DBMS 和数据库系统。

## 五、数据模型与三级模式结构

### 1. 为什么数据库需要数据模型

现实世界非常复杂。

如果不进行抽象，计算机无法直接理解“学生”“课程”“订单”这些对象及其关系。

数据模型的作用，就是把现实问题转换成数据库可以处理的结构化形式。

### 2. 概念模型

概念模型面向现实世界建模。

它关心的是。

- 有哪些实体。
- 每个实体有哪些属性。
- 实体之间有什么联系。

概念模型通常不直接关心某个字段在磁盘上怎么存，而更关心业务层面的结构。

ER 模型就是最经典的概念模型之一。

### 3. 逻辑模型

逻辑模型是在概念模型基础上，进一步用某种数据库支持的逻辑结构来表示数据。

在关系型数据库中，逻辑模型通常表现为。

- 表
- 行
- 列
- 键
- 约束

### 4. 物理模型

物理模型更关注数据在计算机内部如何真正存储和访问。

例如。

- 文件组织方式。
- 页和块。
- 索引结构。
- 存储引擎。

物理模型更接近数据库实现与优化层面。

### 5. 三级模式结构

数据库系统理论中，一个很重要的框架是三级模式结构。

它包括。

- 外模式
- 模式
- 内模式

### 6. 外模式

外模式也叫子模式，面向用户或应用。

不同用户可能看到数据库的不同部分。

例如。

- 教务员看到全部学生信息。
- 任课老师只能看到自己课程学生信息。
- 学生只能看到自己的成绩。

外模式强调的是“不同视角下的数据表示”。

### 7. 模式

模式也叫概念模式，描述的是数据库的全局逻辑结构。

它相当于整个数据库在逻辑层面的总体定义。

### 8. 内模式

内模式描述的是数据库在物理存储层面的组织方式。

例如。

- 数据按什么文件组织。
- 索引如何建立。
- 记录如何存放。

### 9. 两级映像

为了让这三级结构彼此衔接，数据库系统通常引入两级映像。

- 外模式 / 模式映像。
- 模式 / 内模式映像。

它们帮助系统在不同抽象层之间建立对应关系。

### 10. 数据独立性

三级模式结构的重要意义之一，是实现数据独立性。

数据独立性通常分成两类。

#### 物理数据独立性

物理存储方式改变时，尽量不影响逻辑结构和应用程序。

例如加索引、改变文件组织方式，不希望程序大改。

#### 逻辑数据独立性

逻辑结构改变时，尽量不影响外部应用视图。

例如表结构做一定扩展，不希望所有程序都重写。

### 11. 为什么数据独立性重要

因为真实系统不会一成不变。

数据库必须支持。

- 性能优化。
- 结构演进。
- 视图调整。

如果每次底层变化都要求所有应用重写，系统维护成本会非常高。

### 12. 本章小结

学习三级模式结构时，不要只把它当考试名词。

它真正表达的是一个重要思想。

数据库系统要把“用户看到的数据”“逻辑结构”“物理存储”分层处理，从而降低耦合，提高演进能力。

## 六、关系数据库基础

### 1. 什么是关系模型

关系模型是数据库领域最成功、最广泛应用的数据模型之一。

它把数据组织成若干关系，也就是我们熟悉的表。

每张表由行和列组成。

这种表示方式清晰、统一、便于数学化分析，因此非常适合教学和工程实践。

### 2. 关系、元组、属性、域

学习关系模型时，常见几个术语。

#### 关系 Relation

可以直观理解为一张表。

#### 元组 Tuple

可以直观理解为表中的一行记录。

#### 属性 Attribute

可以直观理解为表中的一列字段。

#### 域 Domain

指某个属性允许取值的集合。

例如“年龄”这个属性的域通常是非负整数范围。

### 3. 关系模型为什么清晰

因为它把现实中的实体和关系，统一表达成表。

例如。

- 学生表
- 课程表
- 选课表

其中选课表就可以表达“学生和课程之间的多对多关系”。

### 4. 键的概念

关系模型中，键非常重要。

#### 超键 Super Key

能唯一标识元组的属性集。

#### 候选键 Candidate Key

最小超键，也就是去掉任何一个属性后就不再唯一。

#### 主键 Primary Key

从候选键中选定的一个作为主要标识。

#### 外键 Foreign Key

一个表中的属性，用来引用另一个表的主键或候选键。

### 5. 为什么键重要

键不仅帮助唯一标识数据，还帮助表之间建立联系。

没有键，数据库就很难维护一致关系。

### 6. 关系的基本性质

从理论上讲，关系模型里的表具有一些经典性质。

- 列名通常不同。
- 行的顺序本身不重要。
- 列的顺序本身也不重要。
- 每个单元格应是不可再分的原子值。

这里最后一点非常关键，它和第一范式有关。

### 7. 实体完整性

实体完整性强调。

- 主键值不能为 NULL。
- 主键必须能唯一标识一条记录。

否则系统就无法准确区分记录。

### 8. 参照完整性

参照完整性强调。

- 外键要么为空。
- 要么必须引用被参照表中真实存在的键值。

例如，如果某条选课记录中的学生编号在学生表中根本不存在，这就破坏了参照完整性。

### 9. 用户定义完整性

除实体完整性和参照完整性外，很多业务还有自己的规则。

例如。

- 成绩范围必须在 0 到 100 之间。
- 账户余额不能为负。
- 邮箱地址应该满足格式要求。

这些通常属于用户定义完整性。

### 10. 为什么关系模型成为主流

关系模型之所以长期成为主流，不只是因为“表格直观”。

更重要原因包括。

- 理论基础扎实。
- 查询语言统一。
- 易于表达大多数业务结构。
- 与事务、一致性和规范化设计结合紧密。

### 11. 本章小结

关系模型是关系型数据库的理论核心。

理解这一章后，后面的 SQL、主键外键、约束、规范化设计都会更容易理解。

## 七、ER 模型与数据库设计基础

### 1. 为什么数据库设计不能直接开始建表

很多初学者一上来就开始写 `CREATE TABLE`。

这通常会导致设计混乱。

更合理的流程是。

1. 先理解业务对象。
2. 再识别对象属性与对象之间的关系。
3. 再把概念模型转换为关系表。

这也是 ER 模型存在的重要意义。

### 2. 实体 Entity

实体是现实世界中可以独立区分的对象。

例如在选课系统中。

- 学生是实体。
- 教师是实体。
- 课程是实体。

### 3. 属性 Attribute

属性是实体的特征。

例如学生实体可能有下面这些属性。

- 学号
- 姓名
- 性别
- 年级
- 专业

### 4. 联系 Relationship

联系表示实体之间的关联。

例如。

- 学生选修课程。
- 教师教授课程。

联系是数据库设计中非常关键的部分，因为它决定后续表之间如何关联。

### 5. 一对一联系

例如一个学生对应一张学生证，一张学生证只对应一个学生。

这就是典型的一对一关系。

### 6. 一对多联系

例如一个班级有多个学生，一个学生只属于一个班级。

这是一对多关系。

### 7. 多对多联系

例如一个学生可以选多门课程，一门课程也可以被多个学生选。

这就是多对多关系。

数据库中，多对多关系通常要通过中间表拆开。

### 8. 弱实体

有些实体不能靠自身属性完全独立标识，必须依赖另一个实体。

这类实体通常称为弱实体。

本科阶段只要理解思想即可，不一定一开始就深究所有细节。

### 9. ER 图到关系模式的转换

这是数据库设计中的核心步骤之一。

常见规则包括。

- 一个实体通常转换为一张表。
- 一对多联系通常在“多”的一方放外键。
- 多对多联系通常转换为单独中间表。

### 10. 一个简单建模示例

假设有三个实体。

- Student
- Course
- Teacher

以及两个联系。

- Student 选 Course
- Teacher 教 Course

那么很自然会设计成。

- students 表
- teachers 表
- courses 表
- enrollments 表

其中 `enrollments` 用于表示学生与课程的多对多关系。

### 11. 本章小结

ER 模型的价值，在于让我们先从业务对象和关系出发，再去设计数据库表，而不是一开始就陷入字段细节。

## 八、关系代数与查询思想

### 1. 为什么要学关系代数

很多同学会问，既然实际工作中直接写 SQL，为什么还要学关系代数。

原因在于。

- 关系代数更抽象。
- 它帮助我们理解查询本质。
- 它是很多查询优化理论的重要基础。

SQL 是具体语言。

关系代数更像底层思想框架。

### 2. 选择 Selection

选择操作用于从关系中筛选满足条件的元组。

直观上相当于 SQL 中的 `WHERE` 条件过滤。

### 3. 投影 Projection

投影操作用于从关系中选取若干属性列。

直观上相当于 SQL 中的 `SELECT 某些列`。

### 4. 并、差、交

如果两个关系结构兼容，就可以做集合运算。

- 并：合并结果。
- 差：取一个关系有而另一个没有的元组。
- 交：取共同部分。

这和 SQL 中的 `UNION` 等操作相关。

### 5. 笛卡尔积

笛卡尔积会把两个关系的每一行进行组合。

如果直接使用，结果往往非常大。

在实际查询中，连接通常可以看作在笛卡尔积基础上加条件筛选得到更有意义的结果。

### 6. 连接 Join

连接是关系数据库查询中最常用的核心操作之一。

它用于把多个表按某种关联条件组合起来。

例如把学生表和选课表连接起来，就可以得到某个学生选了哪些课程。

### 7. 为什么连接如此重要

因为规范化设计后，数据通常被拆分到多个表中。

查询时必须再把相关信息合理拼回去。

这就是连接的作用。

### 8. 本章小结

关系代数的意义，不是替代 SQL，而是帮助我们理解 SQL 背后的查询本质：筛选、投影、组合和聚合。

## 九、SQL 语言概览

### 1. SQL 是什么

SQL 是 Structured Query Language，也就是结构化查询语言。

它是关系型数据库中最重要的交互语言。

通过 SQL，我们可以。

- 定义数据库结构。
- 插入和修改数据。
- 查询数据。
- 管理权限。
- 控制事务。

### 2. SQL 的几个特点

- 声明式。
- 面向集合。
- 接近自然语言表达。

所谓声明式，意思是你更关注“要什么结果”，而不是一步步告诉数据库“怎么遍历”。

### 3. SQL 的常见分类

在教学中，通常把 SQL 分成几类。

#### DDL

数据定义语言，用于定义数据库对象。

例如。

- `CREATE`
- `ALTER`
- `DROP`

#### DML

数据操作语言，用于修改数据。

例如。

- `INSERT`
- `UPDATE`
- `DELETE`

#### DQL

数据查询语言，常用来指 `SELECT` 查询。

严格说 SQL 标准里未必单独强调 DQL 这个类别，但在教学和实践中很常见。

#### DCL

数据控制语言，用于权限管理。

例如。

- `GRANT`
- `REVOKE`

#### TCL

事务控制语言。

例如。

- `BEGIN`
- `COMMIT`
- `ROLLBACK`

### 4. 标准 SQL 与数据库方言

SQL 有标准，但不同数据库产品通常会有自己的方言差异。

例如。

- 自增写法不同。
- 字符串函数名字不同。
- 分页语法可能不同。
- 存储过程和触发器语法差异更明显。

这篇文档中的示例会尽量以 MySQL 风格为主，同时保持概念可迁移。

### 5. 学 SQL 时的常见误区

误区一，只记语法，不理解数据结构。

误区二，只会写单表查询，不会从关系角度思考。

误区三，认为 SQL 查询快慢只和语法写法有关，而忽略索引与设计。

误区四，把 SQL 当成“程序语言替代品”，而忽略其声明式特征。

### 6. 本章小结

SQL 是数据库系统最重要的交互语言，但它不是数据库的全部。真正学好 SQL，需要同时理解关系模型、表设计和数据库系统机制。

## 十、SQL 数据定义语言 DDL

### 1. 什么是 DDL

DDL，Data Definition Language，数据定义语言。

它用于定义和修改数据库对象的结构。

常见对象包括。

- 数据库
- 表
- 视图
- 索引

### 2. 创建数据库

在 MySQL 中，创建数据库的基本语法如下。

```sql
CREATE DATABASE university_db;
```

如果希望指定字符集和排序规则，可以写得更明确。

```sql
CREATE DATABASE university_db
CHARACTER SET utf8mb4
COLLATE utf8mb4_unicode_ci;
```

### 3. 查看和使用数据库

```sql
SHOW DATABASES;
USE university_db;
```

第一条用于查看当前数据库列表。

第二条用于切换当前使用的数据库。

### 4. 删除数据库

```sql
DROP DATABASE university_db;
```

这个操作通常不可逆，因此实际环境中必须谨慎。

### 5. 创建表的基本语法

```sql
CREATE TABLE 表名 (
	列名1 数据类型 [约束],
	列名2 数据类型 [约束],
	...
	[表级约束]
);
```

例如创建学生表。

```sql
CREATE TABLE students (
	student_id INT PRIMARY KEY AUTO_INCREMENT,
	student_no VARCHAR(20) NOT NULL UNIQUE,
	name VARCHAR(50) NOT NULL,
	gender CHAR(1),
	major VARCHAR(100),
	admission_year INT,
	created_at DATETIME DEFAULT CURRENT_TIMESTAMP
);
```

### 6. 常见数据类型

SQL 数据类型很多，本科阶段常见的可以先掌握下面这些。

#### 整数类型

- `INT`
- `BIGINT`
- `SMALLINT`

用于存放整数。

#### 小数类型

- `DECIMAL(p, s)`
- `FLOAT`
- `DOUBLE`

其中 `DECIMAL` 更适合金额这类要求精确表示的数据。

#### 字符串类型

- `CHAR(n)`：定长字符串。
- `VARCHAR(n)`：变长字符串。
- `TEXT`：长文本。

#### 日期时间类型

- `DATE`
- `TIME`
- `DATETIME`
- `TIMESTAMP`

### 7. 为什么金额常用 DECIMAL 而不是 FLOAT

因为浮点数存在二进制近似表示问题。

在金额计算中，近似误差可能带来严重后果。

因此数据库中金额类字段通常更适合用 `DECIMAL`。

### 8. 查看表结构

```sql
DESC students;
SHOW CREATE TABLE students;
```

前者用于快速查看字段信息。

后者可以查看完整建表语句。

### 9. 修改表结构

`ALTER TABLE` 是 DDL 中非常重要的命令。

常见用途包括。

- 添加列。
- 修改列类型。
- 删除列。
- 添加约束。
- 删除约束。

### 10. 添加列

```sql
ALTER TABLE students
ADD COLUMN email VARCHAR(100);
```

### 11. 修改列

```sql
ALTER TABLE students
MODIFY COLUMN email VARCHAR(150) NOT NULL;
```

### 12. 删除列

```sql
ALTER TABLE students
DROP COLUMN email;
```

### 13. 创建课程表示例

```sql
CREATE TABLE courses (
	course_id INT PRIMARY KEY AUTO_INCREMENT,
	course_code VARCHAR(20) NOT NULL UNIQUE,
	course_name VARCHAR(100) NOT NULL,
	credit DECIMAL(3, 1) NOT NULL,
	department VARCHAR(100)
);
```

### 14. 创建选课表示例

```sql
CREATE TABLE enrollments (
	enrollment_id INT PRIMARY KEY AUTO_INCREMENT,
	student_id INT NOT NULL,
	course_id INT NOT NULL,
	score DECIMAL(5, 2),
	selected_at DATETIME DEFAULT CURRENT_TIMESTAMP,
	CONSTRAINT fk_enrollment_student
		FOREIGN KEY (student_id) REFERENCES students(student_id),
	CONSTRAINT fk_enrollment_course
		FOREIGN KEY (course_id) REFERENCES courses(course_id)
);
```

### 15. DDL 操作中的常见注意点

- 建表前先想清楚主键设计。
- 金额、时间、编码类字段应选合适类型。
- 外键是否真的需要由数据库强约束，要结合场景评估。
- 生产环境修改表结构要特别谨慎，可能影响性能和可用性。

### 16. 本章小结

DDL 的核心，是定义数据库对象的结构。学会 DDL，不只是会写 `CREATE TABLE`，更重要的是知道字段类型、主键、外键和约束为什么这样设计。

## 十一、SQL 数据操作语言 DML

### 1. 什么是 DML

DML，Data Manipulation Language，数据操作语言。

它主要负责对表中的数据进行增加、修改和删除。

最常见的命令就是。

- `INSERT`
- `UPDATE`
- `DELETE`

### 2. INSERT 基本语法

```sql
INSERT INTO 表名 (列1, 列2, 列3)
VALUES (值1, 值2, 值3);
```

例如插入一条学生记录。

```sql
INSERT INTO students (student_no, name, gender, major, admission_year)
VALUES ('20240001', '张三', 'M', '计算机科学与技术', 2024);
```

### 3. 批量插入

```sql
INSERT INTO students (student_no, name, gender, major, admission_year)
VALUES
('20240002', '李四', 'F', '软件工程', 2024),
('20240003', '王五', 'M', '信息安全', 2023);
```

批量插入在导入测试数据时非常常见。

### 4. INSERT 时常见问题

- 列和值数量不一致。
- 违反主键或唯一约束。
- 插入了不符合字段类型的数据。
- 忘记处理非空列。

### 5. UPDATE 基本语法

```sql
UPDATE 表名
SET 列1 = 值1, 列2 = 值2
WHERE 条件;
```

例如更新学生专业。

```sql
UPDATE students
SET major = '人工智能'
WHERE student_no = '20240001';
```

### 6. 为什么 UPDATE 必须谨慎写 WHERE 条件

如果忘了 `WHERE`，就可能把整张表都改掉。

例如。

```sql
UPDATE students
SET major = '未知';
```

这会把所有学生的专业都改成同一个值。

因此实际工作中，执行 UPDATE 前应养成检查条件范围的习惯。

### 7. DELETE 基本语法

```sql
DELETE FROM 表名
WHERE 条件;
```

例如删除某条选课记录。

```sql
DELETE FROM enrollments
WHERE enrollment_id = 10;
```

### 8. DELETE 和 TRUNCATE 的区别

`DELETE` 通常是按条件删除数据，属于 DML。

`TRUNCATE` 更像快速清空整张表，常被视作更偏 DDL 的操作。

两者在日志、恢复和自增行为上可能有差异。

### 9. 逻辑删除与物理删除

在很多业务系统中，并不直接物理删除数据，而是使用逻辑删除。

例如加一个字段。

```sql
is_deleted TINYINT DEFAULT 0
```

删除时并不执行 `DELETE`，而是。

```sql
UPDATE students
SET is_deleted = 1
WHERE student_id = 1;
```

这样做的好处包括。

- 便于审计。
- 便于恢复。
- 降低误删风险。

### 10. DML 操作中的常见风险

- 误更新整表。
- 误删除整表。
- 批量导入时数据重复。
- 多表关联数据删除导致参照完整性问题。

### 11. 本章小结

DML 的重点不只是“会写语法”，更重要的是理解对数据的修改会带来什么后果，尤其要养成对 `WHERE` 条件、约束和事务边界保持敏感的习惯。

## 十二、SQL 查询语言 DQL 基础

### 1. 什么是 DQL

教学中通常把 `SELECT` 查询相关语句归为 DQL，也就是数据查询语言。

对于很多应用来说，查询其实是数据库最常用的能力。

### 2. SELECT 的基本结构

最基础的查询语句结构如下。

```sql
SELECT 列列表
FROM 表名
WHERE 条件
ORDER BY 排序规则
LIMIT 条数;
```

并不是每次都必须写全这些部分。

但理解整体结构非常重要。

### 3. 查询所有列

```sql
SELECT * FROM students;
```

这条语句表示查询学生表的全部列。

在教学里很常见。

但在实际项目中，通常不建议长期大量使用 `SELECT *`，因为。

- 会取出不需要的列。
- 可能增加网络和解析开销。
- 字段扩展后会影响程序预期。

### 4. 查询指定列

```sql
SELECT student_no, name, major
FROM students;
```

这比 `SELECT *` 更明确，也更适合实际应用。

### 5. 使用别名

```sql
SELECT student_no AS 学号, name AS 姓名
FROM students;
```

别名可以让结果更易读，也有助于简化复杂表达式。

### 6. WHERE 条件过滤

`WHERE` 用于筛选满足条件的记录。

例如。

```sql
SELECT *
FROM students
WHERE admission_year = 2024;
```

### 7. 常见比较运算符

- `=` 等于
- `<>` 或 `!=` 不等于
- `>` 大于
- `<` 小于
- `>=` 大于等于
- `<=` 小于等于

### 8. 逻辑运算符

- `AND`
- `OR`
- `NOT`

例如。

```sql
SELECT *
FROM students
WHERE major = '计算机科学与技术' AND admission_year = 2024;
```

### 9. BETWEEN、IN、LIKE、IS NULL

这些也是 SQL 中非常常见的条件表达。

#### BETWEEN

```sql
SELECT *
FROM enrollments
WHERE score BETWEEN 80 AND 90;
```

#### IN

```sql
SELECT *
FROM students
WHERE major IN ('软件工程', '信息安全');
```

#### LIKE

```sql
SELECT *
FROM students
WHERE name LIKE '张%';
```

这里 `%` 表示任意长度字符。

#### IS NULL

```sql
SELECT *
FROM enrollments
WHERE score IS NULL;
```

### 10. ORDER BY 排序

```sql
SELECT *
FROM students
ORDER BY admission_year DESC, student_no ASC;
```

默认排序通常是升序，也就是 `ASC`。

降序使用 `DESC`。

### 11. DISTINCT 去重

```sql
SELECT DISTINCT major
FROM students;
```

这条语句会返回不重复的专业列表。

### 12. LIMIT 分页

在 MySQL 中常用 `LIMIT` 控制返回条数。

```sql
SELECT *
FROM students
ORDER BY student_id
LIMIT 10;
```

如果要分页。

```sql
SELECT *
FROM students
ORDER BY student_id
LIMIT 20, 10;
```

表示跳过前 20 条，再取 10 条。

### 13. 表达式与函数

SQL 查询中不仅能直接取列，还能使用表达式。

例如。

```sql
SELECT course_name, credit * 20 AS estimated_hours
FROM courses;
```

### 14. 常见函数示例

```sql
SELECT UPPER(name) AS upper_name
FROM students;
```

```sql
SELECT CONCAT(name, '-', major) AS profile
FROM students;
```

### 15. 查询执行的大致逻辑顺序

虽然 SQL 写作顺序是 `SELECT ... FROM ... WHERE ...`，但概念上可以理解为数据库大致先处理。

- `FROM`
- `WHERE`
- `GROUP BY`
- `HAVING`
- `SELECT`
- `ORDER BY`
- `LIMIT`

这个认识对理解聚合和别名作用范围很重要。

### 16. 本章小结

DQL 基础的重点，是掌握 `SELECT` 查询的基本骨架，并能清楚区分列选择、条件过滤、排序、去重和分页分别在做什么。

## 十三、SQL 高级查询

### 1. 为什么高级查询重要

如果 SQL 只会单表 `SELECT ... FROM ... WHERE ...`，那只能处理非常简单的问题。

现实业务中，查询往往要解决更复杂的需求。

例如。

- 每个专业有多少学生。
- 哪些学生没有选任何课程。
- 哪门课的平均分最高。
- 每位学生在自己专业中成绩排名第几。

这些问题需要用到聚合、分组、连接、子查询甚至窗口函数。

### 2. 聚合函数

SQL 中最常见的聚合函数包括。

- `COUNT()`
- `SUM()`
- `AVG()`
- `MAX()`
- `MIN()`

它们用于对一组记录做统计。

例如统计学生总数。

```sql
SELECT COUNT(*) AS total_students
FROM students;
```

### 3. COUNT 的常见写法区别

#### `COUNT(*)`

统计记录条数。

#### `COUNT(列名)`

统计该列非 NULL 的记录数。

例如。

```sql
SELECT COUNT(score) AS scored_count
FROM enrollments;
```

如果某些成绩为空，这里不会把 NULL 统计进去。

### 4. GROUP BY

`GROUP BY` 用于按某些列把记录分组，然后对每组进行聚合计算。

例如按专业统计人数。

```sql
SELECT major, COUNT(*) AS student_count
FROM students
GROUP BY major;
```

### 5. HAVING

`HAVING` 用于对分组后的结果再次筛选。

例如只保留人数大于 10 的专业。

```sql
SELECT major, COUNT(*) AS student_count
FROM students
GROUP BY major
HAVING COUNT(*) > 10;
```

### 6. WHERE 和 HAVING 的区别

这是 SQL 学习中非常容易混淆的点。

- `WHERE` 在分组之前筛选原始记录。
- `HAVING` 在分组之后筛选分组结果。

可以粗略理解为。

- `WHERE` 管单条记录。
- `HAVING` 管一组记录统计后的结果。

### 7. 多表连接为什么重要

规范化设计后，业务数据往往分散在多个表中。

例如。

- 学生信息在 `students`。
- 课程信息在 `courses`。
- 选课关系在 `enrollments`。

要得到“某学生选了什么课”，就必须连接这些表。

### 8. INNER JOIN

内连接返回满足连接条件的匹配记录。

例如查询学生及其所选课程。

```sql
SELECT s.name, c.course_name, e.score
FROM enrollments e
INNER JOIN students s ON e.student_id = s.student_id
INNER JOIN courses c ON e.course_id = c.course_id;
```

### 9. LEFT JOIN

左连接会保留左表全部记录，即使右表没有匹配，也会返回左表数据，右表部分为 NULL。

例如查询所有学生及其选课情况。

```sql
SELECT s.name, c.course_name
FROM students s
LEFT JOIN enrollments e ON s.student_id = e.student_id
LEFT JOIN courses c ON e.course_id = c.course_id;
```

如果某学生还没有选课，这条语句仍会把学生列出来。

### 10. RIGHT JOIN 与 FULL JOIN

右连接和全连接在不同数据库中的支持情况不同。

本科阶段重点通常先掌握 `INNER JOIN` 和 `LEFT JOIN`。

因为它们最常见，也最足够解决绝大多数问题。

### 11. 连接查询时为什么建议写表别名

因为多表查询中列名可能重名。

例如很多表都有 `id`、`name`、`created_at`。

写别名可以。

- 让 SQL 更短。
- 避免歧义。
- 让连接关系更清晰。

### 12. 子查询

子查询，就是把一个查询的结果作为另一个查询的一部分。

例如查找平均分大于全校平均分的课程。

```sql
SELECT course_id, AVG(score) AS avg_score
FROM enrollments
GROUP BY course_id
HAVING AVG(score) > (
	SELECT AVG(score) FROM enrollments
);
```

### 13. 标量子查询与集合子查询

如果子查询结果是单个值，常称为标量子查询。

如果子查询返回一组值，常要结合 `IN`、`ANY`、`ALL` 等语法使用。

例如。

```sql
SELECT *
FROM students
WHERE student_id IN (
	SELECT student_id
	FROM enrollments
	WHERE score >= 90
);
```

### 14. EXISTS

`EXISTS` 常用于判断某个子查询是否至少返回一条记录。

例如查找已经选课的学生。

```sql
SELECT *
FROM students s
WHERE EXISTS (
	SELECT 1
	FROM enrollments e
	WHERE e.student_id = s.student_id
);
```

### 15. IN 和 EXISTS 怎么理解

在逻辑上，它们都能表达“是否存在匹配关系”。

但数据库优化器在不同场景下可能采用不同执行策略。

学习阶段更重要的是先理解语义。

- `IN` 更像“是否属于这个集合”。
- `EXISTS` 更像“是否存在满足条件的记录”。

### 16. UNION 与 UNION ALL

两者都可用于拼接多个查询结果。

#### `UNION`

会自动去重。

#### `UNION ALL`

不去重，通常开销更小。

例如。

```sql
SELECT name FROM students
UNION
SELECT course_name FROM courses;
```

### 17. CASE WHEN

`CASE WHEN` 可以在查询中做条件表达式，非常实用。

例如把成绩分等级。

```sql
SELECT student_id, course_id, score,
	   CASE
		   WHEN score >= 90 THEN '优秀'
		   WHEN score >= 80 THEN '良好'
		   WHEN score >= 60 THEN '及格'
		   ELSE '不及格'
	   END AS level
FROM enrollments;
```

### 18. 窗口函数基础

窗口函数是 SQL 中非常强大的高级能力。

它可以在“保留每一行”的同时，对某个分区内的数据进行排名、累计、对比等操作。

### 19. 一个窗口函数示例

假设要查询每个专业内部，学生按入学年份排序的编号。

```sql
SELECT student_id, name, major, admission_year,
	   ROW_NUMBER() OVER (PARTITION BY major ORDER BY admission_year) AS rn
FROM students;
```

这里的含义是。

- `PARTITION BY major` 表示按专业分区。
- `ORDER BY admission_year` 表示分区内排序。
- `ROW_NUMBER()` 表示给每行编号。

### 20. 为什么窗口函数重要

没有窗口函数时，很多排名、累计、环比问题往往要写复杂子查询。

有了窗口函数，表达会更清晰。

### 21. 一个综合查询示例

假设查询“每门课平均分大于 80 且选课人数不少于 5 人的课程”。

```sql
SELECT c.course_name,
	   COUNT(*) AS student_count,
	   AVG(e.score) AS avg_score
FROM enrollments e
JOIN courses c ON e.course_id = c.course_id
WHERE e.score IS NOT NULL
GROUP BY c.course_id, c.course_name
HAVING COUNT(*) >= 5 AND AVG(e.score) > 80
ORDER BY avg_score DESC;
```

这条语句综合用了。

- 连接
- 过滤
- 分组
- 聚合
- 分组结果筛选
- 排序

### 22. 本章小结

高级查询的核心，不是记住所有语法，而是理解 SQL 如何在多个表和多层条件中表达“筛选、组合、统计与比较”。

只要把连接、分组、子查询和窗口函数的作用分清，高级查询就会清晰很多。

## 十四、约束、视图与模式对象

### 1. 为什么数据库不应该只靠应用程序保证正确性

很多初学者会觉得。

“只要应用代码写得严谨，数据库里不加太多约束也可以。”

这种想法风险很大。

因为。

- 不同应用可能都访问同一数据库。
- 应用代码可能有漏洞。
- 批量导入脚本可能绕过业务层。

数据库自身的约束，是最后一道非常重要的防线。

### 2. 主键约束

主键约束确保一张表中的每条记录有唯一标识，且通常不能为空。

例如。

```sql
student_id INT PRIMARY KEY AUTO_INCREMENT
```

### 3. 外键约束

外键约束用于保持表之间的参照关系。

例如 `enrollments.student_id` 必须引用 `students.student_id`。

```sql
CONSTRAINT fk_enrollment_student
FOREIGN KEY (student_id) REFERENCES students(student_id)
```

### 4. UNIQUE 约束

唯一约束保证某列或某列组合的值不能重复。

例如学号、邮箱、课程编号通常都适合加唯一约束。

```sql
email VARCHAR(100) UNIQUE
```

### 5. NOT NULL 约束

非空约束表示该列必须提供有效值。

例如学生姓名通常不能为 NULL。

### 6. CHECK 约束

检查约束用于限制字段取值范围或条件。

例如。

```sql
score DECIMAL(5, 2) CHECK (score >= 0 AND score <= 100)
```

注意不同数据库版本对 `CHECK` 的支持程度可能不同，但概念本身很重要。

### 7. DEFAULT 默认值

默认值能减少插入数据时的重复输入，也有助于保证状态完整。

例如。

```sql
created_at DATETIME DEFAULT CURRENT_TIMESTAMP
```

### 8. 视图 View

视图可以理解为“基于查询结果定义的虚拟表”。

例如你可以创建一个视图，专门展示学生和课程成绩信息。

```sql
CREATE VIEW student_course_scores AS
SELECT s.student_no, s.name, c.course_name, e.score
FROM enrollments e
JOIN students s ON e.student_id = s.student_id
JOIN courses c ON e.course_id = c.course_id;
```

### 9. 视图的价值

视图至少有几个典型价值。

- 简化复杂查询。
- 提供稳定访问接口。
- 屏蔽底层部分结构细节。
- 做一定程度的权限隔离。

### 10. 视图和真实表的区别

视图通常不直接存储数据本体，而是存储查询定义。

查询视图时，本质上是在执行其定义的查询。

某些数据库支持物化视图，但那又是更进一步的对象类型。

### 11. 索引作为模式对象

索引虽然主要在优化章节展开，但从对象角度看，它也是数据库中非常重要的模式对象。

常见创建语法如下。

```sql
CREATE INDEX idx_students_major
ON students(major);
```

### 12. 触发器与存储过程的基本理解

本科阶段通常至少要知道这两个概念。

#### 触发器 Trigger

某些事件发生时自动执行的数据库逻辑。

例如插入成绩后自动写审计表。

#### 存储过程 Procedure

保存在数据库中的一组可复用 SQL 逻辑。

### 13. 为什么不建议一开始把所有逻辑都塞进存储过程

因为这会带来维护复杂度和数据库层耦合问题。

学习阶段知道其作用即可。

实际项目中是否大量使用，要结合团队风格和系统架构评估。

### 14. 本章小结

约束、视图、索引、触发器、存储过程这些对象共同构成了数据库的模式层能力。

它们的价值不是“让数据库更复杂”，而是帮助系统更稳、更清晰、更易管理。

## 十五、索引与查询优化基础

### 1. 为什么查询会慢

如果一张表很小，即使全表扫描也不一定慢。

但数据量一旦增大，查询性能会迅速成为问题。

导致查询慢的原因可能很多。

- 没有索引。
- 索引设计不合理。
- SQL 写法使索引失效。
- 需要扫描的数据太多。
- 回表开销高。
- 磁盘或缓存命中情况不好。

### 2. 索引是什么

可以把索引理解为数据库为某些列建立的“高效查找辅助结构”。

如果没有索引，数据库查找某个值时可能只能从头到尾扫描。

有了索引，就能更快定位到可能位置。

### 3. 为什么索引能加速查询

索引本质上是用空间换时间。

它额外保存了一份更适合查找的结构。

这样在查询时可以少扫描大量无关数据。

### 4. B+ 树索引的基本思想

关系型数据库中，最经典和最重要的索引结构之一就是 B+ 树。

本科阶段不一定要深入到每个页分裂细节，但至少要理解。

- 它是多路平衡搜索树。
- 高度通常较低。
- 很适合磁盘或页式存储环境。
- 范围查询能力强。

### 5. 聚簇索引与二级索引

这两个概念在 MySQL InnoDB 中非常常见。

#### 聚簇索引

表数据本身按主键组织。

#### 二级索引

对其他列建立的辅助索引。

通过二级索引找到主键后，往往还要再回到主键组织的数据里取完整记录。

这就涉及回表。

### 6. 回表是什么

如果查询字段不完全包含在二级索引中，数据库通过二级索引定位到主键后，还要再去主数据位置取其他列。

这就叫回表。

回表次数多时，性能可能明显受影响。

### 7. 覆盖索引

如果查询需要的列都已经在索引中，那么数据库可能无需回表。

这常被称为覆盖索引。

例如只查询索引列本身时，效率可能更好。

### 8. 什么情况下应该建索引

一般来说，下面这些场景常适合建索引。

- 高频查询条件列。
- 连接条件列。
- 排序和分组常用列。
- 唯一性强的列。

### 9. 什么情况下不应盲目建索引

索引不是越多越好。

因为索引也有代价。

- 占空间。
- 插入、更新、删除时需要维护索引。
- 过多索引可能让优化器选择复杂。

### 10. 索引失效的常见原因

这是 SQL 优化中非常高频的知识点。

常见情况包括。

- 在索引列上做函数运算。
- 模糊匹配以 `%` 开头。
- 隐式类型转换。
- 条件选择性太差。
- 组合索引没有合理使用最左前缀。

### 11. 一个索引失效例子

假设 `name` 上有索引。

如果写成。

```sql
SELECT *
FROM students
WHERE UPPER(name) = 'ZHANGSAN';
```

数据库可能无法直接利用普通索引，因为对列做了函数处理。

### 12. 最左前缀原则

对于组合索引 `(a, b, c)`，查询时通常只有合理利用最左边开始的前缀部分，索引效果才较好。

这也是为什么建组合索引时必须考虑列顺序。

### 13. EXPLAIN 是什么

`EXPLAIN` 用于查看数据库如何执行一条查询。

它能帮助你理解。

- 是否使用了索引。
- 扫描了多少行。
- 使用了什么连接方式。
- 是否进行了临时表或文件排序。

### 14. EXPLAIN 示例

```sql
EXPLAIN
SELECT *
FROM students
WHERE major = '软件工程';
```

学习阶段重点不是死记每个字段，而是先看。

- 走没走索引。
- 扫描范围大不大。
- 有没有明显不合理的全表扫描。

### 15. SQL 优化的常见思路

- 只查需要的列，避免无意义 `SELECT *`。
- 给高频过滤列和关联列建立合理索引。
- 避免在索引列上做不必要函数处理。
- 尽量减少大范围扫描。
- 结合执行计划而不是凭感觉优化。

### 16. 本章小结

索引优化的核心，不是机械背诵术语，而是理解查询为什么慢、索引为什么可能帮上忙、索引为什么也会带来代价。

## 十六、事务与 ACID

### 1. 什么是事务

事务，是一组逻辑上不可分割的数据库操作单元。

这些操作要么全部成功，要么全部失败回滚。

### 2. 为什么事务重要

最经典的例子是银行转账。

从 A 账户扣款 100 元，给 B 账户加款 100 元。

这两个步骤必须视为一个整体。

如果只扣款成功、加款失败，系统就会出错。

### 3. 事务的边界

事务边界定义的是。

- 哪些 SQL 属于同一个原子操作。
- 成功时何时提交。
- 失败时何时回滚。

### 4. ACID

事务最经典的四个特性是 ACID。

#### 原子性 Atomicity

事务中的操作要么全部完成，要么全部不完成。

#### 一致性 Consistency

事务执行前后，数据库应保持一致的合法状态。

#### 隔离性 Isolation

并发执行的多个事务之间不应互相产生不可接受的干扰。

#### 持久性 Durability

事务一旦提交，其结果应被可靠保存，即使系统崩溃也不能丢失。

### 5. 事务控制的基本语法

```sql
BEGIN;

UPDATE accounts
SET balance = balance - 100
WHERE account_id = 1;

UPDATE accounts
SET balance = balance + 100
WHERE account_id = 2;

COMMIT;
```

如果中途出错，可以。

```sql
ROLLBACK;
```

### 6. 一个转账示例

```sql
BEGIN;

UPDATE accounts
SET balance = balance - 500
WHERE account_id = 1001;

UPDATE accounts
SET balance = balance + 500
WHERE account_id = 2001;

COMMIT;
```

如果第二步失败，而第一步已经执行，就必须回滚。

### 7. 为什么事务不是“把多条 SQL 放一起”这么简单

事务背后其实依赖。

- 锁或 MVCC 等并发控制机制。
- 日志和恢复机制。
- 崩溃恢复策略。

所以事务是数据库系统层非常核心的能力，不只是语法层功能。

### 8. 自动提交 Autocommit

很多数据库默认开启自动提交。

这意味着每条单独 SQL 执行完后会自动成为一个事务并提交。

而在需要多个语句构成一个整体时，就需要显式控制事务边界。

### 9. 本章小结

事务的本质，是让数据库能可靠地处理一组必须整体成功或整体失败的操作。ACID 则是理解事务价值的核心框架。

## 十七、并发控制与隔离级别

### 1. 为什么数据库一定要考虑并发

现实数据库几乎不可能只有一个用户顺序访问。

通常会有很多事务同时运行。

如果没有并发控制，就会出现数据读写冲突。

### 2. 脏读

脏读指一个事务读到了另一个事务尚未提交的数据。

如果后者回滚，那么前者读到的其实是无效数据。

### 3. 不可重复读

不可重复读指同一个事务中，多次读取同一行记录，结果却不同。

原因通常是另一个事务在中间提交了更新。

### 4. 幻读

幻读指同一个事务中，多次按同一条件查询记录集合，结果条数不同。

原因通常是另一个事务插入或删除了满足条件的新记录。

### 5. 为什么这些问题不能简单忽略

因为它们可能直接影响业务正确性。

例如库存检查、余额计算、报名人数控制等场景，对一致性要求都很高。

### 6. 锁机制的基本思想

锁的核心思路是。

- 某个事务在访问数据时先加锁。
- 其他事务根据锁类型决定是否等待。

这有助于避免并发冲突，但也会降低并发度。

### 7. 共享锁与排他锁

直观理解。

- 共享锁偏向读。
- 排他锁偏向写。

多个事务通常可以同时持有共享锁。

但排他锁一般要求独占。

### 8. 隔离级别

为了在一致性和性能之间做平衡，数据库通常提供不同隔离级别。

常见包括。

- Read Uncommitted
- Read Committed
- Repeatable Read
- Serializable

### 9. 各隔离级别的直观理解

#### Read Uncommitted

隔离最弱，甚至可能读到未提交数据。

#### Read Committed

只能读到已提交数据，能避免脏读。

#### Repeatable Read

同一事务中多次读取同一行结果一致，通常能避免脏读和不可重复读。

#### Serializable

隔离最强，事务效果接近串行执行，但并发性能通常最低。

### 10. MVCC 的直观理解

很多现代数据库不仅靠锁，还会用多版本并发控制，也就是 MVCC。

它的核心思想可以简单理解为。

- 不同事务看到数据的不同版本。
- 读操作尽量不阻塞写。

这能在很多场景下提升并发性能。

### 11. 并发控制的本质平衡

一致性越强，通常限制越多，并发性能可能越低。

并发越高，控制越宽松，出现异常读取的风险可能越大。

数据库设计和业务系统设计，很多时候都在做这个平衡。

### 12. 本章小结

并发控制的核心问题，是在多个事务同时执行时，如何既尽量保证结果正确，又尽量不让系统变得过慢。

理解脏读、不可重复读、幻读和隔离级别，是数据库系统课程中的关键一环。

## 十八、数据库恢复与日志

### 1. 为什么数据库必须考虑恢复

数据库系统运行在真实环境中，故障不可避免。

例如。

- 断电。
- 系统崩溃。
- 磁盘损坏。
- 程序异常退出。

如果没有恢复机制，提交过的数据可能丢失，数据库一致性也可能被破坏。

### 2. 常见故障类型

本科阶段常见分类可以先掌握。

- 事务内部故障。
- 系统故障。
- 介质故障。

### 3. 日志的作用

数据库日志用于记录事务对数据所做的重要操作信息。

它是数据库恢复能力的核心基础之一。

### 4. Undo 与 Redo 的基本思想

#### Undo

用于撤销尚未提交事务的影响。

#### Redo

用于重做已经提交但还未来得及写入数据文件的操作。

### 5. 为什么既需要 Undo 又需要 Redo

因为数据库既要解决。

- 未提交事务不能留下脏影响。
- 已提交事务不能因为崩溃而丢失。

这分别对应撤销与重做需求。

### 6. 检查点 Checkpoint

检查点可以理解为恢复过程中的一个重要同步标记。

有了检查点，数据库恢复时不必每次都从最早日志开始扫描，有助于提高恢复效率。

### 7. 一个崩溃恢复的直观过程

系统重启后，数据库可能会做这样的事。

1. 分析哪些事务已经提交，哪些没有提交。
2. 对未提交事务做 Undo。
3. 对已提交但未完全落盘的事务做 Redo。

### 8. 本章小结

恢复机制的本质，是让数据库在故障之后依然尽量回到一个一致、可靠的状态。日志、Undo、Redo 和检查点是这套机制的核心关键词。

## 十九、关系规范化与模式设计

### 1. 为什么数据库设计不能只看“能存下数据”

如果表设计得不好，即使暂时能存数据，也可能带来很多长期问题。

例如。

- 数据冗余严重。
- 更新麻烦。
- 插入异常。
- 删除异常。

这就是规范化理论要解决的问题。

### 2. 冗余与异常

假设把学生信息、课程信息、教师信息、成绩信息全放在一张大表里。

那么同一个课程名、教师名可能在很多行重复出现。

这样就会产生冗余。

进一步还会出现异常。

#### 更新异常

修改教师姓名时，要改很多行，漏改就会不一致。

#### 插入异常

如果某门新课程还没有学生选，就可能无法单独插入课程信息。

#### 删除异常

如果删除某门课程最后一条选课记录，课程本身信息可能也一并丢失。

### 3. 函数依赖

规范化理论的重要基础是函数依赖。

直观理解是。

如果由属性集 A 能唯一决定属性集 B，就说 B 函数依赖于 A。

例如学号通常可以唯一决定学生姓名、专业、年级等。

### 4. 第一范式 1NF

第一范式强调每个字段都应是原子值，不能再拆。

例如一个“课程列表”字段里直接存 `数据库,操作系统,网络` 这样的多值串，就不符合规范化思想。

### 5. 第二范式 2NF

第二范式在满足 1NF 的基础上，要求非主属性完全依赖于主键，而不是只依赖主键的一部分。

它主要针对复合主键场景。

### 6. 第三范式 3NF

第三范式在满足 2NF 的基础上，要求非主属性不应传递依赖于主键。

也就是说，尽量避免“主键决定 A，A 再决定 B”这种层层依赖导致的冗余。

### 7. BCNF 的基本理解

BCNF 是比 3NF 更强的规范形式。

本科阶段通常只需理解，它是对函数依赖约束更严格的一种设计要求。

### 8. 规范化不是越高越绝对好

这是一个非常重要的观点。

高规范化有助于减少冗余和异常。

但过度拆分也可能导致。

- 查询要连接很多表。
- 复杂度升高。
- 某些读性能下降。

因此在工程实践中，有时会做一定反规范化。

### 9. 反规范化

反规范化是基于性能和实际需求，有意识地保留部分冗余，以减少连接、提升查询效率。

但反规范化必须非常谨慎，因为它会增加维护一致性的难度。

### 10. 本章小结

规范化设计的核心，不是机械背范式定义，而是理解为什么糟糕的表结构会带来冗余和异常，以及为什么良好设计能让数据库更稳定、更容易维护。

## 二十、数据库安全与管理

### 1. 为什么数据库安全不能只靠应用层

有些系统会认为，只要前端和后端接口做好权限控制，数据库就不需要太关心权限问题。

这是不够安全的。

因为数据库往往有多种访问路径。

- 应用程序。
- 运维脚本。
- 数据同步任务。
- 管理员工具。

如果数据库层完全不做权限控制，风险会很大。

### 2. 用户与角色

数据库通常支持。

- 用户 User
- 角色 Role

用户代表具体登录身份。

角色用于组织一组权限，便于批量授权。

### 3. 常见权限管理语句

#### 创建用户

```sql
CREATE USER 'db_reader'@'localhost' IDENTIFIED BY 'StrongPassword123';
```

#### 授权

```sql
GRANT SELECT ON university_db.* TO 'db_reader'@'localhost';
```

#### 撤权

```sql
REVOKE SELECT ON university_db.* FROM 'db_reader'@'localhost';
```

### 4. 最小权限原则

权限管理最重要的原则之一是最小权限。

也就是。

- 只给真正需要的权限。
- 只开放真正需要的对象范围。

不要为了图省事就直接给所有权限。

### 5. 备份与恢复

数据库安全不仅是“防攻击”，也包括“防丢数据”。

因此备份非常关键。

常见备份方式包括。

- 逻辑备份，例如导出 SQL。
- 物理备份，例如复制数据文件或使用备份工具。

### 6. 为什么有备份还要做恢复演练

因为“备份成功”不等于“恢复一定可用”。

如果从未验证恢复过程，真正出问题时可能发现备份不可用或恢复步骤不完整。

### 7. 审计与日志

数据库日志和审计记录有助于回答下面这些问题。

- 谁在什么时候访问了数据。
- 谁执行了高风险语句。
- 故障发生前系统经历了什么。

### 8. 本章小结

数据库管理不仅是建表和写 SQL，还包括权限、安全、备份、恢复和审计。一个能用的数据库系统，不等于一个可长期稳定管理的数据库系统。

## 二十一、非关系型数据库与现代数据系统

### 1. 为什么会出现 NoSQL

关系型数据库非常强大，但它并不是所有场景下的唯一答案。

随着互联网应用、海量数据和高并发场景发展，一些需求推动了 NoSQL 的出现。

例如。

- 超大规模数据。
- 极高并发访问。
- 非固定结构数据。
- 图关系问题。

### 2. NoSQL 不是“不需要 SQL”这么简单

NoSQL 更准确地说，是一组不完全遵循传统关系模型的数据库类型统称。

它们可能包括。

- 键值数据库。
- 文档数据库。
- 列族数据库。
- 图数据库。

### 3. 键值数据库

典型特征是。

- 用 key 找 value。
- 结构简单。
- 访问速度快。

例如 Redis 很多场景下就可以看作典型键值系统。

### 4. 文档数据库

文档数据库通常以 JSON 类结构保存数据。

它适合。

- 字段结构灵活。
- 迭代频繁。
- 半结构化数据。

MongoDB 是典型代表。

### 5. 列族数据库

列族数据库更适合海量数据和分布式场景。

例如 HBase、Cassandra。

本科阶段知道其定位即可。

### 6. 图数据库

图数据库擅长处理关系网络问题。

例如。

- 社交图谱。
- 知识图谱。
- 路径分析。

### 7. CAP 的直观理解

CAP 理论常用于讨论分布式系统中的取舍。

- Consistency 一致性
- Availability 可用性
- Partition Tolerance 分区容错性

它的细节并不简单，但本科阶段至少要知道。

在分布式环境中，不同系统常常需要在一致性和可用性之间做权衡。

### 8. 最终一致性

一些 NoSQL 系统不追求每个时刻都强一致，而是接受一段时间后最终达成一致。

这在某些高并发分布式场景中是有价值的。

### 9. 为什么现代系统通常不是“只选一种数据库”

因为不同数据库擅长的问题不同。

很多真实系统会组合使用。

- MySQL 存核心交易数据。
- Redis 做缓存。
- Elasticsearch 做搜索。
- 图数据库处理复杂关系。

### 10. 本章小结

理解 NoSQL 的重点，不是把它和关系数据库对立起来，而是理解不同数据模型各自适合解决什么问题。

## 二十二、数据库实战环境与学习方式

### 1. 为什么数据库学习一定要做实验

数据库和数学证明不同。

很多概念如果不在真实环境中执行 SQL，很难真正建立直觉。

例如。

- 表结构设计到底长什么样。
- 外键报错时会发生什么。
- 分组查询结果具体如何呈现。
- 事务回滚到底是什么效果。

### 2. 推荐的实验环境

对于本科阶段学习，比较推荐两类环境。

- MySQL 8
- PostgreSQL

如果你更重视通用教学和资料丰富度，MySQL 很适合入门。

### 3. 为什么推荐用 Docker

Docker 搭建数据库环境有几个优点。

- 安装快。
- 不容易污染本机环境。
- 便于重建实验环境。
- 适合不同项目隔离。

### 4. 使用 Docker 启动 MySQL 示例

```bash
docker run --name mysql-lab \
  -e MYSQL_ROOT_PASSWORD=123456 \
  -p 3306:3306 \
  -d mysql:8.0
```

### 5. 连接数据库

命令行连接示例。

```bash
mysql -u root -p
```

输入密码后即可进入 MySQL 命令行环境。

### 6. 图形化工具

如果不想一直使用命令行，可以使用图形化工具。

常见工具包括。

- DBeaver
- Navicat
- MySQL Workbench

图形化工具的价值在于。

- 便于浏览表结构。
- 便于查看数据。
- 便于执行和保存 SQL 脚本。

### 7. 如何准备测试数据

学习 SQL 时，如果数据太少，很多查询效果看不出来。

建议为每张表准备至少十几条到几十条测试数据。

这样更容易观察。

- 分组效果。
- 连接结果。
- 排序和分页。
- NULL 值处理。

### 8. 推荐的学习方式

建议按这样的节奏练习。

1. 先手写建表语句。
2. 再插入测试数据。
3. 再写单表查询。
4. 再写多表连接和统计查询。
5. 再尝试事务和索引分析。

### 9. 本章小结

数据库学习如果只停留在看书和背语法，很难真正掌握。搭建实验环境、写 SQL、看结果，是建立理解的关键步骤。

## 二十三、综合实战：选课管理系统数据库设计与 SQL 练习

### 1. 实战目标

这一部分用一个简单但典型的“选课管理系统”来把前面的知识串起来。

我们要实现下面几件事。

- 建立数据库和表结构。
- 插入测试数据。
- 完成常见单表和多表查询。
- 做统计分析。
- 演示事务。
- 演示索引与执行计划。

### 2. 需求分析

系统至少包含下面几个核心对象。

- 学生 students
- 教师 teachers
- 课程 courses
- 开课表 classes
- 选课表 enrollments

其中。

- 一个教师可以教授多门课。
- 一门课程可以开设多个班次。
- 一个学生可以选多个开课班次。
- 一个开课班次可以被多个学生选择。

### 3. 建库

```sql
CREATE DATABASE course_system
CHARACTER SET utf8mb4
COLLATE utf8mb4_unicode_ci;

USE course_system;
```

### 4. 建表：学生表

```sql
CREATE TABLE students (
	student_id INT PRIMARY KEY AUTO_INCREMENT,
	student_no VARCHAR(20) NOT NULL UNIQUE,
	name VARCHAR(50) NOT NULL,
	gender CHAR(1),
	major VARCHAR(100) NOT NULL,
	grade INT NOT NULL,
	created_at DATETIME DEFAULT CURRENT_TIMESTAMP
);
```

### 5. 建表：教师表

```sql
CREATE TABLE teachers (
	teacher_id INT PRIMARY KEY AUTO_INCREMENT,
	teacher_no VARCHAR(20) NOT NULL UNIQUE,
	name VARCHAR(50) NOT NULL,
	title VARCHAR(50),
	department VARCHAR(100)
);
```

### 6. 建表：课程表

```sql
CREATE TABLE courses (
	course_id INT PRIMARY KEY AUTO_INCREMENT,
	course_code VARCHAR(20) NOT NULL UNIQUE,
	course_name VARCHAR(100) NOT NULL,
	credit DECIMAL(3, 1) NOT NULL,
	department VARCHAR(100)
);
```

### 7. 建表：开课表

```sql
CREATE TABLE classes (
	class_id INT PRIMARY KEY AUTO_INCREMENT,
	course_id INT NOT NULL,
	teacher_id INT NOT NULL,
	semester VARCHAR(20) NOT NULL,
	capacity INT NOT NULL,
	room VARCHAR(50),
	CONSTRAINT fk_classes_course
		FOREIGN KEY (course_id) REFERENCES courses(course_id),
	CONSTRAINT fk_classes_teacher
		FOREIGN KEY (teacher_id) REFERENCES teachers(teacher_id)
);
```

### 8. 建表：选课记录表

```sql
CREATE TABLE enrollments (
	enrollment_id INT PRIMARY KEY AUTO_INCREMENT,
	student_id INT NOT NULL,
	class_id INT NOT NULL,
	score DECIMAL(5, 2),
	selected_at DATETIME DEFAULT CURRENT_TIMESTAMP,
	CONSTRAINT fk_enrollments_student
		FOREIGN KEY (student_id) REFERENCES students(student_id),
	CONSTRAINT fk_enrollments_class
		FOREIGN KEY (class_id) REFERENCES classes(class_id),
	CONSTRAINT uk_student_class UNIQUE (student_id, class_id)
);
```

这里的联合唯一约束表示。

- 同一个学生不能重复选同一个开课班次。

### 16. 多表查询：查询每个学生所选课程

```sql
SELECT s.name AS student_name,
	   c.course_name,
	   e.score
FROM enrollments e
JOIN students s ON e.student_id = s.student_id
JOIN classes cl ON e.class_id = cl.class_id
JOIN courses c ON cl.course_id = c.course_id
ORDER BY s.student_id, c.course_id;
```

### 17. 统计查询：按专业统计学生人数

```sql
SELECT major, COUNT(*) AS student_count
FROM students
GROUP BY major
ORDER BY student_count DESC;
```

### 18. 统计查询：每门课平均分

```sql
SELECT c.course_name,
	   AVG(e.score) AS avg_score,
	   COUNT(*) AS selected_count
FROM enrollments e
JOIN classes cl ON e.class_id = cl.class_id
JOIN courses c ON cl.course_id = c.course_id
GROUP BY c.course_id, c.course_name;
```

### 19. 查询没有选课的学生

```sql
SELECT s.student_id, s.name
FROM students s
LEFT JOIN enrollments e ON s.student_id = e.student_id
WHERE e.enrollment_id IS NULL;
```

这个查询很典型。

它利用左连接保留所有学生，再用 `IS NULL` 找出没有匹配选课记录的人。

### 20. 子查询：查询平均分高于全校平均分的学生

```sql
SELECT s.student_id, s.name, AVG(e.score) AS student_avg
FROM students s
JOIN enrollments e ON s.student_id = e.student_id
GROUP BY s.student_id, s.name
HAVING AVG(e.score) > (
	SELECT AVG(score) FROM enrollments
);
```

### 21. CASE WHEN：给成绩打等级

```sql
SELECT s.name,
	   c.course_name,
	   e.score,
	   CASE
		   WHEN e.score >= 90 THEN '优秀'
		   WHEN e.score >= 80 THEN '良好'
		   WHEN e.score >= 60 THEN '及格'
		   ELSE '不及格'
	   END AS score_level
FROM enrollments e
JOIN students s ON e.student_id = s.student_id
JOIN classes cl ON e.class_id = cl.class_id
JOIN courses c ON cl.course_id = c.course_id;
```

### 22. 窗口函数：课程内成绩排名

```sql
SELECT c.course_name,
	   s.name,
	   e.score,
	   RANK() OVER (PARTITION BY c.course_id ORDER BY e.score DESC) AS rank_in_course
FROM enrollments e
JOIN students s ON e.student_id = s.student_id
JOIN classes cl ON e.class_id = cl.class_id
JOIN courses c ON cl.course_id = c.course_id;
```

### 23. 事务实战：模拟选课

假设某个班次容量有限，学生选课时要保证。

- 不能超容量。
- 插入成功后才算完成。

一个教学化简后的事务示例如下。

```sql
BEGIN;

SELECT capacity
FROM classes
WHERE class_id = 1;

SELECT COUNT(*) AS current_count
FROM enrollments
WHERE class_id = 1;

INSERT INTO enrollments (student_id, class_id, score)
VALUES (5, 1, NULL);

COMMIT;
```

现实中这类逻辑通常还要结合锁、约束或应用层判断，防止并发下超选。

### 24. 索引实战：为查询条件建立索引

假设经常按专业查询学生。

可以创建索引。

```sql
CREATE INDEX idx_students_major
ON students(major);
```

### 25. 执行计划实战

```sql
EXPLAIN
SELECT *
FROM students
WHERE major = '计算机科学与技术';
```

观察执行计划时，可以重点看。

- 是否走索引。
- 扫描行数是否合理。
- 是否仍在做全表扫描。

### 26. 实战中最容易出的问题

常见问题包括。

- 建表时主键和外键没想清楚。
- 测试数据太少，看不出分组和连接效果。
- 多表连接时把连接条件写错，导致笛卡尔积。
- 忘记用事务包裹多步修改。
- 索引建了却没有真正提升查询效果。

### 27. 如何用这套实战继续扩展

你可以继续增加下面这些功能练习。

- 成绩表与补考记录。
- 学院与专业维表。
- 课程容量控制。
- 学分统计。
- 选课时间窗口控制。
- 逻辑删除和审计表。

### 28. 本章小结

这组实战的价值，不是搭一个复杂系统，而是把数据库设计、SQL 查询、事务和索引这些概念真正串起来。只要你能完整把这个案例跑通，数据库学习就会从“看懂概念”进入“能够落地”的阶段。

## 二十四、如何建立数据库系统的整体图景

### 1. 从“会写 SQL”到“理解数据库系统”

很多人学习数据库时，会停留在会写一些 SQL 语句的阶段。

这当然是必要的，但还不够。

真正的数据库能力通常包含三个层次。

- 第一层，会写基本 SQL。
- 第二层，会设计合理表结构和索引。
- 第三层，理解事务、并发、恢复、安全和性能优化这些系统机制。

### 2. 一张数据库知识总图

```text
业务需求
   |
概念建模：实体、属性、联系
   |
逻辑设计：关系模式、键、约束、规范化
   |
SQL 实现：DDL / DML / DQL / DCL / TCL
   |
系统机制：事务、并发控制、恢复、索引优化、安全
   |
工程实践：部署、监控、备份、调优、实战案例
```

### 3. 推荐学习顺序

对于本科阶段，比较合理的学习顺序通常是。

1. 先理解数据库为什么出现。
2. 再理解关系模型、键与约束。
3. 然后学习 SQL 基础和高级查询。
4. 再学事务、隔离级别和恢复。
5. 最后理解规范化、索引优化、安全和 NoSQL 扩展。

### 4. 理论和实践如何结合

如果只学理论，容易觉得数据库很多内容很抽象。

如果只做 SQL 练习，又容易停留在“会用命令”层面。

更有效的方式是。

- 学一个理论点。
- 立刻配一个小实验验证。
- 再复盘它和整个系统的关系。

### 5. 学习中的常见误区

误区一，只记 SQL 语法，不理解关系模型。

误区二，只会写查询，不会设计表结构。

误区三，只知道 ACID 四个词，不知道它们解决什么问题。

误区四，把数据库性能问题简单归结为“机器不够好”。

### 6. 进阶方向

如果你想继续深入，可以沿下面方向扩展。

1. 更深入学习存储引擎和页式存储。
2. 深入理解 B+ 树索引和查询优化器。
3. 系统学习 MVCC、锁协议和分布式事务。
4. 深入学习 PostgreSQL 或 MySQL 的实际实现细节。
5. 对比关系型数据库与 Redis、MongoDB、Elasticsearch 等系统的差异。

### 7. 最后总结

数据库系统概论的真正价值，不只是教你写几条 SQL，而是帮你建立一套关于“数据如何被可靠管理”的系统认识。

当你真正理解了数据库系统，你会更清楚。

- 为什么表要这样设计。
- 为什么查询要这样写。
- 为什么事务和并发控制这么重要。
- 为什么性能优化不能只看一条 SQL。

这也是数据库课程在计算机专业中始终重要的根本原因。
