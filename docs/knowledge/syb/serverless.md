## 二、Serverless 是什么

### 1. Serverless 的定义

Serverless，字面意思是“无服务器”，但它并不是说真的没有服务器，而是说开发者不再直接管理服务器。底层服务器、操作系统、扩容策略、运行环境维护等工作，主要由云平台负责。

开发者更关注的是业务代码和业务逻辑本身，而不是机器配置、进程守护、扩容部署这些底层细节。

如果用一句话概括，Serverless 是一种让应用以更细粒度、更托管、更按需的方式运行的架构模式。

它通常具备以下特点：

- 基础设施由平台托管。
- 应用按请求或事件触发执行。
- 资源能够自动伸缩。
- 计费更接近实际使用量。
- 开发者更关注业务，而不是环境管理。

### 2. Serverless 与云计算、PaaS、容器的关系

Serverless 经常和其他云相关概念一起出现，它们之间既有联系，也有区别。

#### 云计算

云计算是总体模式，强调把计算、存储、网络、中间件等能力作为服务提供。

Serverless 可以看作云计算继续向上抽象的一种结果。它不是在卖一台机器，而是在卖一种“按需执行代码或直接调用能力”的方式。

#### PaaS

PaaS 是平台即服务，平台帮用户管理更多底层能力，用户主要关注应用开发和部署。

Serverless 可以理解为比传统 PaaS 更进一步的托管模式。它通常管理得更细，自动化程度更高，资源使用也更贴近实际请求。

#### 容器

容器是一种应用打包和运行技术，强调环境一致和轻量部署。

Serverless 和容器并不是对立关系。很多 Serverless 平台底层可能就是用容器来运行函数或服务。区别在于：

- 使用容器时，用户往往还需要关注镜像、实例、副本数、发布策略等。
- 使用 Serverless 时，用户更多只关注函数入口、事件来源和业务逻辑。

所以，容器更像是一种运行技术，Serverless 更像是一种使用模式和架构形态。

### 3. Serverless 想解决什么问题

传统应用部署模式通常存在以下问题：

- 即使业务访问很少，也需要长期运行服务器。
- 应用扩容和缩容需要人工配置或额外运维系统。
- 小型业务、工具类服务、定时任务等场景中，服务器常常大量空闲。
- 开发者需要花时间处理环境、部署、监控和容量规划。
- 某些场景只需要执行一个很短的逻辑，却仍然要维护完整服务。

Serverless 试图解决的核心问题是：让计算资源使用更细粒度、更自动化、更接近实际业务消耗，同时减少开发和运维的基础设施负担。

## 三、为什么会出现 Serverless

### 1. 传统部署模式的典型问题

在传统模式下，不管业务请求量高还是低，服务通常都要长期部署在线上机器上。这意味着：

- 需要提前准备容量，应对高峰时会有大量闲置资源。
- 需要持续监控服务器健康和服务状态。
- 小功能也可能需要完整的部署链路和运维流程。
- 某些任务实际上只偶尔执行，但资源却一直占用。

这种模式在大型核心系统中仍然常见，但对于事件型、突发型、短时型任务来说，并不总是经济。

### 2. Serverless 带来的核心价值

Serverless 的核心价值通常体现在以下几个方面：

- 更轻：开发者少管服务器和运行环境。
- 更快：函数或服务可以围绕单一业务能力快速交付。
- 更省：按调用计费，低频场景下成本优势明显。
- 更弹：平台自动扩缩容，应对流量变化更灵活。
- 更聚焦：团队把精力更多放在业务逻辑，而不是基础设施管理。

### 3. Serverless 适合哪些业务场景

Serverless 特别适合以下场景：

- 图片处理、音视频转码、消息消费等事件驱动场景。
- API 后端、轻量业务接口、Webhook 处理。
- 定时任务、数据同步、自动化脚本。
- 短周期、高波动、访问量不稳定的业务。
- 需要快速验证想法的小型应用或创新业务。

但它并不天然适合所有场景。长时间持续运行、低延迟要求极高、状态管理复杂的系统，往往需要谨慎评估是否采用 Serverless。

## 四、Serverless 的核心特征

理解 Serverless，关键是理解它和传统服务部署模式相比，到底改变了什么。

### 1. 无服务器运维体验

Serverless 的“无服务器”强调的是使用体验。服务器仍然存在，但开发者通常不用自己关心：

- 机器如何创建
- 操作系统如何维护
- 服务如何扩容
- 节点故障如何恢复

这些工作主要交给云平台处理。开发者看到的是函数、接口、事件和配置，而不是一台台具体的机器。

### 2. 事件驱动

Serverless 非常强调事件驱动。所谓事件驱动，就是当某个事件发生时，函数或服务才被触发执行。

常见事件包括：

- 用户发起 HTTP 请求
- 文件上传到对象存储
- 消息进入消息队列
- 数据库表发生变更
- 定时器到达某个时间点

这意味着应用不一定要一直运行等待请求，而是“有事才执行”。

### 3. 按调用计费

传统服务通常按机器、实例或固定容量付费。Serverless 更常见的方式是按调用次数、执行时长、资源消耗量等维度计费。

这带来的好处是：

- 没有请求时，资源成本可能非常低。
- 资源成本与业务使用量更接近。
- 对于低频任务或突发业务更经济。

### 4. 自动弹性伸缩

当请求量突然增加时，Serverless 平台通常会自动增加并发执行能力；请求量下降时，再自动回收资源。

开发者不必像传统服务那样预先规划大量副本或手工调节实例数量。这种弹性能力是 Serverless 很重要的吸引力之一。

### 5. 托管化与高可用

Serverless 平台通常会把很多底层能力做成托管服务，例如：

- 运行环境托管
- 自动故障恢复
- 日志收集
- 指标监控
- 版本管理

这意味着用户以更少的运维投入，获得较高的可用性和平台能力。

## 五、Serverless 的基础组成

Serverless 并不是只有“函数”这一个概念，它通常由多种能力共同组成。

### 1. FaaS

FaaS 是 Function as a Service，即函数即服务。它是最典型的 Serverless 形态。

开发者编写一个函数，定义输入和输出，再配置触发条件。平台在事件到来时执行函数，并在执行完成后回收资源。

FaaS 的特点是：

- 粒度小
- 启动快
- 更适合短时任务
- 更适合事件驱动场景

### 2. BaaS

BaaS 是 Backend as a Service，即后端即服务。它指的是直接使用云平台或第三方提供的后端能力，而不是自己开发和维护这些后端系统。

常见 BaaS 能力包括：

- 身份认证
- 数据库
- 文件存储
- 消息服务
- 推送服务

在 Serverless 架构中，应用常常把一部分能力交给 BaaS，把另一部分定制逻辑放到函数里完成。

### 3. 事件源与触发器

事件源是事件产生的地方，触发器是把事件和函数连接起来的机制。

例如：

- 对象存储上传文件后触发图片处理函数。
- API 网关收到请求后触发接口函数。
- 消息队列接收到消息后触发消费函数。

没有触发器，函数只是代码；有了触发器，函数才真正参与业务流程。

### 4. API 网关

如果 Serverless 函数需要对外提供 HTTP 接口，通常会配合 API 网关使用。API 网关负责：

- 接收请求
- 路由请求到对应函数
- 做鉴权、限流、协议处理
- 返回统一接口结果

它是 Serverless Web/API 场景中的重要入口。

### 5. 监控与日志

虽然 Serverless 减少了服务器运维，但并不意味着不需要运维能力。相反，监控和日志依然非常重要。需要关注的内容包括：

- 函数调用次数
- 执行时长
- 错误率
- 并发数
- 超时情况
- 冷启动影响

这些信息决定了系统是否稳定，也决定了后续如何优化。

## 六、Serverless 的常见服务模型

### 1. FaaS

FaaS 是最典型的 Serverless 服务模型，适合把一个明确的业务动作封装成可触发执行的函数。

例如：

- 提交订单后发送通知
- 文件上传后自动压缩
- 定时执行数据清洗脚本

### 2. BaaS

BaaS 更适合那些通用后端能力，例如用户认证、对象存储、数据库和消息系统。开发者直接调用服务，不必自己从头建设后端基础设施。

### 3. Serverless Container

除了函数模式外，一些平台也提供 Serverless Container，也就是容器仍然作为运行单位，但平台负责弹性伸缩、按需调度和部分运维工作。

它适合：

- 想保留容器交付方式的应用
- 对运行环境有更多自定义要求的业务
- 需要比函数更完整运行时的场景

可以理解为，它处于传统容器服务和纯 FaaS 之间。

## 七、Serverless 的运行模式

### 1. 请求驱动

请求驱动是指用户的 HTTP 或 RPC 请求直接触发后端函数执行。这常见于 Web API、移动端后端接口、小程序接口等场景。

### 2. 事件驱动

事件驱动是 Serverless 最典型的运行方式。某个业务事件发生后，函数被自动触发执行。这种模式非常适合解耦系统之间的关系。

### 3. 定时任务驱动

很多周期性任务并不需要长期运行一个服务，只需要在固定时间执行一次。Serverless 可以通过定时触发器执行这些任务，例如：

- 每天生成报表
- 每小时同步数据
- 每晚执行备份任务

### 4. 流式处理驱动

在日志流、消息流、实时数据流场景中，Serverless 也可以根据输入数据流持续触发处理逻辑。

这种模式常用于：

- 实时日志分析
- 消息队列消费
- 事件总线处理

## 八、Serverless 下的应用设计变化

### 1. 从长期运行服务到按需执行

传统服务通常常驻运行，而 Serverless 更强调“需要时才执行”。这改变了应用的资源使用方式，也改变了很多设计思路。

开发者会更多考虑：

- 一个业务动作能否拆成独立函数
- 这个逻辑是不是只在事件发生时才需要运行
- 是否真的需要长期占用一台机器或一个进程

### 2. 从面向主机到面向函数

传统架构里，很多思考方式围绕主机展开，例如端口、进程、实例、副本。Serverless 更强调围绕函数或服务能力本身来思考。

关注点从“服务部署在哪台机器”转向“某个业务动作在什么条件下被触发、如何执行、如何编排”。

### 3. 状态外置

Serverless 函数通常更适合无状态处理。因为函数实例可能随时创建、销毁或迁移，本地内存中的状态并不可靠。

因此，状态通常要放到外部系统中，例如：

- 数据库
- 缓存
- 对象存储
- 消息队列

这和云化、微服务中的无状态设计思路是一致的。

### 4. 更强调拆分与组合

Serverless 常把一个大流程拆成多个小步骤，由多个函数和托管服务共同完成，再通过消息、事件或工作流编排起来。

这样做的优点是灵活、易扩展，但也会带来链路变长、调试更复杂的问题。

## 九、Serverless 中的常见概念

### 1. 函数

函数是 Serverless 中最基础的执行单元。它通常围绕一个相对明确的动作编写，例如处理一个请求、消费一条消息、处理一张图片。

### 2. 触发器

触发器定义“什么情况下执行函数”。没有触发器，函数只是代码；配置了触发器后，它才真正与业务事件连接起来。

### 3. 冷启动

冷启动是 Serverless 中经常被提到的概念。它指的是平台第一次创建函数运行环境时，需要额外做初始化，因此响应会比已经热起来的实例更慢。

冷启动并不是每次都会发生，但在低频调用和高突发场景中，需要特别关注。

### 4. API 网关

API 网关是对外暴露接口时的重要组件。它负责把外部请求转换为对函数的调用，并承担认证、路由和限流等能力。

### 5. 编排工作流

当业务不是一步完成，而是多个函数协作完成时，往往需要工作流编排。它负责定义执行顺序、异常处理、重试逻辑和状态流转。

### 6. IaC

IaC 即基础设施即代码。在 Serverless 场景中，函数、触发器、API 网关、权限配置等资源通常都可以通过代码化方式声明和管理。

这有助于环境一致、自动化部署和版本化管理。

## 十、Serverless 的收益与挑战

### 1. 主要收益

#### 成本收益

对于请求不连续、负载波动大、执行时间短的业务，Serverless 能明显减少空闲资源浪费，成本通常更贴近真实使用量。

#### 效率收益

开发者不需要过多关心服务器配置和扩容细节，交付重心更集中在业务逻辑本身，很多小功能可以更快上线。

#### 运维收益

平台托管了大量底层能力，团队在主机维护、容量规划和基础故障处理上的负担会下降。

### 2. 主要挑战

#### 冷启动问题

对于高实时性、低延迟要求严格的接口，冷启动可能带来体验波动，这是 Serverless 设计中必须考虑的问题。

#### 调试与排障复杂度

系统拆得更细、事件链更长后，问题定位可能比单体应用更复杂，需要更好的日志、链路追踪和监控体系。

#### 状态管理困难

Serverless 更适合无状态处理。如果业务流程强依赖本地会话、长连接或复杂状态，设计成本会明显上升。

#### 平台依赖较强

使用某家云平台的 Serverless 能力越深，后续迁移到别的平台时的改造成本可能越高。

#### 成本并非一定最低

Serverless 对低频、突发、事件型任务很有优势，但对于长期高负载、持续运行的大型服务，成本不一定比传统服务更低。

## 十一、如何理解 Serverless 的整体图景

### 1. Serverless 不是“没有服务器”

Serverless 最容易被误解的地方，就是名字里的“无服务器”。实际上服务器始终存在，只是这些服务器被平台屏蔽起来了。

所以，Serverless 的本质不是“服务器消失了”，而是“开发者不再直接管理服务器”。

### 2. Serverless 是一种更彻底的托管与按需计算模式

如果说传统云服务是把“机器”交给用户使用，那么 Serverless 更像是把“执行能力”直接交给用户使用。

用户不再主要购买一台机器的固定容量，而是购买：

- 某段代码被执行的能力
- 某项后端能力被调用的能力
- 某个事件流被处理的能力

从这个角度看，Serverless 是云计算进一步向自动化、托管化和细粒度资源使用演进的结果。

---

## 十二、总结

Serverless 架构模式的本质，是把应用从“长期占用资源运行”的模式，转变为“按请求或事件触发、按实际使用计费、由平台托管大量基础设施”的模式。

要建立完整概念，可以抓住以下主线：

- Serverless 的核心不是没有服务器，而是开发者少管理服务器。
- Serverless 的典型能力，是事件驱动、自动弹性、按调用计费和托管运行。
- Serverless 的主要组成，包括 FaaS、BaaS、触发器、API 网关和可观测能力。
- Serverless 更适合短时、事件型、波动型和快速交付类业务。
- Serverless 并不是万能方案，在冷启动、状态管理、调试排障和平台依赖上也有明显挑战。

如果从学习路径上看，接下来可以继续深入以下主题：

- FaaS 与容器服务的区别
- Serverless 中的冷启动原理与优化方式
- API 网关、消息队列和函数触发的配合方式
- Serverless 工作流编排
- 典型云厂商的 Serverless 产品体系

学完这篇后，应该先建立这样一个判断：Serverless 不是单个产品，而是一种更强调托管化、按需执行和事件驱动的软件架构模式。
