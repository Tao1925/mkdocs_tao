## 二、数据结构与算法是什么

### 1. 数据结构的定义

数据结构，简单来说，就是数据在计算机中的组织方式，以及围绕这种组织方式所支持的操作。

很多初学者会把数据结构理解成“某个具体容器”，比如数组、链表、树、图。

这当然没错，但还不够完整。

更准确地说，数据结构至少包含两个方面。

- 第一，数据元素之间是什么关系。
- 第二，这些数据在计算机中如何存储与操作。

例如，一组学生成绩，如果只是单纯顺序存放，适合用数组。

如果需要频繁在中间插入和删除结点，链表可能更合适。

如果需要快速表示层次关系，比如“公司组织架构”或“文件目录”，树会更自然。

### 2. 算法的定义

算法，是为解决某一类问题而设计的、有限且明确的一组步骤。

这里有几个关键点。

- 算法必须是为了解决问题而存在。
- 算法必须有清晰步骤，而不是模糊描述。
- 算法必须能在有限时间内结束。

比如“在一组有序数据中查找某个值”，顺序查找是一种算法，二分查找也是一种算法。

两者都能解决问题，但效率可能差很多。

### 3. 为什么数据结构与算法总是一起出现

在很多教材里，有一句非常经典的话。

程序 = 数据结构 + 算法。

这句话并不严格等同于现实中的所有软件开发，但它高度概括了计算的核心。

原因在于。

- 数据结构决定数据如何组织。
- 算法决定数据如何被处理。

如果把算法比作“做事的方法”，那么数据结构就是“做事所用的工具和材料摆放方式”。

工具和材料摆放得不好，再好的方法也会变慢。

方法不合理，再好的数据结构也发挥不出效果。

### 4. 一个直观例子

假设你要设计一个通讯录系统，支持下面几种操作。

- 添加联系人。
- 删除联系人。
- 按姓名查找联系人。
- 按最近联系顺序显示联系人。

如果数据量很小，直接用数组存就够了。

如果数据量很大，且查找操作非常频繁，就可能需要哈希表。

如果既要按照字典序管理，还要支持范围查询，可能又会想到树结构。

这说明一个问题。

解决同一个业务问题时，数据结构的选择会直接影响算法设计和系统性能。

### 5. 数据结构不只是“存数据”

很多初学者会认为，数据结构就是拿来“装东西”的。

实际上，数据结构最重要的价值往往在于“支持什么操作、这些操作代价如何”。

例如数组的随机访问很快。

但数组中间插入元素往往要移动大量数据。

链表中间插入元素可能更容易。

但它不支持像数组那样的 $O(1)$ 随机访问。

所以学习数据结构，不是只认识名字，而是理解它的性能特征和应用边界。

### 6. 算法不只是“写出代码”

很多人初学算法时容易把注意力放在代码技巧上。

例如。

- 如何把循环写对。
- 如何用递归。
- 如何通过题目样例。

这些都重要，但它们不是算法学习的核心。

算法更关键的是下面几件事。

- 你有没有把问题抽象清楚。
- 你有没有找到正确的求解思路。
- 你能不能分析这个思路的正确性和效率。

换句话说，代码只是算法思想的最终表达形式，不是算法本身。

### 7. 数据结构与算法在计算机中的位置

数据结构与算法是计算机科学中的基础课程。

它不仅服务于编程练习，还会影响后续很多课程和方向。

例如。

- 操作系统中的进程调度会用到队列、堆。
- 数据库索引会用到 B 树、哈希。
- 编译原理中的语法分析会用到栈和树。
- 图形学、网络路由、搜索引擎会大量用到图算法。

### 8. 学习这门课真正要得到什么

对本科生来说，这门课最重要的收获通常有三类。

- 第一，建立抽象思维。
- 第二，理解效率与资源消耗。
- 第三，学会用合适的方法解决问题。

如果只会把答案背下来，这些能力都很难真正形成。

### 9. 本章小结

本章最重要的结论可以概括成一句话。

数据结构研究“数据怎么组织”，算法研究“问题怎么求解”，两者共同决定程序能否正确、高效地完成任务。

## 三、为什么要学习数据结构与算法

### 1. 程序能运行，不代表程序写得好

很多初学者会有一种错觉：只要程序输出正确结果，就说明程序没有大问题。

这在小规模样例里常常成立，但在真实系统中远远不够。

例如有两个程序都能完成查找任务。

- 一个在 100 个数据里表现正常，但在 1000 万个数据里会很慢。
- 另一个在数据量增大后仍然保持较好性能。

它们都“能运行”，但工程价值完全不同。

### 2. 数据规模会改变问题本质

当输入规模很小时，很多方法看起来都差不多。

但当规模上升后，复杂度差异就会被迅速放大。

例如。

- $O(n)$ 和 $O(\log n)$ 在小数据下差距不一定明显。
- $O(n^2)$ 和 $O(n \log n)$ 在大数据下差距会很大。

所以学习数据结构与算法，本质上是在学习如何面对规模增长。

### 3. 它能帮助你理解“为什么有些程序快，有些程序慢”

很多性能问题，表面上看是“机器不够好”。

但深入分析后，往往是算法或数据结构选错了。

例如。

- 使用线性查找代替哈希查找。
- 在频繁插入删除场景里错误使用数组。
- 对图问题使用不合适的存储方式。
- 在可用动态规划的问题上做了大量重复递归。

### 4. 它是很多后续课程的底层语言

学完数据结构与算法后，再去看其他课程时，你会发现很多概念变得容易理解。

例如。

- 操作系统中的等待队列、调度优先级。
- 数据库中的索引、连接算法、缓冲区管理。
- 计算机网络中的路由算法、最短路径。
- 人工智能中的搜索、状态空间、图模型。

### 5. 它能训练抽象建模能力

很多现实问题一开始并不会告诉你应该用什么结构。

你需要自己把问题抽象出来。

例如。

- 地图导航可以抽象成图。
- 网页 DOM 可以抽象成树。
- 浏览器前进后退可以抽象成栈。
- 打印任务管理可以抽象成队列或优先队列。

这种“把现实问题转换为结构化问题”的能力，是计算机专业非常核心的能力。

### 6. 它也能训练证明与分析能力

本科阶段学习数据结构与算法，并不是为了把自己变成竞赛选手。

更重要的是学会下面这些思维方式。

- 这个方法为什么正确。
- 这个方法为什么快或慢。
- 这个方法在哪些条件下成立。
- 如果条件变化，方法是否还适用。

这类能力在软件工程、科研和工程设计中都很重要。

### 7. 它对面试有帮助，但不能只把它当面试工具

现实中，数据结构与算法确实是技术面试的重要部分。

但如果只把它当“刷题工具”，学习会变得很机械。

更合理的认识是。

- 面试只是检验你是否具备基本问题分析能力。
- 真正长期有价值的，是你对结构、复杂度和算法思想的理解。

### 8. 它和工程实践并不脱节

有些同学会觉得这门课“太理论”，和写项目关系不大。

实际上很多工程问题都直接依赖这门课。

例如。

- 缓存淘汰策略会用到链表和哈希表。
- 搜索建议和前缀匹配会用到 Trie。
- 日志处理和任务调度会用到堆。
- 社交关系和推荐系统会大量涉及图。

### 9. 本章小结

学习数据结构与算法，不只是为了会做几道题，而是为了理解计算问题的组织方式、求解方式和效率边界。

## 四、学习数据结构与算法前需要掌握的基础概念

### 1. 数据与数据元素

在教材定义中，数据是对客观事物的符号表示。

例如数字、字符、图像、音频、坐标，都可以看作数据。

数据元素则是数据的基本单位。

例如在学生信息表中，一个学生记录可以看成一个数据元素。

### 2. 数据对象

数据对象通常指性质相同的数据元素集合。

例如“一个班所有学生记录”可以视作一个数据对象。

这个概念比较理论，但它有助于我们理解“问题中的数据规模和边界”。

### 3. 逻辑结构

逻辑结构描述的是数据元素之间的逻辑关系，而不是它们在内存中的实际放置方式。

常见逻辑结构包括。

- 集合结构：元素之间除了“同属一个集合”外没有明显关系。
- 线性结构：元素之间一对一关系明显。
- 树形结构：元素之间是一对多层次关系。
- 图状结构：元素之间可能是多对多关系。

### 4. 存储结构

存储结构描述的是数据在计算机内存中如何表示。

常见方式包括。

- 顺序存储。
- 链式存储。
- 索引存储。
- 散列存储。

逻辑结构和存储结构不是一回事。

同一种逻辑结构，可能有多种不同存储方式。

### 5. 为什么要区分逻辑结构和存储结构

因为这会直接影响算法实现和效率。

例如线性表可以用数组实现，也可以用链表实现。

逻辑上它们都是“线性表”。

但由于存储结构不同，插入、删除、访问的性能也不同。

### 6. 抽象数据类型 ADT

抽象数据类型强调的是。

- 这个结构对外提供哪些操作。
- 这些操作的语义是什么。

而不强调具体实现细节。

例如“栈”作为一种抽象数据类型，核心操作是。

- 入栈
- 出栈
- 取栈顶
- 判空

至于底层是数组实现还是链表实现，是另一层问题。

### 7. 算法的五个基本特性

教材中通常会讲，算法应具备以下五个特性。

- 有穷性：执行有限步后必须结束。
- 确定性：每一步含义明确。
- 可行性：每一步都能通过有限操作实现。
- 输入：可以有零个或多个输入。
- 输出：至少有一个输出。

这个定义看似抽象，但它帮助我们区分“模糊想法”和“真正可执行的算法”。

### 8. 正确性

一个算法首先要正确。

所谓正确，不只是“样例过了”。

更重要的是对所有符合输入条件的数据，都能给出满足题意的结果。

本科阶段需要慢慢培养一种意识。

算法是否正确，往往需要通过逻辑分析或数学归纳来说明。

### 9. 可读性

很多同学写代码只关注“快不快”，容易忽略可读性。

实际上，可读性非常重要。

可读性好的实现更容易。

- 调试。
- 复查。
- 复用。
- 被别人理解。

### 10. 健壮性

健壮性强调程序面对异常输入或边界情况时，是否还能合理处理。

例如。

- 空数组。
- 单个元素。
- 重复值。
- 极端大输入。
- 非法删除操作。

如果算法只在“理想输入”上工作，它的实际价值会很有限。

### 11. 学习这门课时的常见误区

误区一，只记模板，不理解原因。

误区二，只背复杂度，不知道为什么是这个复杂度。

误区三，把每种结构看成孤立知识点，不建立联系。

误区四，只会看答案，不会自己从问题出发建模。

### 12. 一个基本学习方法

学习每一种数据结构或算法时，都建议按下面几个问题来理解。

1. 它解决什么问题。
2. 它的核心思想是什么。
3. 它支持哪些操作。
4. 各操作复杂度是多少。
5. 适合什么场景。
6. 不适合什么场景。
7. 常见实现细节和坑点是什么。

### 13. 本章小结

本章的重点，是为后面正式学习各种结构和算法建立统一语言。

只有把逻辑结构、存储结构、抽象数据类型和算法性质这些基础概念理顺，后面才不会一直混淆。

## 五、算法复杂度分析

### 1. 为什么复杂度分析很重要

如果不分析复杂度，我们很难知道一个算法在大规模输入下能否工作。

写程序时，运行结果正确当然重要。

但如果运行 10 秒和运行 10 分钟的程序都叫“正确”，那这个定义对工程几乎没有帮助。

复杂度分析，就是用一种更抽象、更稳定的方式来衡量算法代价。

### 2. 时间复杂度

时间复杂度，描述的是算法运行时间随输入规模增长的大致趋势。

注意，它不是精确到毫秒的运行时间。

因为实际运行时间还会受。

- 机器性能。
- 编译器优化。
- 常数因子。
- 输入分布。

等多种因素影响。

所以时间复杂度更像是一种“增长级别”的衡量。

### 3. 空间复杂度

空间复杂度，描述的是算法额外占用存储空间随输入规模变化的大致趋势。

例如。

- 原地交换排序可能只需要 $O(1)$ 额外空间。
- 归并排序往往需要 $O(n)$ 额外空间。

空间同样是一种重要资源。

尤其在嵌入式系统、超大规模数据处理和缓存敏感场景中，空间消耗可能决定算法是否可行。

### 4. 大 O 表示法

大 O 表示法用于描述增长上界，实际学习中最常使用。

例如。

- 顺序遍历数组常写作 $O(n)$。
- 双重循环常见为 $O(n^2)$。
- 二分查找常见为 $O(\log n)$。

大 O 分析时通常关注主导项，忽略常数和低阶项。

例如。

- $3n + 5$ 记为 $O(n)$。
- $2n^2 + n + 1$ 记为 $O(n^2)$。

### 5. Theta 和 Omega

除了大 O，还常见两个符号。

- $\Theta$：表示紧确界，也就是增长级别恰好在这个数量级。
- $\Omega$：表示下界。

本科阶段如果不是做更严格理论分析，最常用的仍然是大 O。

但知道这些符号的意义有助于阅读教材和论文。

### 6. 最好、最坏、平均情况

分析复杂度时，常常会区分。

- 最好情况。
- 最坏情况。
- 平均情况。

例如顺序查找。

- 如果目标恰好在第一个位置，最好情况是 $O(1)$。
- 如果目标不在数组里或在最后一个位置，最坏情况是 $O(n)$。

工程实践中，最坏情况常常更值得关注，因为它关系到算法的稳定上界。

### 7. 常见复杂度级别

下面是一些非常常见的复杂度级别。

| 复杂度 | 直观含义 | 常见例子 |
|--------|----------|----------|
| $O(1)$ | 常数时间 | 数组按下标访问 |
| $O(\log n)$ | 对数级 | 二分查找 |
| $O(n)$ | 线性级 | 单次遍历 |
| $O(n \log n)$ | 线性对数级 | 归并排序、堆排序 |
| $O(n^2)$ | 平方级 | 简单双重循环 |
| $O(2^n)$ | 指数级 | 一些无剪枝回溯 |
| $O(n!)$ | 阶乘级 | 全排列枚举 |

### 8. 常见复杂度之间的差别

对于小规模输入，差别不一定明显。

但输入规模增大后，增长速度会迅速拉开。

这也是为什么本科阶段必须建立复杂度敏感度。

### 9. 循环复杂度如何分析

分析循环复杂度时，常用的基本方法是看循环执行次数。

例如。

```cpp
for (int i = 0; i < n; ++i) {
	// 常数操作
}
```

这个循环执行了 $n$ 次，所以复杂度是 $O(n)$。

再例如。

```cpp
for (int i = 0; i < n; ++i) {
	for (int j = 0; j < n; ++j) {
		// 常数操作
	}
}
```

这里总执行次数约为 $n \times n$，因此复杂度是 $O(n^2)$。

### 10. 对数复杂度的直观理解

很多同学最开始对 $O(\log n)$ 不敏感。

可以这样理解：每一步把问题规模缩小一半。

例如二分查找。

如果数据规模是。

- 8，最多查 3 次左右。
- 16，最多查 4 次左右。
- 1024，最多查 10 次左右。

这就是对数级增长的威力。

### 11. 递归复杂度如何分析

递归复杂度通常需要写出递推式。

例如归并排序可以写成。

$$
T(n) = 2T(n/2) + O(n)
$$

这表示。

- 问题被分成两个规模为 $n/2$ 的子问题。
- 合并阶段需要线性时间。

最终结果是 $O(n \log n)$。

### 12. 递归分析不一定要会复杂定理才能入门

本科初学阶段，不一定要一下子掌握非常严格的主定理。

更重要的是建立直觉。

- 每一层做多少工作。
- 一共有多少层。
- 总工作量如何叠加。

### 13. 摊还分析

有些操作单次看起来可能很慢，但放到一系列操作中平均来看又并不贵。

这时常用摊还分析。

一个经典例子是动态数组扩容。

`vector` 平时尾插通常是 $O(1)$。

虽然某些时刻会因扩容复制变成 $O(n)$，但平均到大量插入操作上，摊还复杂度仍然常被视为 $O(1)$。

### 14. 复杂度分析中的常见误区

误区一，认为常数完全不重要。

其实在小数据或工程实现中，常数也可能影响明显。

误区二，认为复杂度越低就一定越好。

还需要结合。

- 实现复杂度。
- 空间开销。
- 场景是否匹配。

误区三，把理论复杂度和实际运行时间混为一谈。

### 15. 一个简单复杂度对比例子

如果要在一个长度为 $n$ 的数组中查找元素。

- 顺序查找复杂度是 $O(n)$。
- 如果数组有序，二分查找复杂度是 $O(\log n)$。

当 $n$ 很大时，差距会非常明显。

### 16. 本章小结

复杂度分析的核心不是为了写出符号，而是帮助我们理解算法随规模变化的表现。

本科阶段最重要的是做到。

- 能分析常见循环复杂度。
- 能理解常见递归复杂度。
- 能比较多个算法在大规模输入下的优劣。

## 六、线性表、数组与链表

### 1. 线性表的定义

线性表是最基础的一类数据结构。

它的特点是数据元素之间存在一对一的线性关系。

除第一个元素外，每个元素都有唯一前驱。

除最后一个元素外，每个元素都有唯一后继。

### 2. 线性表的常见操作

围绕线性表，通常会讨论这些基本操作。

- 初始化。
- 插入。
- 删除。
- 查找。
- 修改。
- 遍历。
- 判空。
- 获取长度。

### 3. 顺序存储

顺序存储是指用一段连续内存存放线性表元素。

数组就是最典型的顺序存储结构。

顺序存储的优点是。

- 支持随机访问。
- 实现简单。
- 缓存友好，常数因子通常较小。

缺点是。

- 中间插入和删除可能要移动大量元素。
- 容量管理需要提前考虑或扩容处理。

### 4. 数组的特点

数组最大的优势，是根据下标进行随机访问的效率高。

例如访问第 $i$ 个元素，通常是 $O(1)$。

这是因为数组在内存中连续存放，可以通过首地址加偏移快速定位。

### 5. 数组适合什么场景

数组通常适合下面这些场景。

- 需要频繁按下标访问。
- 数据规模相对稳定。
- 中间插入删除不频繁。
- 需要较好的局部性和遍历性能。

### 6. 链式存储

链式存储是通过指针或引用，把不连续的结点连接起来形成逻辑线性结构。

链表是最典型的链式存储结构。

它的核心优点在于，不要求元素连续存储。

### 7. 单链表

单链表中，每个结点通常包含两部分。

- 数据域。
- 指针域，指向下一个结点。

这样多个结点串起来，就形成一条链。

### 8. 双链表

双链表中，每个结点通常包含。

- 数据域。
- 指向前驱的指针。
- 指向后继的指针。

双链表比单链表更灵活，便于向前和向后遍历。

但它的存储开销也更大。

### 9. 循环链表

循环链表是让尾结点重新指向头结点或头结点相关位置，从而形成环。

它常用于需要“循环访问”或便于从尾部回到头部的场景。

### 10. 数组和链表的比较

| 方面 | 数组 | 链表 |
|------|------|------|
| 存储方式 | 连续存储 | 非连续结点链接 |
| 随机访问 | 快，$O(1)$ | 慢，$O(n)$ |
| 中间插入删除 | 慢，通常 $O(n)$ | 已知位置时可较快调整指针 |
| 空间开销 | 较紧凑 | 有额外指针开销 |
| 缓存友好性 | 好 | 一般较差 |

### 11. 什么时候该选数组，什么时候该选链表

可以从需求出发判断。

如果你更关心。

- 快速按下标访问。
- 大量遍历。
- 内存连续性。

通常更偏向数组。

如果你更关心。

- 频繁插入删除。
- 结构动态变化。
- 不强调随机访问。

则链表更值得考虑。

### 12. C++ 中常见对应结构

在 C++ 里。

- `std::vector` 常对应动态数组思想。
- `std::list` 常对应双向链表。

不过学习阶段仍然建议至少手写一次链表，这样才会真正理解指针操作。

### 13. 单链表 C++ 实现示例

下面给出一个简化的单链表实现，支持头插、查找和删除。

```cpp
#include <iostream>

struct ListNode {
	int value;
	ListNode* next;
	ListNode(int v) : value(v), next(nullptr) {}
};

class SinglyLinkedList {
private:
	ListNode* head;

public:
	SinglyLinkedList() : head(nullptr) {}

	~SinglyLinkedList() {
		ListNode* current = head;
		while (current != nullptr) {
			ListNode* nextNode = current->next;
			delete current;
			current = nextNode;
		}
	}

	void pushFront(int value) {
		ListNode* node = new ListNode(value);
		node->next = head;
		head = node;
	}

	bool find(int target) const {
		ListNode* current = head;
		while (current != nullptr) {
			if (current->value == target) {
				return true;
			}
			current = current->next;
		}
		return false;
	}

	bool erase(int target) {
		if (head == nullptr) {
			return false;
		}

		if (head->value == target) {
			ListNode* oldHead = head;
			head = head->next;
			delete oldHead;
			return true;
		}

		ListNode* current = head;
		while (current->next != nullptr && current->next->value != target) {
			current = current->next;
		}

		if (current->next == nullptr) {
			return false;
		}

		ListNode* toDelete = current->next;
		current->next = toDelete->next;
		delete toDelete;
		return true;
	}

	void print() const {
		ListNode* current = head;
		while (current != nullptr) {
			std::cout << current->value << " ";
			current = current->next;
		}
		std::cout << '\n';
	}
};

int main() {
	SinglyLinkedList list;
	list.pushFront(3);
	list.pushFront(2);
	list.pushFront(1);
	list.print();

	std::cout << std::boolalpha << list.find(2) << '\n';
	list.erase(2);
	list.print();
	return 0;
}
```

### 14. 这段链表代码体现了什么

它体现了几个很典型的链表特征。

- 结点不是连续存储的。
- 查找必须从头顺着指针走。
- 删除操作本质上是改指针。
- 需要手动管理内存，避免泄漏。

### 15. 线性表这一章最容易犯的错误

常见错误包括。

- 数组越界。
- 链表空指针访问。
- 删除结点后忘记释放内存。
- 指针修改顺序错误导致链表断裂。

### 16. 本章小结

线性表是后续很多结构的基础。

理解这一章时，最重要的不是背定义，而是建立这种判断能力。

- 数据是否需要连续存储。
- 是否频繁插入删除。
- 是否需要随机访问。

这些判断，会直接决定你选数组还是链表。

## 七、栈与队列

### 1. 栈的定义

栈是一种只允许在一端进行插入和删除的线性结构。

它遵循后进先出，也就是 LIFO，Last In First Out。

最直观的例子是盘子堆。

后放上去的盘子，通常先被拿走。

### 2. 栈的基本操作

栈最核心的操作包括。

- `push`：入栈。
- `pop`：出栈。
- `top`：取栈顶元素。
- `empty`：判空。

### 3. 栈适合解决什么问题

栈特别适合处理具有“最近进入的状态先被处理”特征的问题。

常见场景包括。

- 函数调用栈。
- 括号匹配。
- 表达式求值。
- 深度优先搜索的递归过程。
- 浏览器回退。

### 4. 队列的定义

队列是一种只允许一端插入、另一端删除的线性结构。

它遵循先进先出，也就是 FIFO，First In First Out。

最直观的例子是排队。

先来的人通常先被服务。

### 5. 队列的基本操作

队列最核心的操作包括。

- `push` 或 `enqueue`：入队。
- `pop` 或 `dequeue`：出队。
- `front`：取队头。
- `empty`：判空。

### 6. 队列适合解决什么问题

队列特别适合解决按到达顺序处理任务的问题。

例如。

- 打印任务调度。
- 操作系统等待队列。
- 图的广度优先搜索。
- 消息缓冲。

### 7. 栈和队列的本质差异

栈与队列都属于线性结构。

但它们处理元素的顺序不同。

- 栈强调“后来的先处理”。
- 队列强调“先来的先处理”。

很多问题之所以需要其中之一，恰恰是因为这个顺序约束。

### 8. 顺序栈

顺序栈通常用数组实现。

优点是实现简单，访问效率高。

缺点是容量需要管理。

### 9. 链栈

链栈通常用链表实现。

它避免了固定容量问题，但需要额外指针空间。

### 10. 顺序队列与循环队列

如果用普通数组直接实现队列，随着出队操作进行，前面会留下空位。

如果不做优化，就可能造成“假溢出”现象。

循环队列的思想是把数组首尾接起来逻辑上看成一个环，从而更充分利用空间。

### 11. 双端队列

双端队列允许在两端插入和删除。

它比普通队列更灵活。

在一些滑动窗口、单调结构问题中很常见。

### 12. 单调栈与单调队列的基本思想

这两个结构在算法题和工程优化中都很常见。

单调栈通常维护一个单调递增或单调递减序列。

它常用于。

- 求每个元素左边或右边第一个更大或更小元素。
- 处理柱状图、温度、股票价格等问题。

单调队列则常用于滑动窗口最值问题。

### 13. 顺序栈 C++ 示例

```cpp
#include <iostream>
#include <vector>
#include <stdexcept>

class Stack {
private:
	std::vector<int> data;

public:
	void push(int value) {
		data.push_back(value);
	}

	void pop() {
		if (data.empty()) {
			throw std::runtime_error("stack is empty");
		}
		data.pop_back();
	}

	int top() const {
		if (data.empty()) {
			throw std::runtime_error("stack is empty");
		}
		return data.back();
	}

	bool empty() const {
		return data.empty();
	}
};
```

### 14. 循环队列 C++ 示例

```cpp
#include <iostream>
#include <vector>
#include <stdexcept>

class CircularQueue {
private:
	std::vector<int> data;
	int frontIndex;
	int rearIndex;
	int currentSize;

public:
	explicit CircularQueue(int capacity)
		: data(capacity), frontIndex(0), rearIndex(0), currentSize(0) {}

	bool empty() const {
		return currentSize == 0;
	}

	bool full() const {
		return currentSize == static_cast<int>(data.size());
	}

	void push(int value) {
		if (full()) {
			throw std::runtime_error("queue is full");
		}
		data[rearIndex] = value;
		rearIndex = (rearIndex + 1) % data.size();
		++currentSize;
	}

	void pop() {
		if (empty()) {
			throw std::runtime_error("queue is empty");
		}
		frontIndex = (frontIndex + 1) % data.size();
		--currentSize;
	}

	int front() const {
		if (empty()) {
			throw std::runtime_error("queue is empty");
		}
		return data[frontIndex];
	}
};
```

### 15. 括号匹配为什么适合用栈

因为匹配过程中，总是最后出现的未匹配左括号要最先和后面的右括号配对。

这个“最后进入、最先处理”的规律，与栈完全一致。

### 16. 本章小结

栈和队列看起来简单，但它们是非常重要的基础结构。

很多复杂算法，其实只是把状态组织成栈或队列后变得更清晰。

## 八、串与基础字符串处理

### 1. 串的定义

串可以看作由零个或多个字符组成的有限序列。

从抽象上讲，它和线性表很像。

但由于元素类型固定为字符，且很多操作围绕模式匹配、比较、拼接展开，所以通常会被单独讨论。

### 2. 串和一般线性表的区别

从结构上讲，串也是线性结构。

但它有两个明显特点。

- 元素通常是字符。
- 常见操作更强调“模式”而不是单个元素。

例如我们会关心。

- 子串查找。
- 前缀与后缀。
- 模式匹配。
- 字典序比较。

### 3. C++ 中的字符串

在 C++ 中，常见字符串表示有两类。

- C 风格字符串。
- `std::string`。

在现代 C++ 学习和工程里，更推荐使用 `std::string`，因为它更安全、易用。

### 4. 常见字符串操作

常见操作包括。

- 获取长度。
- 拼接。
- 访问某个字符。
- 截取子串。
- 比较。
- 查找子串。

### 5. 朴素模式匹配思想

假设主串是 `ababcabcacbab`，模式串是 `abcac`。

最直接的思路，是让模式串从主串每个可能起点依次尝试比较。

如果某个位置匹配失败，就把模式串整体向后移动一位重新比较。

这种方法简单，但当字符串很长时效率可能不高。

### 6. 为什么字符串算法值得单独学习

因为字符串问题看似简单，实际上经常涉及很强的模式结构。

例如。

- 搜索引擎中的关键词匹配。
- 编辑器中的查找替换。
- DNA 序列比对。
- 编译器中的词法分析。

这类问题都不是简单逐字符处理就能高效解决的。

### 7. 本章小结

字符串在结构上属于线性结构，但由于其模式匹配、前缀后缀等问题非常典型，所以后面会再单独介绍 KMP、Trie 等更专门的字符串算法。

## 九、树与二叉树

### 1. 树的定义

树是一种层次型数据结构。

它由结点组成，并且具有明显的父子关系。

和线性结构不同，树强调的是层次和分支。

### 2. 树的基本术语

学习树时，先要理解几个常见术语。

- 结点：树中的基本单位。
- 根结点：最顶层结点。
- 父结点：某个结点的直接上层结点。
- 子结点：某个结点的直接下层结点。
- 叶子结点：没有子结点的结点。
- 结点的度：该结点子结点个数。
- 树的高度：树的最大层数或最大深度。

### 3. 树适合描述什么问题

树很适合描述层次结构问题。

例如。

- 文件目录。
- 组织架构。
- 表达式语法树。
- HTML DOM。
- 决策过程。

### 4. 二叉树的定义

二叉树是一种特殊的树。

它的每个结点最多只有两个孩子，通常称为左孩子和右孩子。

注意，“最多两个”并不等于“恰好两个”。

### 5. 二叉树为什么重要

二叉树之所以特别重要，是因为它结构清晰、分析方便，而且很多经典结构都和它有关。

例如。

- 二叉搜索树。
- 堆。
- AVL 树。
- 红黑树。

### 6. 满二叉树与完全二叉树

满二叉树，是指每一层都达到最大结点数。

完全二叉树，是指除最后一层外都满，最后一层结点尽量靠左排列。

完全二叉树很重要，因为堆通常就建立在完全二叉树的性质上。

### 7. 二叉树的性质

二叉树有很多经典性质，本科阶段常见的包括。

- 第 $i$ 层最多有 $2^{i-1}$ 个结点。
- 高度为 $h$ 的二叉树最多有 $2^h - 1$ 个结点。
- 完全二叉树可以方便地用数组存储。

### 8. 二叉树的存储方式

二叉树常见有两种表示方式。

- 顺序存储。
- 链式存储。

顺序存储更适合完全二叉树，比如堆。

链式存储更灵活，适合一般二叉树。

### 9. 二叉树的遍历

二叉树最经典的操作之一就是遍历。

常见遍历方式有。

- 前序遍历：根、左、右。
- 中序遍历：左、根、右。
- 后序遍历：左、右、根。
- 层序遍历：按层从上到下，从左到右。

### 10. 为什么中序遍历在二叉搜索树中特别重要

因为对于二叉搜索树来说，中序遍历会得到一个有序序列。

这个性质在分析和应用中非常有用。

### 11. 二叉搜索树 BST

二叉搜索树是一类重要的二叉树。

它满足。

- 左子树所有结点值小于根结点值。
- 右子树所有结点值大于根结点值。
- 左右子树也分别是二叉搜索树。

因此，它可以把查找、插入、删除操作的效率做到平均较好。

### 12. BST 的优点和问题

优点是结构简单，支持动态插入和查找。

问题是如果插入数据接近有序，BST 可能退化成链表，从而导致效率下降到 $O(n)$。

### 13. 平衡树的基本思想

为了解决 BST 退化问题，出现了平衡树。

它们的核心思想是：在插入和删除后，通过一定调整，尽量保持树高较低。

常见平衡树包括。

- AVL 树。
- 红黑树。

本科阶段通常先理解思想，不一定要一开始就手写完整实现。

### 14. 二叉树遍历 C++ 示例

```cpp
#include <iostream>

struct TreeNode {
	int value;
	TreeNode* left;
	TreeNode* right;

	explicit TreeNode(int v) : value(v), left(nullptr), right(nullptr) {}
};

void preorder(TreeNode* root) {
	if (root == nullptr) {
		return;
	}
	std::cout << root->value << " ";
	preorder(root->left);
	preorder(root->right);
}

void inorder(TreeNode* root) {
	if (root == nullptr) {
		return;
	}
	inorder(root->left);
	std::cout << root->value << " ";
	inorder(root->right);
}

void postorder(TreeNode* root) {
	if (root == nullptr) {
		return;
	}
	postorder(root->left);
	postorder(root->right);
	std::cout << root->value << " ";
}
```

### 15. 二叉搜索树插入与查找 C++ 示例

```cpp
#include <iostream>

struct BSTNode {
	int value;
	BSTNode* left;
	BSTNode* right;

	explicit BSTNode(int v) : value(v), left(nullptr), right(nullptr) {}
};

BSTNode* insert(BSTNode* root, int value) {
	if (root == nullptr) {
		return new BSTNode(value);
	}

	if (value < root->value) {
		root->left = insert(root->left, value);
	} else if (value > root->value) {
		root->right = insert(root->right, value);
	}

	return root;
}

bool search(BSTNode* root, int target) {
	if (root == nullptr) {
		return false;
	}
	if (root->value == target) {
		return true;
	}
	if (target < root->value) {
		return search(root->left, target);
	}
	return search(root->right, target);
}
```

### 16. 这段 BST 代码说明了什么

它说明 BST 的查找过程，本质上是在利用有序性不断缩小范围。

如果树比较平衡，查找、插入通常接近 $O(\log n)$。

但如果树很偏斜，就会退化。

### 17. 树这一章最容易混淆的点

常见混淆包括。

- 树高和结点深度混用。
- 满二叉树和完全二叉树混用。
- 二叉树和二叉搜索树混用。
- 遍历顺序记错。

### 18. 本章小结

树结构是从线性结构迈向层次结构的关键一步。

这一章真正要掌握的是。

- 树如何表达层次关系。
- 二叉树遍历为什么重要。
- BST 如何通过有序性提升查找效率。

## 十、堆与优先队列

### 1. 堆的定义

堆是一种特殊的完全二叉树。

常见有两种。

- 大根堆：每个结点值都不小于其孩子结点值。
- 小根堆：每个结点值都不大于其孩子结点值。

### 2. 堆和普通二叉树的区别

堆不是任意二叉树。

它有两个关键约束。

- 形态上必须是完全二叉树。
- 数值上必须满足堆序性质。

### 3. 为什么堆适合做优先队列

优先队列的核心需求是：每次都能快速取出当前优先级最高或最低的元素。

堆正好擅长这个。

在大根堆中，最大值始终在堆顶。

在小根堆中，最小值始终在堆顶。

### 4. 堆的常见操作

堆最常见的操作包括。

- 插入元素。
- 删除堆顶。
- 获取堆顶。
- 建堆。

### 5. 堆操作的复杂度特点

由于堆高通常是 $O(\log n)$，所以。

- 插入通常是 $O(\log n)$。
- 删除堆顶通常是 $O(\log n)$。
- 取堆顶通常是 $O(1)$。

### 6. 为什么堆适合 Top K 问题

如果要动态维护当前最小的几个值、最大的几个值，堆通常非常合适。

例如。

- 维护成绩前 10 名。
- 实时调度优先级最高任务。
- 求海量数据中的第 K 大元素。

### 7. 堆与二叉搜索树的区别

两者都和二叉树有关，但用途不同。

堆只保证堆顶与孩子之间的大小关系，不保证整棵树的全局有序。

BST 则保证左小右大，适合查找某个具体值。

所以。

- 想快速取最值，堆更合适。
- 想做有序查找和范围查询，BST 更自然。

### 8. 小根堆 C++ 示例

```cpp
#include <iostream>
#include <vector>
#include <stdexcept>

class MinHeap {
private:
	std::vector<int> heap;

	void siftUp(int index) {
		while (index > 0) {
			int parent = (index - 1) / 2;
			if (heap[parent] <= heap[index]) {
				break;
			}
			std::swap(heap[parent], heap[index]);
			index = parent;
		}
	}

	void siftDown(int index) {
		int n = static_cast<int>(heap.size());
		while (true) {
			int left = index * 2 + 1;
			int right = index * 2 + 2;
			int smallest = index;

			if (left < n && heap[left] < heap[smallest]) {
				smallest = left;
			}
			if (right < n && heap[right] < heap[smallest]) {
				smallest = right;
			}
			if (smallest == index) {
				break;
			}
			std::swap(heap[index], heap[smallest]);
			index = smallest;
		}
	}

public:
	void push(int value) {
		heap.push_back(value);
		siftUp(static_cast<int>(heap.size()) - 1);
	}

	int top() const {
		if (heap.empty()) {
			throw std::runtime_error("heap is empty");
		}
		return heap[0];
	}

	void pop() {
		if (heap.empty()) {
			throw std::runtime_error("heap is empty");
		}
		heap[0] = heap.back();
		heap.pop_back();
		if (!heap.empty()) {
			siftDown(0);
		}
	}

	bool empty() const {
		return heap.empty();
	}
};
```

### 9. 本章小结

堆的重点，不是把每个细节都死记，而是理解它为什么适合“动态维护最值”。

只要理解完全二叉树和堆序性质这两个核心约束，后面的优先队列、堆排序、Dijkstra 等算法都会更容易理解。

## 十一、哈希表与并查集

### 1. 哈希表的基本思想

哈希表的核心思想是：通过哈希函数，把“键”映射到数组下标附近，从而实现高效查找。

理想情况下。

- 插入很快。
- 删除很快。
- 查找也很快。

### 2. 为什么哈希表常常很快

因为它试图把“查找某个值”的过程，从逐个比较，变成“直接去可能的位置找”。

这和顺序查找相比，效率可能有非常大差别。

### 3. 哈希函数

哈希函数负责把键映射为整数位置。

一个好的哈希函数通常希望满足。

- 计算快。
- 分布尽量均匀。
- 冲突尽量少。

### 4. 哈希冲突

由于不同键可能被映射到同一位置，所以冲突几乎不可避免。

冲突本身不是错误。

关键在于如何处理冲突。

### 5. 常见冲突处理方法

本科阶段常见两大类。

- 拉链法。
- 开放定址法。

拉链法是在每个桶后面挂一条链表或其他结构。

开放定址法则是在冲突后继续寻找其他可用位置。

### 6. 哈希表的复杂度特点

在平均情况下，哈希表常见操作复杂度可接近 $O(1)$。

但在最坏情况下，如果大量冲突聚集，复杂度也可能退化到 $O(n)$。

### 7. C++ 中的哈希结构

在 C++ 标准库中，常见哈希相关容器有。

- `std::unordered_map`
- `std::unordered_set`

学习阶段建议知道它们背后的核心思想，而不是只把它们当黑盒。

### 8. 并查集是什么

并查集是一种非常适合处理“元素分组与连通性”问题的数据结构。

它主要支持两个操作。

- `find`：查询某个元素属于哪个集合。
- `union`：合并两个集合。

### 9. 并查集适合什么问题

并查集常见于。

- 判断两个点是否连通。
- 动态合并集合。
- 最小生成树中的 Kruskal 算法。
- 社交关系分组问题。

### 10. 路径压缩与按秩合并

并查集之所以高效，关键在于两个优化。

- 路径压缩：查询根结点时顺便压平路径。
- 按秩合并：让更矮的树挂到更高的树下。

这两个优化配合后，实际效率通常非常好。

### 11. 并查集 C++ 示例

```cpp
#include <iostream>
#include <vector>

class DisjointSetUnion {
private:
	std::vector<int> parent;
	std::vector<int> rank;

public:
	explicit DisjointSetUnion(int n) : parent(n), rank(n, 0) {
		for (int i = 0; i < n; ++i) {
			parent[i] = i;
		}
	}

	int find(int x) {
		if (parent[x] != x) {
			parent[x] = find(parent[x]);
		}
		return parent[x];
	}

	void unite(int x, int y) {
		int rootX = find(x);
		int rootY = find(y);
		if (rootX == rootY) {
			return;
		}

		if (rank[rootX] < rank[rootY]) {
			parent[rootX] = rootY;
		} else if (rank[rootX] > rank[rootY]) {
			parent[rootY] = rootX;
		} else {
			parent[rootY] = rootX;
			++rank[rootX];
		}
	}

	bool isConnected(int x, int y) {
		return find(x) == find(y);
	}
};
```

### 12. 这段并查集代码说明了什么

它说明并查集的关键不是“保存所有成员列表”，而是通过父指针维护集合代表元。

只要能快速找到代表元，就能快速判断是否属于同一集合。

### 13. 本章小结

哈希表和并查集都不属于最直观的“线性或树形结构”，但它们在实际问题里非常高频。

它们共同体现了一点。

有时候高效解题的关键，并不是更复杂的遍历，而是换一个更适合问题本质的数据组织方式。

## 十二、图与图的存储方式

### 1. 图的定义

图是一种用来描述多对多关系的数据结构。

它通常由两部分组成。

- 顶点集合。
- 边集合。

和树相比，图更加一般化。

树可以看作一种特殊的图。

### 2. 图适合描述什么问题

图几乎是计算机科学中最有表现力的结构之一。

很多问题都可以抽象为图。

例如。

- 地图导航。
- 社交关系网络。
- 课程依赖关系。
- 电路连接关系。
- 网络路由。

### 3. 有向图与无向图

如果边有方向，称为有向图。

例如课程先修关系，A 必须先于 B，就有明显方向。

如果边没有方向，称为无向图。

例如朋友圈双向好友关系，更适合用无向图描述。

### 4. 带权图与无权图

如果边带有权值，比如距离、时间、费用，就称为带权图。

如果边只表示是否连接，不带额外代价，就可视作无权图。

### 5. 图中的常见术语

学习图时常见概念包括。

- 顶点。
- 边。
- 路径。
- 环。
- 连通图。
- 连通分量。
- 入度与出度。
- 邻接点。

### 6. 邻接矩阵

邻接矩阵是用二维数组表示图。

如果有 $n$ 个顶点，就开一个 $n \times n$ 的矩阵。

矩阵中某个位置表示两个顶点之间是否有边，或边权是多少。

优点是。

- 判断两点是否直接相连很快。
- 实现直观。

缺点是。

- 空间复杂度通常为 $O(n^2)$。
- 对稀疏图不够节省空间。

### 7. 邻接表

邻接表是为每个顶点维护一个“相邻顶点列表”。

它对稀疏图更高效。

因为只存实际存在的边。

### 8. 邻接矩阵与邻接表的比较

| 方面 | 邻接矩阵 | 邻接表 |
|------|----------|--------|
| 空间 | 通常 $O(n^2)$ | 通常 $O(n + m)$ |
| 查一条边是否存在 | 快 | 需遍历相邻边 |
| 遍历所有邻居 | 一般 | 更方便 |
| 适合场景 | 稠密图 | 稀疏图 |

### 9. 图的遍历

图的遍历和树有些相似，但由于图可能有环，所以必须小心重复访问。

最经典的两种遍历方式是。

- 深度优先搜索 DFS。
- 广度优先搜索 BFS。

### 10. 深度优先搜索 DFS

DFS 的核心思想是。

从一个顶点出发，尽可能深地往下走，走不动再回退。

它常用递归或显式栈实现。

### 11. 广度优先搜索 BFS

BFS 的核心思想是。

从一个顶点出发，一层一层向外扩展。

它通常依赖队列实现。

### 12. DFS 与 BFS 的区别

DFS 更像“先走到尽头再回头”。

BFS 更像“先把离起点近的都看完，再看更远的”。

因此。

- BFS 很适合求无权图最短路。
- DFS 更适合做连通块、拓扑相关搜索、回溯遍历等。

### 13. 邻接表 + DFS/BFS 的 C++ 示例

```cpp
#include <iostream>
#include <vector>
#include <queue>

class Graph {
private:
	int n;
	std::vector<std::vector<int>> adj;

public:
	explicit Graph(int vertices) : n(vertices), adj(vertices) {}

	void addEdge(int u, int v) {
		adj[u].push_back(v);
		adj[v].push_back(u);
	}

	void dfs(int start) {
		std::vector<bool> visited(n, false);
		dfsHelper(start, visited);
		std::cout << '\n';
	}

	void bfs(int start) {
		std::vector<bool> visited(n, false);
		std::queue<int> q;

		visited[start] = true;
		q.push(start);

		while (!q.empty()) {
			int u = q.front();
			q.pop();
			std::cout << u << " ";

			for (int v : adj[u]) {
				if (!visited[v]) {
					visited[v] = true;
					q.push(v);
				}
			}
		}
		std::cout << '\n';
	}

private:
	void dfsHelper(int u, std::vector<bool>& visited) {
		visited[u] = true;
		std::cout << u << " ";

		for (int v : adj[u]) {
			if (!visited[v]) {
				dfsHelper(v, visited);
			}
		}
	}
};
```

### 14. 图这一章最重要的建模意识

很多同学学图时会被术语吓住。

其实真正关键的是先问自己一句话。

“这个问题中的对象之间，是否存在多对多关系？”

如果答案是“是”，图通常就值得考虑。

### 15. 本章小结

图是表达关系网络的核心结构。

学好图，不只是学几个遍历模板，更重要的是学会把现实中的关系、路径、依赖和连通问题抽象成图模型。

## 十三、查找算法

### 1. 查找问题是什么

查找，是在数据集合中确定某个元素是否存在，或者找到某个键对应位置的过程。

这听起来很简单，但它是最基础也最重要的计算问题之一。

很多复杂系统的核心，都可以还原成查找问题。

例如。

- 在学生名单中找某个学号。
- 在数据库索引中找某条记录。
- 在字典中找某个单词。
- 在缓存中找某个键对应的值。

### 2. 为什么查找方法不止一种

因为数据的组织方式不同。

如果数据是无序数组，查找方法会比较有限。

如果数据是有序数组，二分查找就能发挥作用。

如果数据存在哈希表中，查找方式又不同。

所以查找算法从来不是孤立的，它总是和数据结构绑定在一起。

### 3. 顺序查找

顺序查找是最直接的方法。

从头到尾依次比较，直到找到目标或遍历结束。

它的优点是。

- 实现简单。
- 不要求数据有序。
- 对结构要求低。

缺点是效率一般。

最坏情况下需要看完全部元素，因此时间复杂度通常为 $O(n)$。

### 4. 顺序查找适合什么场景

顺序查找并不是没有价值。

当下面这些条件成立时，它仍然很实用。

- 数据量较小。
- 查找不频繁。
- 数据结构不支持高效索引。
- 一次遍历中顺便就能完成比较。

### 5. 折半查找，也就是二分查找

二分查找是查找算法中最经典的方法之一。

它的核心思想是：每次比较中间位置元素，根据大小关系把搜索范围缩小一半。

### 6. 二分查找的前提条件

二分查找并不是对所有数据都能直接使用。

它最重要的前提是。

- 数据必须有序。

如果数据无序，就不能直接根据中间值判断目标位于左半边还是右半边。

### 7. 二分查找为什么快

因为它每次都把问题规模缩小一半。

所以查找次数不是线性增长，而是对数增长。

这就是它复杂度通常为 $O(\log n)$ 的原因。

### 8. 二分查找的常见错误

很多同学第一次写二分查找时容易出错。

常见错误包括。

- `left` 和 `right` 边界更新错误。
- 死循环。
- 中间位置计算溢出。
- 没处理目标不存在的情况。

### 9. 二分查找 C++ 示例

```cpp
#include <vector>

int binarySearch(const std::vector<int>& nums, int target) {
	int left = 0;
	int right = static_cast<int>(nums.size()) - 1;

	while (left <= right) {
		int mid = left + (right - left) / 2;

		if (nums[mid] == target) {
			return mid;
		}
		if (nums[mid] < target) {
			left = mid + 1;
		} else {
			right = mid - 1;
		}
	}

	return -1;
}
```

### 10. 为什么 `mid` 常写成 `left + (right - left) / 2`

这是为了避免 `left + right` 直接相加时可能发生整数溢出。

虽然在很多小范围数据里看不出区别，但这是良好的实现习惯。

### 11. 二分查找不仅能查“是否存在”

它还常用于。

- 查找第一个大于等于某值的位置。
- 查找最后一个小于等于某值的位置。
- 在答案空间上做判定，也就是“二分答案”。

这说明二分查找不是单一模板，而是一种重要的思想。

### 12. 二叉搜索树查找

如果数据放在二叉搜索树中，查找过程和二分查找有相似之处。

都在利用“有序性”缩小搜索范围。

区别在于。

- 二分查找常基于有序数组。
- BST 查找基于树结构。

### 13. 哈希查找

如果数据放在哈希表中，查找通常可在平均情况下达到接近 $O(1)$。

这比线性查找和树查找都更快。

但它也有前提和代价。

- 需要设计合理哈希函数。
- 需要处理冲突。
- 不天然支持有序遍历。

### 14. 几类查找方法对比

| 查找方法 | 典型前提 | 平均复杂度 | 适合场景 |
|----------|----------|------------|----------|
| 顺序查找 | 无特殊前提 | $O(n)$ | 小规模、无序数据 |
| 二分查找 | 数据有序 | $O(\log n)$ | 静态有序数据 |
| BST 查找 | 树结构较平衡 | 平均较优 | 动态有序集合 |
| 哈希查找 | 良好哈希设计 | 平均接近 $O(1)$ | 高频查找、精确匹配 |

### 15. 查找问题的核心不是背模板

真正重要的是先判断。

- 数据是否有序。
- 数据是否经常变化。
- 是否需要范围查询。
- 是否只关心精确匹配。

不同需求决定不同查找结构。

### 16. 本章小结

查找算法看起来只是一个“找东西”的问题，但它几乎贯穿了所有软件系统。

掌握本章，最重要的是理解。

- 顺序查找为什么慢。
- 二分查找为什么快。
- 树查找和哈希查找分别适合什么场景。

## 十四、排序算法

### 1. 排序问题的定义

排序，就是把一组数据按照某种规则重新排列。

最常见的是按从小到大或从大到小排序。

排序看似基础，但它几乎是算法中最经典的一类问题。

### 2. 为什么排序这么重要

因为很多更复杂的问题，在排序之后会变得更容易。

例如。

- 去重。
- 区间合并。
- 二分查找。
- Top K 问题。
- 贪心策略选择。

很多时候，题目真正的第一步不是直接解，而是先想“要不要先排序”。

### 3. 内部排序与外部排序

本科阶段常先学习内部排序，也就是数据基本能够放进内存中完成排序。

外部排序则更多面向海量数据，需要借助磁盘和分块处理。

初学阶段重点通常放在内部排序。

### 4. 稳定排序与不稳定排序

如果两个关键字相等的元素，排序后相对次序不变，则称排序算法稳定。

稳定性在很多业务场景里很重要。

例如先按班级排序，再按成绩排序时，如果第二次排序稳定，就能保留之前的某些次序信息。

### 5. 冒泡排序

冒泡排序的思想很直观。

相邻元素两两比较，把较大元素逐步“冒”到后面。

优点是容易理解。

缺点是效率不高，通常为 $O(n^2)$。

### 6. 选择排序

选择排序的思想是每一轮在剩余元素中找最小值，放到当前应放位置。

它同样直观，但复杂度通常也是 $O(n^2)$。

### 7. 插入排序

插入排序像整理扑克牌。

每次取一个新元素，把它插入到前面已经有序的部分中。

它在小规模数据或“基本有序”数据中表现往往不错。

### 8. 归并排序

归并排序是分治思想的经典代表。

它把数组不断二分，分别排序后再合并。

优点包括。

- 时间复杂度稳定为 $O(n \log n)$。
- 稳定排序。

缺点是通常需要额外辅助空间。

### 9. 快速排序

快速排序也是经典分治算法。

它通过选取一个基准值，把数组划分成两部分。

- 左边都不大于基准。
- 右边都不小于基准。

再递归处理左右两边。

它平均性能很好，工程上非常常见。

### 10. 快速排序的优点与风险

优点是。

- 平均复杂度好。
- 常数通常较小。
- 原地性较好。

风险是。

- 最坏情况下可能退化为 $O(n^2)$。
- 递归实现可能带来栈深度问题。

### 11. 堆排序

堆排序利用堆这一结构。

它先建堆，然后不断取出堆顶元素并调整堆。

堆排序的时间复杂度通常为 $O(n \log n)$，并且不依赖额外大块辅助数组。

### 12. 常见排序算法对比

| 排序算法 | 平均时间复杂度 | 最坏时间复杂度 | 空间复杂度 | 稳定性 |
|----------|----------------|----------------|------------|--------|
| 冒泡排序 | $O(n^2)$ | $O(n^2)$ | $O(1)$ | 稳定 |
| 选择排序 | $O(n^2)$ | $O(n^2)$ | $O(1)$ | 不稳定 |
| 插入排序 | $O(n^2)$ | $O(n^2)$ | $O(1)$ | 稳定 |
| 归并排序 | $O(n \log n)$ | $O(n \log n)$ | $O(n)$ | 稳定 |
| 快速排序 | 平均 $O(n \log n)$ | $O(n^2)$ | 递归栈开销 | 不稳定 |
| 堆排序 | $O(n \log n)$ | $O(n \log n)$ | $O(1)$ | 不稳定 |

### 13. 归并排序 C++ 示例

```cpp
#include <vector>

void merge(std::vector<int>& nums, int left, int mid, int right) {
	std::vector<int> temp;
	int i = left;
	int j = mid + 1;

	while (i <= mid && j <= right) {
		if (nums[i] <= nums[j]) {
			temp.push_back(nums[i++]);
		} else {
			temp.push_back(nums[j++]);
		}
	}

	while (i <= mid) {
		temp.push_back(nums[i++]);
	}
	while (j <= right) {
		temp.push_back(nums[j++]);
	}

	for (int k = 0; k < static_cast<int>(temp.size()); ++k) {
		nums[left + k] = temp[k];
	}
}

void mergeSort(std::vector<int>& nums, int left, int right) {
	if (left >= right) {
		return;
	}
	int mid = left + (right - left) / 2;
	mergeSort(nums, left, mid);
	mergeSort(nums, mid + 1, right);
	merge(nums, left, mid, right);
}
```

### 14. 快速排序 C++ 示例

```cpp
#include <vector>

int partition(std::vector<int>& nums, int left, int right) {
	int pivot = nums[right];
	int i = left;

	for (int j = left; j < right; ++j) {
		if (nums[j] < pivot) {
			std::swap(nums[i], nums[j]);
			++i;
		}
	}
	std::swap(nums[i], nums[right]);
	return i;
}

void quickSort(std::vector<int>& nums, int left, int right) {
	if (left >= right) {
		return;
	}
	int pivotIndex = partition(nums, left, right);
	quickSort(nums, left, pivotIndex - 1);
	quickSort(nums, pivotIndex + 1, right);
}
```

### 15. 为什么工程里常直接用 `std::sort`

因为标准库实现通常经过充分优化，综合了性能、边界处理和工程可靠性。

但学习阶段仍然必须理解手写排序。

因为只有理解这些经典算法，才能知道它们为什么工作、什么时候适合用。

### 16. 本章小结

排序算法是理解复杂度、分治、稳定性和工程取舍的绝佳案例。

本科阶段最重要的不是把所有排序都背熟，而是理解。

- 为什么简单排序通常是 $O(n^2)$。
- 为什么归并和快排常见于更大规模数据。
- 稳定性和空间开销为什么重要。

## 十五、递归与分治

### 1. 什么是递归

递归，是函数直接或间接调用自身来解决问题的方法。

递归之所以自然，是因为很多问题本身就具有“同类子问题”的结构。

例如。

- 树的遍历。
- 归并排序。
- 快速排序。
- 汉诺塔。

### 2. 递归的三个关键要素

写递归时，最重要的三个问题是。

1. 递归边界是什么。
2. 当前问题如何分解为规模更小的同类问题。
3. 子问题结果如何组合回原问题答案。

如果这三个问题没想清楚，递归代码通常就会出错。

### 3. 递归为什么容易让人困惑

因为函数在调用自己时，程序会进入多层调用状态。

如果没有“分解问题”的视角，只盯着代码执行顺序，就容易迷失。

学习递归时，一个很重要的习惯是。

不要一开始就试图把每一层细节全部追踪清楚。

先看“当前函数职责”是否定义清楚。

### 4. 递归与调用栈

递归本质上依赖调用栈。

每进入一层递归，系统都会保存当前函数上下文。

如果递归层数太深，可能造成栈溢出。

所以递归不仅是思维问题，也是资源问题。

### 5. 分治法是什么

分治法的核心思想是。

- 分解：把原问题拆成若干规模更小、相互独立的子问题。
- 解决：递归求解这些子问题。
- 合并：把子问题的答案组合起来。

归并排序和快速排序就是分治法的典型代表。

### 6. 递归与分治的关系

递归是一种实现方式。

分治是一种问题求解思想。

很多分治算法用递归实现，但递归不一定都属于分治。

例如树的遍历是递归，但并不一定强调“独立子问题合并”的分治结构。

### 7. 递归什么时候特别适合

当问题具有下面这些特征时，递归往往自然。

- 问题定义本身具有自相似性。
- 可以清晰写出递推关系。
- 子问题规模明显缩小。
- 结果组合方式清晰。

### 8. 快速幂为什么是分治思想

要求 $a^n$ 时，如果每次乘一次，需要做很多次乘法。

但如果利用。

- 当 $n$ 为偶数，$a^n = (a^{n/2})^2$
- 当 $n$ 为奇数，$a^n = a \cdot a^{n-1}$

就能显著减少计算次数。

### 9. 快速幂 C++ 示例

```cpp
long long fastPower(long long a, long long n) {
	if (n == 0) {
		return 1;
	}
	long long half = fastPower(a, n / 2);
	if (n % 2 == 0) {
		return half * half;
	}
	return half * half * a;
}
```

### 10. 递归的优点

递归的优点通常是。

- 代码表达简洁。
- 更贴合问题结构。
- 在树、图、分治类问题中很自然。

### 11. 递归的缺点

缺点包括。

- 有函数调用开销。
- 容易栈溢出。
- 如果存在重复子问题而不记忆，会很低效。

### 12. 重复子问题为什么危险

例如朴素递归求斐波那契数列，会反复计算相同子问题。

这会导致指数级复杂度。

而一旦加入记忆化或改用动态规划，效率就会大幅提升。

### 13. 递归转迭代的意识

很多递归过程都可以改写为显式栈或循环。

这在工程中有时很重要。

因为迭代方式可以更好地控制空间开销，避免深递归风险。

### 14. 本章小结

学习递归与分治，核心不是把代码写得神秘，而是理解如何把一个大问题拆成更小、更同质的子问题。

只要把边界、递推和组合三件事想清楚，递归就会自然得多。

## 十六、贪心算法

### 1. 贪心算法的核心思想

贪心算法，是在每一步都做出当前看起来最优的选择，希望最终得到整体最优解。

它的魅力在于。

- 思路常常简洁。
- 实现通常高效。
- 在适合的问题上效果非常好。

### 2. 贪心为什么不总是正确

因为“局部最优”并不总能推出“全局最优”。

有些问题只看眼前最好，会错过整体更优解。

所以贪心法最关键的，不是会不会写，而是会不会判断是否适用。

### 3. 贪心算法常见适用条件

教材中常提两个关键词。

- 贪心选择性质。
- 最优子结构。

直观理解是。

- 当前局部决策不会破坏后续最优性。
- 整体最优解可以由子问题最优解组成。

### 4. 一个经典例子：区间调度

假设有很多活动，每个活动有开始时间和结束时间。

目标是选出最多个互不冲突的活动。

这个问题的经典贪心策略是。

- 每次优先选择结束时间最早的活动。

### 5. 为什么“结束时间最早”是合理的

因为它为后续活动留下了尽可能多的时间空间。

这类分析正是贪心正确性的核心。

### 6. 区间调度 C++ 示例

```cpp
#include <algorithm>
#include <iostream>
#include <vector>

struct Interval {
	int start;
	int end;
};

int maxNonOverlappingIntervals(std::vector<Interval>& intervals) {
	std::sort(intervals.begin(), intervals.end(), [](const Interval& a, const Interval& b) {
		return a.end < b.end;
	});

	int count = 0;
	int currentEnd = -1;

	for (const auto& interval : intervals) {
		if (interval.start >= currentEnd) {
			++count;
			currentEnd = interval.end;
		}
	}
	return count;
}
```

### 7. 找零钱问题为什么不能一概使用贪心

在一些货币系统中，每次优先取最大面值是正确的。

例如常见的人民币、美元面额设置通常适合这样做。

但如果面值体系特殊，贪心可能失败。

例如面值为 1、3、4，凑 6 时。

- 贪心可能选 4 + 1 + 1，共 3 枚。
- 最优解其实是 3 + 3，共 2 枚。

### 8. 哈夫曼编码为什么是贪心算法经典例子

因为它每次都选择权值最小的两个结点合并。

这个局部策略最终能够导向最优前缀编码树。

### 9. 如何判断一道题是否可能用贪心

可以先问自己。

- 当前这一步做出的局部最好选择，会不会限制后面得到更优解。
- 我能否给出一个交换论证或结构性证明，说明局部选择不会变差。

### 10. 本章小结

贪心算法的学习重点，不是模板，而是“正确性判断”。

如果不能证明局部最优能通向全局最优，就不能轻易使用贪心。

## 十七、动态规划

### 1. 动态规划是什么

动态规划，是通过把原问题拆成若干子问题，并保存子问题结果，避免重复计算，从而高效求解的一类方法。

它尤其适用于。

- 存在重叠子问题。
- 具有最优子结构。

的问题。

### 2. 为什么动态规划常让人觉得难

因为动态规划不像二分查找那样有非常固定的套路。

它更多依赖对问题结构的理解。

很多同学卡在动态规划，不是因为不会写循环，而是因为不会定义状态。

### 3. 动态规划的核心四问

做动态规划时，通常要想清楚四件事。

1. 状态是什么。
2. 状态转移方程是什么。
3. 初始条件是什么。
4. 计算顺序是什么。

### 4. 状态

状态，是对“子问题答案”的抽象表示。

例如在 0/1 背包问题中，可以把状态定义为。

- 前 $i$ 件物品中，容量不超过 $j$ 时的最大价值。

状态定义清楚后，动态规划就容易展开。

### 5. 状态转移

状态转移描述当前状态如何由更小的状态得到。

例如 0/1 背包里。

- 不选当前物品。
- 选当前物品。

两种决策共同决定当前最优值。

### 6. 记忆化搜索与自底向上

动态规划有两种常见实现风格。

- 记忆化搜索：递归 + 缓存。
- 自底向上：直接按状态顺序填表。

前者更贴近递归思维。

后者更常见于竞赛和工程实现。

### 7. 斐波那契数列是最简单的 DP 入门例子

朴素递归会反复求同样的子问题。

而动态规划只算一次，每次保存结果。

这很好地体现了“避免重复计算”的价值。

### 8. 0/1 背包问题

0/1 背包是动态规划的经典问题。

问题描述是。

- 有若干物品，每件物品有重量和价值。
- 背包容量有限。
- 每件物品只能选或不选一次。
- 目标是在不超过容量的前提下让总价值最大。

### 9. 0/1 背包状态转移思想

设 `dp[j]` 表示容量不超过 `j` 时的最大价值。

遍历每件物品时。

- 如果不选它，价值保持原样。
- 如果选它，就看剩余容量的最优值加上当前价值。

### 10. 0/1 背包 C++ 示例

```cpp
#include <algorithm>
#include <iostream>
#include <vector>

int knapsack01(const std::vector<int>& weight, const std::vector<int>& value, int capacity) {
	std::vector<int> dp(capacity + 1, 0);

	for (int i = 0; i < static_cast<int>(weight.size()); ++i) {
		for (int j = capacity; j >= weight[i]; --j) {
			dp[j] = std::max(dp[j], dp[j - weight[i]] + value[i]);
		}
	}
	return dp[capacity];
}
```

### 11. 为什么背包里容量循环要倒序

因为 0/1 背包要求每件物品最多只能使用一次。

如果正序更新，就可能在同一轮中重复使用当前物品。

倒序更新可以避免这个问题。

### 12. 最长公共子序列 LCS

LCS 也是本科阶段非常经典的动态规划问题。

它强调的是“子序列”，不是“子串”。

如果两个字符相等，就可以由更短前缀的最优值转移过来。

如果不等，就取删掉一边字符后的最优结果。

### 13. 动态规划和贪心的区别

两者都可能追求最优解。

但。

- 贪心每一步只做当前最优选择。
- 动态规划会系统比较子问题结果。

动态规划更稳，但通常代价更高。

### 14. 动态规划和分治的区别

分治常要求子问题相互独立。

动态规划则正好相反，它擅长处理有重叠子问题的场景。

### 15. 动态规划常见难点

最难的地方通常不在代码，而在建模。

常见难点包括。

- 不会定义状态。
- 不会写转移方程。
- 边界条件处理混乱。
- 不知道为什么循环顺序要这样写。

### 16. 动态规划学习建议

建议先从小问题入手。

- 斐波那契。
- 爬楼梯。
- 背包。
- 最长公共子序列。

等对状态与转移敏感之后，再做更复杂问题。

### 17. 本章小结

动态规划的核心不是“背表格”，而是学会把一个最优问题拆成一系列可复用的子问题。

真正理解了状态、转移和边界，动态规划就会越来越自然。

## 十八、回溯与搜索

### 1. 回溯法是什么

回溯法本质上是一种系统搜索方法。

它会按照某种规则逐步构造解。

一旦发现当前路径不可能得到合法解或最优解，就撤销上一步，换其他选择。

### 2. 为什么回溯像“在解空间树上搜索”

因为每做一次选择，都相当于在搜索树中向下一层扩展。

所有可能选择连起来，就形成一棵状态空间树。

回溯做的事，就是在这棵树上做深度优先搜索。

### 3. 回溯法适合什么问题

典型场景包括。

- 全排列。
- 子集生成。
- 组合枚举。
- N 皇后。
- 数独求解。

### 4. 回溯与暴力枚举的关系

回溯可以看作一种更有结构的暴力枚举。

它并没有神奇地消除所有可能性。

它的价值在于。

- 搜索过程更系统。
- 可以结合约束进行剪枝。

### 5. 剪枝为什么重要

如果不剪枝，很多回溯问题会爆炸式增长。

剪枝的作用是。

- 尽早发现不可能成功的路径。
- 不再继续向下搜索。

这往往能极大减少实际搜索量。

### 6. 全排列问题

给定一组元素，要求列出所有排列。

这是最经典的回溯问题之一。

每一步选择一个还没用过的元素加入当前路径。

当路径长度达到元素总数时，得到一个完整解。

如果输入是 `[1, 2, 3]`，那么搜索树可以粗略理解为。

```text
[]
├── [1]
│   ├── [1,2]
│   │   └── [1,2,3]
│   └── [1,3]
│       └── [1,3,2]
├── [2]
│   ├── [2,1]
│   │   └── [2,1,3]
│   └── [2,3]
│       └── [2,3,1]
└── [3]
    ├── [3,1]
    │   └── [3,1,2]
    └── [3,2]
        └── [3,2,1]
```

这棵树说明。

- 根结点表示“什么都还没选”。
- 每往下一层，表示多做一次选择。
- 到叶子结点时，就得到一个完整排列。

### 7. 回溯解决全排列时到底在做什么

以 `[1, 2, 3]` 为例，可以用“选、递归、撤销”三个动作来理解。

第一步，路径为空，三个元素都可选。

- 先选 `1`，路径变成 `[1]`。
- 继续递归，在剩余元素里选。

第二步，路径为 `[1]` 时，可选 `2` 和 `3`。

- 先选 `2`，路径变成 `[1, 2]`。
- 再选 `3`，得到 `[1, 2, 3]`。

第三步，得到一个完整解后，要撤销上一步。

- 把 `3` 移出路径。
- 返回上一层，尝试别的选择。

这个“撤销选择，再尝试别的可能”就是回溯名字的来源。

### 8. 全排列 C++ 示例

```cpp
#include <iostream>
#include <vector>

// nums: 原始输入数组
// used: 标记某个位置的元素是否已经放入当前排列
// path: 当前已经构造出的排列前缀
// result: 保存所有完整排列
void backtrack(std::vector<int>& nums,
			   std::vector<bool>& used,
			   std::vector<int>& path,
			   std::vector<std::vector<int>>& result) {
	// 当路径长度等于元素总数时，说明找到一个完整排列
	if (path.size() == nums.size()) {
		result.push_back(path);
		return;
	}

	for (int i = 0; i < static_cast<int>(nums.size()); ++i) {
		// 已经使用过的元素不能重复选择
		if (used[i]) {
			continue;
		}

		// 1. 做选择
		used[i] = true;
		path.push_back(nums[i]);

		// 2. 进入下一层搜索
		backtrack(nums, used, path, result);

		// 3. 撤销选择，恢复现场
		path.pop_back();
		used[i] = false;
	}
}

std::vector<std::vector<int>> permute(std::vector<int> nums) {
	std::vector<std::vector<int>> result;
	std::vector<int> path;
	std::vector<bool> used(nums.size(), false);
	backtrack(nums, used, path, result);
	return result;
}
```

### 9. 这段全排列代码应该怎么读

学习时不要只看代码表面。

建议把它拆成四部分理解。

- 搜索状态：`path`
- 可选元素：`used` 为 `false` 的位置
- 结束条件：`path.size() == nums.size()`
- 回退动作：`pop_back()` 和 `used[i] = false`

一旦这四件事看清，绝大多数基础回溯题都能套进这个框架。

### 10. 子集问题

除了全排列，子集问题也是经典回溯场景。

给定一个集合，要求列出它的所有子集。

例如 `[1, 2]` 的子集有。

- `[]`
- `[1]`
- `[2]`
- `[1, 2]`

和全排列不同，子集问题并不要求把所有元素都选完，而是在每个位置都面临“选或不选”的决策。

### 11. 子集问题的搜索思路

如果当前走到第 `index` 个元素，那么通常有两种选择。

- 选它。
- 不选它。

只要持续递归到数组末尾，就得到一个完整子集。

### 12. N 皇后为什么是回溯经典问题

N 皇后问题要求在 $n \times n$ 的棋盘上放置 $n$ 个皇后，使得任意两个皇后都不在同一行、同一列、同一对角线上。

这个问题非常适合用回溯来理解“约束 + 剪枝”。

因为。

- 每一行只放一个皇后。
- 某一行放置后，会限制后续很多格子不能用。

### 13. N 皇后中的剪枝思想

假设我们正在第 `row` 行选择列位置。

如果某一列已经有皇后，或者该位置和之前皇后处于同一对角线，就不需要继续往下搜索。

这就是剪枝。

它的意义在于。

- 不去尝试明显非法的分支。
- 把指数级搜索尽量缩小。

### 14. N 皇后 C++ 示例

```cpp
#include <iostream>
#include <string>
#include <vector>

class NQueensSolver {
private:
	std::vector<std::vector<std::string>> solutions;
	std::vector<std::string> board;
	std::vector<bool> colUsed;
	std::vector<bool> diag1Used;
	std::vector<bool> diag2Used;

	void dfs(int row, int n) {
		// 如果已经成功放完 n 行，说明找到一个合法方案
		if (row == n) {
			solutions.push_back(board);
			return;
		}

		for (int col = 0; col < n; ++col) {
			int diag1 = row - col + n - 1; // 主对角线编号
			int diag2 = row + col;         // 副对角线编号

			// 当前列或对角线已被占用，直接剪枝
			if (colUsed[col] || diag1Used[diag1] || diag2Used[diag2]) {
				continue;
			}

			// 1. 做选择：在 (row, col) 放一个皇后
			board[row][col] = 'Q';
			colUsed[col] = true;
			diag1Used[diag1] = true;
			diag2Used[diag2] = true;

			// 2. 递归到下一行
			dfs(row + 1, n);

			// 3. 撤销选择，恢复现场
			board[row][col] = '.';
			colUsed[col] = false;
			diag1Used[diag1] = false;
			diag2Used[diag2] = false;
		}
	}

public:
	std::vector<std::vector<std::string>> solve(int n) {
		solutions.clear();
		board.assign(n, std::string(n, '.'));
		colUsed.assign(n, false);
		diag1Used.assign(2 * n - 1, false);
		diag2Used.assign(2 * n - 1, false);
		dfs(0, n);
		return solutions;
	}
};
```

### 15. 这段 N 皇后代码体现了什么

它体现了回溯法的标准结构。

- 不是所有回溯问题都只靠一个 `path` 数组表示状态。
- 很多问题还需要额外辅助数组来快速判断约束。
- 剪枝条件越准确，搜索树缩得越明显。

在 N 皇后中，辅助结构就是。

- 列占用数组。
- 主对角线占用数组。
- 副对角线占用数组。

### 16. 回溯模板的通用写法

很多回溯题虽然表面不同，但内部框架相似。

可以抽象成下面这种思路。

1. 判断是否到达结束条件。
2. 枚举当前层所有可能选择。
3. 做选择。
4. 递归进入下一层。
5. 撤销选择。

### 17. 回溯和 DFS 的关系

很多回溯代码在形式上就是 DFS。

区别在于。

- DFS 更强调遍历图或树。
- 回溯更强调“构造解 + 撤销选择”。

如果只是遍历所有点并标记访问，重点通常在 DFS。

如果要不断试不同方案并恢复现场，重点通常在回溯。

### 18. 回溯和动态规划的区别

两者都可能处理搜索空间问题。

但。

- 回溯更常枚举可能解。
- 动态规划更强调子问题复用。

如果同一子问题会反复出现，动态规划可能更合适。

如果需要列举所有解，回溯通常更自然。

### 19. 回溯问题最常见的错误

常见错误包括。

- 结束条件写错，导致少解或多解。
- 撤销操作不完整，导致状态污染。
- 剪枝条件过强，误删合法解。
- 剪枝条件过弱，搜索效率太低。

### 20. 本章小结

回溯法的本质，是在搜索空间中有组织地试错。

学习回溯时，最重要的是始终盯住四件事。

- 当前状态是什么。
- 当前能做哪些选择。
- 什么时候得到合法答案。
- 撤销时要恢复哪些现场。

## 十九、图论经典算法

### 1. 图论算法为什么重要

图结构一旦建立起来，就会引出很多典型问题。

例如。

- 从一点到另一点的最短路径。
- 如何用最小代价把所有点连起来。
- 一个任务依赖图是否存在合法执行顺序。

这些问题在工程和理论中都非常常见。

### 2. 无权图最短路径与 BFS

如果图是无权图，或者每条边权都相同，那么从某个起点到其他点的最短路径长度，可以通过 BFS 求得。

因为 BFS 是按层扩展的，第一次到达某点时通常就是最短步数。

### 3. 一个无权图最短路示例

假设图如下。

```text
0 -- 1 -- 3
|    |
2 -- 4
```

如果从顶点 `0` 出发。

- 到 `1` 的最短距离是 1。
- 到 `2` 的最短距离是 1。
- 到 `4` 的最短距离是 2。
- 到 `3` 的最短距离是 2。

原因是 BFS 会先访问距离起点 1 条边的所有顶点，再访问距离起点 2 条边的顶点。

### 4. 无权图 BFS 最短路 C++ 示例

```cpp
#include <queue>
#include <vector>

std::vector<int> bfsShortestPath(int n, const std::vector<std::vector<int>>& graph, int source) {
	std::vector<int> dist(n, -1); // -1 表示当前还不可达
	std::queue<int> q;

	// 起点到自己的距离是 0
	dist[source] = 0;
	q.push(source);

	while (!q.empty()) {
		int u = q.front();
		q.pop();

		for (int v : graph[u]) {
			// 第一次到达某个点时，记录的就是最短距离
			if (dist[v] == -1) {
				dist[v] = dist[u] + 1;
				q.push(v);
			}
		}
	}

	return dist;
}
```

### 5. Dijkstra 算法

Dijkstra 是解决“单源最短路径”问题的经典算法。

它适用于。

- 边权非负。

核心思想是。

- 每次从当前未确定顶点中，选择距离起点最近的一个。
- 用它去更新其他顶点距离。

### 6. Dijkstra 为什么要求边权非负

因为它依赖“当前最小距离一旦确定，就不会再被更优路径更新”的性质。

如果存在负边，这个性质会被破坏。

### 7. Dijkstra 的执行过程应该如何理解

假设从起点 `s` 出发。

一开始。

- `dist[s] = 0`
- 其他点距离设为无穷大。

然后不断重复下面动作。

1. 选出当前距离最小的未确定顶点 `u`。
2. 认为 `u` 的最短距离已经确定。
3. 用 `u` 去尝试更新它的邻居。

这里的“更新邻居”常被称为松弛。

即判断。

$$
dist[u] + w(u, v) < dist[v]
$$

如果成立，就说明经过 `u` 到 `v` 更短，需要更新 `dist[v]`。

### 8. Dijkstra C++ 示例

```cpp
#include <iostream>
#include <queue>
#include <utility>
#include <vector>

const int INF = 1e9;

std::vector<int> dijkstra(int n,
					  const std::vector<std::vector<std::pair<int, int>>>& graph,
					  int source) {
	// dist[i] 表示起点 source 到顶点 i 的当前最短距离估计值
	std::vector<int> dist(n, INF);
	dist[source] = 0;

	using State = std::pair<int, int>; // {distance, node}
	std::priority_queue<State, std::vector<State>, std::greater<State>> pq;
	pq.push({0, source});

	while (!pq.empty()) {
		auto [currentDist, u] = pq.top();
		pq.pop();

		// 如果堆里这个状态已经过期，就跳过
		if (currentDist > dist[u]) {
			continue;
		}

		for (const auto& [v, weight] : graph[u]) {
			// 尝试用 u 去松弛邻居 v
			if (dist[u] + weight < dist[v]) {
				dist[v] = dist[u] + weight;
				pq.push({dist[v], v});
			}
		}
	}

	return dist;
}
```

### 9. 如何手动走一遍 Dijkstra

假设边权如下。

```text
0 --2--> 1
0 --5--> 2
1 --1--> 2
1 --3--> 3
2 --2--> 3
```

从 `0` 出发。

初始时。

- `dist[0] = 0`
- `dist[1] = INF`
- `dist[2] = INF`
- `dist[3] = INF`

第一次确定顶点 `0` 后。

- 更新 `1` 为 2。
- 更新 `2` 为 5。

第二次取最小的是 `1`。

- 通过 `1` 到 `2` 的距离变成 `2 + 1 = 3`，比原来 5 更短。
- 通过 `1` 到 `3` 的距离变成 `2 + 3 = 5`。

第三次取最小的是 `2`。

- 通过 `2` 到 `3` 的距离是 `3 + 2 = 5`。
- 和当前 `dist[3]` 相同，不变。

最终结果。

- `0 -> 1 = 2`
- `0 -> 2 = 3`
- `0 -> 3 = 5`

### 10. Floyd 算法的思想

Floyd 算法用于求所有点对之间最短路径。

它的思想是逐步考虑“是否允许经过某个中间点”，并不断更新路径。

时间复杂度通常为 $O(n^3)$，适合点数不太大的场景。

### 11. Floyd 的状态理解

可以把 `dist[i][j]` 理解为。

- 从 `i` 到 `j` 的当前最短路长度。

然后枚举中间点 `k`，判断。

$$
dist[i][k] + dist[k][j] < dist[i][j]
$$

如果成立，就说明“经过 `k` 更短”，需要更新。

### 12. 最小生成树

如果要在一个带权无向连通图中，用尽量小的总代价把所有顶点连接起来，就会得到最小生成树问题。

经典算法有。

- Prim。
- Kruskal。

### 13. Prim 和 Kruskal 的思路差异

这两个算法都能求最小生成树，但思路不同。

#### Prim

像“从一个点开始逐步向外扩展”，每次选连接已选集合和未选集合的最小边。

#### Kruskal

像“从全图角度挑边”，每次选当前最小且不会成环的边。

### 14. Kruskal 的核心思想

Kruskal 会把边按权值从小到大排序，然后尽量选当前最小且不会形成环的边。

这也是为什么它常和并查集配合使用。

### 15. Kruskal C++ 示例

```cpp
#include <algorithm>
#include <vector>

struct Edge {
	int u;
	int v;
	int w;
};

class DSU {
private:
	std::vector<int> parent;
	std::vector<int> rank;

public:
	explicit DSU(int n) : parent(n), rank(n, 0) {
		for (int i = 0; i < n; ++i) {
			parent[i] = i;
		}
	}

	int find(int x) {
		if (parent[x] != x) {
			parent[x] = find(parent[x]);
		}
		return parent[x];
	}

	bool unite(int x, int y) {
		int rootX = find(x);
		int rootY = find(y);

		// 已经属于同一集合，再连接就会成环
		if (rootX == rootY) {
			return false;
		}

		if (rank[rootX] < rank[rootY]) {
			parent[rootX] = rootY;
		} else if (rank[rootX] > rank[rootY]) {
			parent[rootY] = rootX;
		} else {
			parent[rootY] = rootX;
			++rank[rootX];
		}
		return true;
	}
};

int kruskal(int n, std::vector<Edge>& edges) {
	std::sort(edges.begin(), edges.end(), [](const Edge& a, const Edge& b) {
		return a.w < b.w;
	});

	DSU dsu(n);
	int totalWeight = 0;
	int usedEdges = 0;

	for (const Edge& edge : edges) {
		// 只有当这条边连接了两个不同连通块时，才加入生成树
		if (dsu.unite(edge.u, edge.v)) {
			totalWeight += edge.w;
			++usedEdges;

			// n 个点的生成树恰好需要 n - 1 条边
			if (usedEdges == n - 1) {
				break;
			}
		}
	}

	return totalWeight;
}
```

### 16. 为什么 Kruskal 要判断“是否成环”

因为生成树要求。

- 所有顶点连通。
- 不存在环。

如果选了一条已经连接同一连通块内部的边，就只会形成环，而不会让更多顶点被有效连接。

### 17. 拓扑排序

拓扑排序用于有向无环图，也就是 DAG。

它给出一种线性顺序，使得所有有向边都从前指向后。

典型应用包括。

- 课程依赖。
- 编译顺序。
- 任务调度。

### 18. 拓扑排序的直观理解

如果一门课程 `A -> B`，表示学 B 前必须先学 A。

那么在拓扑序中，A 一定要排在 B 前面。

如果图中存在环，比如。

- A 依赖 B。
- B 依赖 C。
- C 又依赖 A。

那就说明根本不存在合法学习顺序。

### 19. 入度法为什么能做拓扑排序

一个顶点入度为 0，说明当前没有任何前置依赖。

所以它可以先执行。

执行后，再把它对后继结点的影响移除，可能让新的顶点入度变为 0。

这就形成了一个不断“剥洋葱”的过程。

### 20. 拓扑排序 C++ 示例

```cpp
#include <queue>
#include <vector>

std::vector<int> topologicalSort(int n, const std::vector<std::vector<int>>& graph) {
	std::vector<int> indegree(n, 0);

	// 先统计每个顶点的入度
	for (int u = 0; u < n; ++u) {
		for (int v : graph[u]) {
			++indegree[v];
		}
	}

	std::queue<int> q;
	for (int i = 0; i < n; ++i) {
		// 入度为 0，说明当前没有前置依赖
		if (indegree[i] == 0) {
			q.push(i);
		}
	}

	std::vector<int> order;
	while (!q.empty()) {
		int u = q.front();
		q.pop();
		order.push_back(u);

		for (int v : graph[u]) {
			// 删除 u -> v 这条边的影响
			--indegree[v];
			if (indegree[v] == 0) {
				q.push(v);
			}
		}
	}

	return order;
}
```

### 21. 如何判断图中是否有环

在入度法拓扑排序中，如果最后输出顶点数量少于总顶点数，就说明图中有环。

因为有些顶点永远无法变成入度 0。

### 22. 图论算法的难点在哪里

很多时候难点不只是算法本身，而是建模。

例如题目中的。

- 点是什么。
- 边是什么。
- 权值表示什么。
- 是否有方向。
- 是否存在环。

这些前置判断如果错了，后面算法选对了也没用。

### 23. 图论学习时的一个建议

不要一上来就死记算法名字。

先问问题属于哪一类。

- 是找最短路。
- 是找最小生成树。
- 是找依赖顺序。
- 还是只做普通遍历和连通性判断。

问题分类清楚后，算法选择会容易很多。

### 24. 本章小结

图论算法常让人觉得内容很多，但其实主线很清晰。

- BFS 最短路解决“无权图里从哪到哪最近”。
- Dijkstra 解决“非负权图里从起点到其他点最省”。
- 最小生成树解决“怎么全部连起来最便宜”。
- 拓扑排序解决“有依赖时怎样安排顺序”。

## 二十、字符串经典算法

### 1. 子串、子序列、前缀、后缀

学习字符串算法前，必须先分清几个基本概念。

- 子串：连续的一段字符。
- 子序列：不要求连续，但相对顺序不变。
- 前缀：从开头开始的一段。
- 后缀：到结尾为止的一段。

这些概念如果混淆，后面很多题都会做错。

### 2. 为什么字符串问题经常不简单

因为字符串不是孤立字符堆积。

它常常包含大量重复结构和模式信息。

例如。

- 前缀和后缀可能重合。
- 某些片段可能重复出现。
- 匹配失败后，不一定要完全回到起点。

这些特点使得专门的字符串算法非常有价值。

### 3. 朴素字符串匹配为什么会慢

最直接的做法是。

- 从文本串某个起点开始比较。
- 一旦失配，就把模式串整体右移一位。
- 然后重新从模式串开头开始比较。

这个方法容易理解，但会重复比较很多已经比较过的字符。

例如文本串是 `aaaaab`，模式串是 `aaaab`。

前面许多 `a` 会被反复比较。

所以朴素算法在最坏情况下复杂度可达 $O(nm)$。

### 4. KMP 算法的核心问题

朴素匹配在失配时，往往把模式串整体右移一位重新开始。

这会浪费前面已经比较过的信息。

KMP 的核心，就是利用模式串自身的前后缀结构，在失配时避免无意义回退。

### 5. 前缀函数或 next 数组的直观意义

它表示。

对于模式串某个前缀，最长相等真前缀和真后缀的长度是多少。

这能告诉我们失配后，模式串可以从哪里继续尝试。

### 6. 什么是真前缀和真后缀

以字符串 `ababa` 为例。

- 前缀有 `a`、`ab`、`aba`、`abab`、`ababa`
- 后缀有 `a`、`ba`、`aba`、`baba`、`ababa`

其中“真前缀”不能等于整个串，“真后缀”也不能等于整个串。

所以。

- 真前缀有 `a`、`ab`、`aba`、`abab`
- 真后缀有 `a`、`ba`、`aba`、`baba`

它们中最长相等的是 `aba`，长度为 3。

### 7. LPS 数组如何手动理解

LPS 是 longest prefix suffix 的缩写。

设模式串为 `ababaca`。

我们按位置逐步看。

- `a` 的 LPS 是 0
- `ab` 的 LPS 是 0
- `aba` 的 LPS 是 1，因为公共真前后缀是 `a`
- `abab` 的 LPS 是 2，因为公共真前后缀是 `ab`
- `ababa` 的 LPS 是 3，因为公共真前后缀是 `aba`

所以这个过程不是在比较文本串，而是在分析模式串自己的结构。

### 8. KMP 匹配过程应该怎么理解

KMP 的匹配过程里，有两个指针。

- `i` 指向文本串。
- `j` 指向模式串。

如果匹配成功，就一起前进。

如果失配。

- 不是让 `i` 回退。
- 而是根据 `lps[j - 1]` 调整 `j`。

这意味着。

- 文本串已经比过的位置不必重来。
- 模式串利用自身结构继续尝试。

### 9. 一个 KMP 手工示例

假设文本串是 `ababcabcacbab`，模式串是 `abcac`。

当比较到一半发生失配时，KMP 不会把模式串重新挪回最开始，而是利用模式串前后缀重合信息，把模式串移动到一个“仍然可能匹配”的位置。

这也是它能把整体复杂度控制在 $O(n + m)$ 的关键。

### 10. KMP C++ 示例

```cpp
#include <string>
#include <vector>

std::vector<int> buildLPS(const std::string& pattern) {
	int m = static_cast<int>(pattern.size());
	std::vector<int> lps(m, 0);

	// len 表示当前已经匹配好的最长公共真前后缀长度
	int len = 0;
	for (int i = 1; i < m; ) {
		if (pattern[i] == pattern[len]) {
			// 当前字符也能延长已有公共前后缀
			lps[i] = ++len;
			++i;
		} else if (len > 0) {
			// 继续尝试更短的公共前后缀
			len = lps[len - 1];
		} else {
			// 完全无法延长，只能记为 0
			lps[i] = 0;
			++i;
		}
	}
	return lps;
}

int kmpSearch(const std::string& text, const std::string& pattern) {
	if (pattern.empty()) {
		return 0;
	}

	std::vector<int> lps = buildLPS(pattern);
	int i = 0;
	int j = 0;

	while (i < static_cast<int>(text.size())) {
		if (text[i] == pattern[j]) {
			// 当前字符匹配，双指针同时前进
			++i;
			++j;
			if (j == static_cast<int>(pattern.size())) {
				// 找到完整匹配，返回起始位置
				return i - j;
			}
		} else if (j > 0) {
			// 失配后，模式串根据 lps 数组回退
			j = lps[j - 1];
		} else {
			// 模式串已经在开头，只能让文本串继续前进
			++i;
		}
	}

	return -1;
}
```

### 11. 这段 KMP 代码要重点看什么

重点不是把代码背下来，而是看清两个阶段。

#### 第一阶段

先预处理模式串，得到 `lps` 数组。

#### 第二阶段

用 `lps` 数组指导匹配过程中的回退。

如果这两个阶段的职责分不清，就很容易觉得 KMP 很绕。

### 12. KMP 常见错误

常见错误包括。

- `lps` 构造时下标写错。
- 失配时把 `i` 和 `j` 一起回退。
- 忘记处理空模式串。
- 匹配成功后不知道是否继续找下一个出现位置。

### 13. Trie 字典树

Trie 是一种专门处理字符串集合前缀关系的数据结构。

它特别适合。

- 字典存储。
- 前缀查询。
- 搜索建议。
- 自动补全。

### 14. Trie 的核心思想

它把字符逐层展开。

从根到某个结点的路径，表示一个前缀。

如果某条路径终点被标记为单词结束，那么就表示一个完整单词存在。

### 15. 用例子理解 Trie

假设插入三个单词。

- `cat`
- `car`
- `dog`

那么。

- `cat` 和 `car` 会共享前缀 `ca`
- `dog` 会走另一条分支

这说明 Trie 的优势在于。

- 公共前缀只存一份
- 前缀查询非常自然

### 16. 简化 Trie C++ 示例

```cpp
#include <array>
#include <string>

struct TrieNode {
	std::array<TrieNode*, 26> children{};
	bool isEnd;

	TrieNode() : isEnd(false) {
		children.fill(nullptr);
	}
};

class Trie {
private:
	TrieNode* root;

	const TrieNode* findPrefixNode(const std::string& word) const {
		const TrieNode* node = root;
		for (char ch : word) {
			int index = ch - 'a';
			if (node->children[index] == nullptr) {
				return nullptr;
			}
			node = node->children[index];
		}
		return node;
	}

public:
	Trie() : root(new TrieNode()) {}

	void insert(const std::string& word) {
		TrieNode* node = root;
		for (char ch : word) {
			int index = ch - 'a';
			// 如果当前字符对应的结点不存在，就新建
			if (node->children[index] == nullptr) {
				node->children[index] = new TrieNode();
			}
			node = node->children[index];
		}
		// 标记一个完整单词在这里结束
		node->isEnd = true;
	}

	bool search(const std::string& word) const {
		const TrieNode* node = findPrefixNode(word);
		return node != nullptr && node->isEnd;
	}

	bool startsWith(const std::string& prefix) const {
		return findPrefixNode(prefix) != nullptr;
	}
};
```

### 17. Trie 代码中的核心点

看 Trie 代码时，要抓住两个判断。

- 路径是否存在。
- 当前结点是否是完整单词结尾。

很多同学会把“前缀存在”和“单词存在”混为一谈。

例如。

- 如果字典里只有 `apple`
- 那么 `app` 的前缀存在
- 但 `app` 未必是完整单词

### 18. Trie 的局限性

Trie 的查询通常很快，但也有代价。

- 指针较多，空间开销可能较大。
- 字符集一大，实现会更复杂。
- 如果字符串很少，未必比哈希表更划算。

### 19. 字符串哈希的基本思想

字符串哈希通常是把一个字符串映射成一个数值，以便快速比较某些子串是否相等。

它常用于。

- 快速子串比较。
- 查重。
- 模式匹配优化。

不过哈希会有碰撞风险，因此使用时要理解概率与安全性边界。

### 20. 为什么前缀哈希能快速比较子串

如果我们先预处理出前缀哈希，就能在 $O(1)$ 或均摊很小的代价内取出任意子串的哈希值。

这样就不必每次都逐字符比较。

这在下面几类场景中尤其常见。

- 判断两个子串是否相同。
- 二分答案时做字符串比较。
- 回文或重复子串问题的快速判定。

### 21. 滚动哈希的公式直观理解

假设使用进制 `base`。

字符串 `s = s_0 s_1 ... s_{n-1}` 的哈希可以看成。

$$
H(s) = s_0 \cdot base^{n-1} + s_1 \cdot base^{n-2} + \cdots + s_{n-1}
$$

如果提前算好前缀哈希和幂次，就能快速截取任意区间哈希。

### 22. 前缀哈希 C++ 示例

```cpp
#include <string>
#include <vector>

class StringHash {
private:
	static const unsigned long long BASE = 131;
	std::vector<unsigned long long> prefix;
	std::vector<unsigned long long> power;

public:
	explicit StringHash(const std::string& text) {
		int n = static_cast<int>(text.size());
		prefix.assign(n + 1, 0);
		power.assign(n + 1, 1);

		for (int i = 0; i < n; ++i) {
			// prefix[i + 1] 保存前 i + 1 个字符的哈希值
			prefix[i + 1] = prefix[i] * BASE + static_cast<unsigned long long>(text[i]);
			power[i + 1] = power[i] * BASE;
		}
	}

	unsigned long long getHash(int left, int right) const {
		// 返回闭区间 [left, right] 的哈希值
		return prefix[right + 1] - prefix[left] * power[right - left + 1];
	}
};
```

### 23. 使用字符串哈希时要注意什么

虽然哈希比较很快，但“哈希值相等”并不严格等于“字符串一定相等”。

因此在严谨场景下常见做法是。

- 使用双哈希降低碰撞概率。
- 在关键步骤上再做一次原串校验。

### 24. 回文字符串问题为什么经典

字符串中还有一类非常经典的问题，就是回文。

回文的特点是正着读和反着读一样。

典型问题包括。

- 判断某个子串是否为回文。
- 求最长回文子串。
- 统计回文子串数量。

### 25. 最长回文子串的基础思路

本科阶段最容易掌握的做法是中心扩展。

因为每个回文都可以围绕某个“中心”向两边扩展。

中心有两类。

- 单个字符为中心，例如 `aba`
- 两个字符中间为中心，例如 `abba`

### 26. 中心扩展法 C++ 示例

```cpp
#include <string>

std::string longestPalindrome(const std::string& s) {
	if (s.empty()) {
		return "";
	}

	int start = 0;
	int maxLength = 1;

	auto expand = [&](int left, int right) {
		while (left >= 0 && right < static_cast<int>(s.size()) && s[left] == s[right]) {
			int currentLength = right - left + 1;
			if (currentLength > maxLength) {
				maxLength = currentLength;
				start = left;
			}
			--left;
			++right;
		}
	};

	for (int i = 0; i < static_cast<int>(s.size()); ++i) {
		// 以 s[i] 为中心，处理奇数长度回文
		expand(i, i);
		// 以 i 和 i + 1 之间为中心，处理偶数长度回文
		expand(i, i + 1);
	}

	return s.substr(start, maxLength);
}
```

### 27. Manacher 算法可以解决什么

如果要在线性时间内求最长回文子串，可以学习 Manacher 算法。

它的核心思想也是利用回文结构的对称性，避免重复扩展。

本科阶段先理解“为什么普通中心扩展是 $O(n^2)$，而 Manacher 能优化到 $O(n)$”就已经很有价值。

### 28. 本章小结

字符串算法看起来零散，实际上有几条非常清晰的主线。

- KMP 利用模式串自身结构解决匹配问题。
- Trie 利用公共前缀组织字符串集合。
- 字符串哈希利用数值映射做快速比较。
- 回文算法利用左右对称性质减少重复判断。

学习这一章时，不要只记住代码名字。

更重要的是看清每类算法到底利用了字符串的哪一种结构信息。

字符串算法的重点，在于学会利用字符串自身的结构信息。

KMP 利用前后缀关系。

Trie 利用公共前缀结构。

字符串哈希则利用数值映射做快速比较。

## 二十一、常见算法设计思想之间的比较

### 1. 为什么需要比较这些思想

很多同学学习算法时，容易把每种方法当成孤立章节。

结果遇到题目时，不知道该用哪一种。

真正重要的是建立“方法选择能力”。

### 2. 暴力枚举

暴力枚举的特点是直接尝试所有可能。

优点是。

- 思路直接。
- 容易验证正确性。

缺点是。

- 在规模大时往往不可接受。

### 3. 递归与分治

它们适合具有明显子问题结构的问题。

如果问题可以自然拆分，并且子问题和原问题形式相似，递归和分治通常值得优先考虑。

### 4. 贪心

贪心适合局部选择不会破坏全局最优的问题。

优点是简洁高效。

难点是正确性证明。

### 5. 动态规划

动态规划适合。

- 存在重叠子问题。
- 需要求最优值。
- 可以定义清晰状态转移。

### 6. 回溯

回溯适合需要枚举所有可行解或做约束搜索的问题。

它通常比纯暴力更有结构，但本质上仍在搜索空间里探索。

### 7. 图搜索

如果问题天然是“点与点关系网络”，那么 BFS、DFS、最短路、拓扑排序等图算法往往是更自然的工具。

### 8. 一个直观对比表

| 方法 | 适合问题 | 优点 | 风险 |
|------|----------|------|------|
| 暴力枚举 | 小规模、验证思路 | 简单直接 | 复杂度高 |
| 分治 | 子问题独立、结构相似 | 清晰高效 | 组合步骤可能复杂 |
| 贪心 | 局部最优可导向全局最优 | 简洁快 | 容易误判适用性 |
| 动态规划 | 最优子结构、重叠子问题 | 稳定可靠 | 建模难 |
| 回溯 | 搜索所有解、约束满足 | 通用性强 | 搜索空间爆炸 |
| 图算法 | 关系网络问题 | 建模能力强 | 图建模本身可能困难 |

### 9. 如何判断题目更像哪一类方法

可以先问自己几个问题。

1. 是否需要最优解。
2. 是否存在重复子问题。
3. 是否需要枚举所有方案。
4. 问题是否天然具有图结构。
5. 当前局部选择是否足以决定后续最优性。

### 10. 本章小结

学习各种算法思想的最终目的，不是把分类表背下来，而是遇到新问题时能快速建立初步判断。

这比单纯记住代码模板更重要。

## 二十二、C++ 中学习和实现数据结构与算法的建议

### 1. 为什么要同时会“手写”和“用 STL”

如果只会调用 STL，很容易把底层原理当黑盒。

如果只坚持手写所有结构，又会脱离实际工程效率。

更合理的方式是。

- 学习阶段手写经典结构和算法，理解原理。
- 工程实践中优先使用成熟标准库，实现更可靠、更高效。

### 2. 常见 STL 容器与数据结构对应关系

| STL 容器 | 常见对应思想 |
|----------|--------------|
| `vector` | 动态数组 |
| `list` | 双向链表 |
| `stack` | 栈适配器 |
| `queue` | 队列适配器 |
| `deque` | 双端队列 |
| `priority_queue` | 堆 / 优先队列 |
| `set` / `map` | 有序树结构 |
| `unordered_set` / `unordered_map` | 哈希表 |

### 3. `vector` 为什么最常用

因为它通常拥有。

- 良好的随机访问能力。
- 较好的缓存局部性。
- 动态扩容能力。

在很多不要求频繁中间插入删除的场景中，`vector` 往往是首选。

### 4. `list` 什么时候才值得用

虽然 `list` 提供链表语义，但现代工程中并不是特别常用。

因为它的缓存局部性较差，常数开销也不小。

只有当你确实需要频繁在已知位置做插入删除时，它才更有优势。

### 5. `priority_queue` 的默认行为

在 C++ 中，`priority_queue` 默认是大根堆。

如果你需要小根堆，通常要自己指定比较器。

### 6. `set` 和 `unordered_set` 的区别

- `set` 通常有序，底层常是平衡树。
- `unordered_set` 通常无序，底层常是哈希表。

因此。

- 想要顺序和范围操作，用 `set`。
- 想要平均更快的精确查找，用 `unordered_set`。

### 7. 模板和泛型的意义

数据结构与算法往往不只处理 `int`。

模板允许我们把“结构和操作逻辑”与“具体数据类型”分离。

这正是 C++ 泛型编程的重要价值。

### 8. 引用与 const 的使用习惯

在传递大对象，如 `vector`、`string` 时，常使用。

- `const std::vector<int>&`

这样可以避免不必要拷贝，又能保证函数不会修改参数。

这是写高质量 C++ 算法代码的重要习惯。

### 9. 迭代器

迭代器是 STL 非常核心的概念。

它相当于一种统一访问容器元素的方式。

理解迭代器，有助于你真正读懂标准库算法。

### 10. lambda 表达式

lambda 在排序、自定义比较、局部函数逻辑中非常常见。

例如。

```cpp
std::sort(nums.begin(), nums.end(), [](int a, int b) {
	return a < b;
});
```

### 11. 学习阶段最容易出现的 C++ 错误

常见问题包括。

- 数组越界。
- 迭代器失效。
- 忘记处理空容器。
- 指针悬空。
- 递归栈过深。
- 比较器写错导致排序逻辑异常。

### 12. 调试建议

写算法时，建议养成以下习惯。

- 先用最小样例测试。
- 再测边界样例。
- 手动画出关键变量变化。
- 对递归、DP、二分这类容易出边界错误的问题做逐步跟踪。

### 13. 本章小结

学习 C++ 数据结构与算法，不是为了炫技巧，而是为了把抽象算法思想写成清晰、可靠、可验证的代码。

掌握 STL 与手写实现之间的平衡，非常重要。

## 二十三、总结与进阶学习路径

### 1. 如何理解整门课的主线

如果把整门数据结构与算法课程压缩成一句话，可以这样理解。

它研究的是：如何组织数据，如何设计算法，以及如何在正确的前提下尽可能高效地解决问题。

### 2. 一张知识总图

```text
问题建模
   |
选择数据结构: 数组 / 链表 / 栈 / 队列 / 树 / 堆 / 哈希 / 图
   |
选择算法思想: 枚举 / 二分 / 排序 / 递归 / 分治 / 贪心 / 动态规划 / 回溯 / 图算法
   |
分析复杂度与正确性
   |
写出可靠的 C++ 实现
```

### 3. 本科阶段最应该掌握的核心能力

对本科生来说，最重要的能力通常不是“秒做难题”，而是下面这些。

- 能把问题抽象成合适的数据模型。
- 能分析基本复杂度。
- 能写出清晰可靠的实现。
- 能解释为什么自己的方法是正确的。

### 4. 推荐学习顺序

如果你要系统复习，可以按下面顺序进行。

1. 先掌握复杂度分析。
2. 再掌握数组、链表、栈、队列。
3. 然后学习树、堆、哈希、并查集。
4. 接着学习图及其遍历。
5. 再系统学习查找与排序。
6. 最后重点攻克递归、分治、贪心、动态规划、回溯和图论算法。

### 5. 如何把“会看懂”变成“会使用”

建议对每一类结构和算法至少做三件事。

- 自己手写一次基础实现。
- 自己分析一次复杂度。
- 自己举出一个真实或半真实场景说明它为什么有用。

### 6. 如何把“会做题”升级成“会分析”

遇到一道题时，尽量形成固定思路。

1. 输入规模多大。
2. 数据是否有序。
3. 是否需要最优解。
4. 是否存在重复子问题。
5. 是否需要枚举所有方案。
6. 数据之间关系更像线性、树形还是图。

这套思路比背模板更重要。

### 7. 学习中的常见瓶颈

常见瓶颈包括。

- 只会抄代码，不会解释为什么正确。
- 只记答案，不会自己建模。
- 动态规划和图论一遇到就慌。
- 代码能过样例，但边界情况经常出错。

这些问题都不是“天赋不够”，更多是训练方式不对。

### 8. 更有效的训练方法

建议采用下面的训练节奏。

- 先学理论。
- 再做小规模手推。
- 再写代码。
- 再复盘复杂度与错误点。

如果只是不断刷题而不复盘，提升会比较慢。

### 9. 最后总结

数据结构与算法的真正价值，不只是通过考试，也不只是通过面试。

它更深层的价值在于。

- 帮助你理解程序为什么这样组织。
- 帮助你理解性能为什么会这样变化。
- 帮助你面对新问题时，能有条理地分析和建模。

当你真正掌握这些能力后，很多原来看起来零散的知识点，就会逐渐连成体系。

### 10. 进阶建议

如果你还想继续深入，可以沿着下面几个方向扩展。

1. 深入学习平衡树、B 树、跳表等高级数据结构。
2. 深入学习最短路、网络流、强连通分量等更完整图论体系。
3. 系统学习字符串高级算法，如后缀数组、AC 自动机。
4. 继续训练动态规划建模与状态压缩能力。
5. 结合实际工程，观察数据库索引、缓存系统、调度器中的数据结构设计。
