# 第 3 天：项目整体管理与立项管理

## 一、今天的目标

第 3 天开始进入信息系统项目管理师最核心的主干内容之一：项目整体管理与立项管理。

如果说前两天解决的是“考试怎么考”和“项目背景像不像真实信息系统项目”，那么今天解决的是：

- 一个项目为什么能立起来
- 项目经理为什么有权管这个项目
- 项目到底按什么计划往前推进
- 项目出了变化以后到底怎么处理
- 项目结束时到底要收什么尾

完成今天后，你至少要达到以下状态：

- 能说清项目整体管理的主线，从立项到收尾串起来是什么顺序。
- 能区分项目建议、商业论证、可行性研究、项目章程、项目管理计划之间的关系。
- 能说清项目章程、项目管理计划、基准、变更、收尾之间的逻辑链条。
- 能识别整体管理类题目中常见的问题，例如没有章程、计划不完整、变更无审批、收尾不规范等。
- 能基于 day1 的主项目背景，写出一版“整体管理/变更管理”论文提纲。

今天的最低交付物有 4 项：

## 三、今天必须先了解的信息

### 1. 为什么整体管理是高级资格的主线模块

整体管理不是单独的一章知识，而是把项目其他管理活动串起来的“总线”。

在信息系统项目管理师考试里，它的重要性体现在三个层面：

- 综合知识会考概念和流程顺序。
- 案例分析会考项目有没有正式授权、有没有计划、变更是否规范、收尾是否完整。
- 论文会要求你把项目从启动、执行、监控到收尾讲出一条完整主线。

很多人后面学范围、进度、成本、风险时会觉得内容多而散，原因往往不是那些模块太难，而是没有先抓住整体管理这根主线。

### 2. 今天学习的重点不是死背流程名，而是抓住“项目为什么能管起来”

今天你要抓住下面 5 个问题：

1. 项目为什么要做，也就是为什么要立项。
2. 谁批准这个项目，谁授权项目经理。
3. 项目按什么计划执行。
4. 项目中途出现变化时怎么控制。
5. 项目结束时怎样正式关闭，而不是“做完就散”。

### 3. 今天和 day1、day2 的关系

- day1 让你明确考试规则，并建立了主项目背景。
- day2 让你的项目背景更像一个真实的信息系统项目。
- day3 开始，你要用这个项目背景进入“管理动作”层面。

也就是说，从今天开始，你在论文和案例题里不能只会写：

- 项目很复杂
- 技术很多
- 干系人很多

你要开始能写：

- 项目是如何被批准启动的
- 项目经理如何获得授权
- 项目管理计划如何形成
- 变更请求如何提出、分析、审批和落实
- 项目如何进行阶段收尾与最终收尾

## 四、今天要掌握的核心知识

## 1. 立项管理是什么

立项管理的核心不是“写几份材料”，而是论证项目是否值得做、能不能做、该不该现在做，并形成正式批准。

### 1. 立项管理要解决的问题

立项阶段主要解决：

- 这个项目为什么要做
- 这个项目是不是符合组织目标
- 这个项目从技术、经济、管理上是否可行
- 这个项目由谁投资、谁负责、做成什么样

### 2. 立项阶段常见成果

考试里不一定要求你死记所有文件名称，但至少要理解这些概念：

- 项目建议或需求来源
- 商业论证或业务需求说明
- 可行性研究
- 立项批复或批准决策
- 项目章程

### 3. 可行性研究的作用

可行性研究通常是为了回答：

- 技术上能不能实现
- 经济上是否值得投入
- 管理上和组织上能不能落地
- 进度上是否可控
- 风险是否在可接受范围内

不要把可行性研究理解成“形式化文件”。在案例题里，只要项目一开始目标不清、边界模糊、预算工期不现实，往往都能追溯到立项阶段论证不足。

## 2. 项目整体管理主线

项目整体管理可以理解为一条从启动到收尾的主线。今天你要把这条线牢牢记住：

1. 制定项目章程
2. 制定项目管理计划
3. 指导与管理项目工作
4. 管理项目知识
5. 监控项目工作
6. 实施整体变更控制
7. 结束项目或阶段

### 1. 这条主线怎么理解

你可以把它理解成一条管理闭环：

- 先获得授权
- 再形成计划
- 按计划执行
- 在执行中收集经验、监控偏差
- 对变化进行控制
- 最后正式收尾

### 2. 为什么叫“整体管理”

因为它不是只管一个专题，而是负责把范围、进度、成本、质量、资源、风险等各模块协调起来，保持项目目标一致、节奏一致、文件一致、行动一致。

## 3. 项目章程

### 1. 项目章程的作用

项目章程最核心的作用有两个：

- 正式批准项目启动
- 正式授权项目经理使用组织资源

只要没有项目章程或同等级正式授权文件，项目经理的管理动作就很容易缺乏依据。

### 2. 项目章程通常包含什么

项目章程通常会包含高层次内容，而不是详细执行细节，例如：

- 项目目标
- 项目范围概述
- 主要里程碑
- 初步预算
- 关键干系人
- 高层级风险
- 项目经理及其权限
- 项目批准人

### 3. 项目章程不是什么

项目章程不是：

- 详细进度计划
- 详细资源分配表
- 全部风险应对清单
- 详细测试方案

这些内容一般属于项目管理计划或其子计划层面。

### 4. 常见易混点

| 概念 | 核心作用 |
| --- | --- |
| 商业论证 | 说明为什么值得做 |
| 可行性研究 | 论证能不能做、值不值得做 |
| 项目章程 | 正式批准项目并授权项目经理 |
| 项目管理计划 | 说明项目具体怎么管、怎么做 |

## 4. 项目管理计划

### 1. 项目管理计划是什么

项目管理计划是指导项目执行、监控和收尾的主计划，它不是一张表，而是一套经过整合的管理安排。

### 2. 项目管理计划通常包括什么

项目管理计划通常由两部分构成：

- 基准
- 子管理计划

其中高频基准包括：

- 范围基准
- 进度基准
- 成本基准

常见子计划包括：

- 范围管理计划
- 进度管理计划
- 成本管理计划
- 质量管理计划
- 资源管理计划
- 沟通管理计划
- 风险管理计划
- 采购管理计划
- 干系人参与计划
- 变更管理计划
- 配置管理计划

### 3. 项目管理计划的关键特征

- 要经过整合，而不是各写各的
- 要经过批准，而不是草稿状态就直接执行
- 可以逐步细化，但不能随意变更

## 5. 基准是什么

基准可以理解为“经过批准、后续用于比较和控制的正式版本”。

考试里你至少要把下面这个关系记住：

- 范围基准：后续拿来比较范围有没有偏
- 进度基准：后续拿来比较项目有没有拖期
- 成本基准：后续拿来比较项目有没有超支

基准一旦形成，后面如果要改，通常不能口头改，而是要走变更控制。

## 6. 指导与管理项目工作

### 1. 核心含义

这一过程的核心不是“项目经理盯人干活”，而是按经批准的项目管理计划组织实施项目工作，协调资源，产生交付成果。

### 2. 常见活动

- 分配和协调资源
- 执行项目任务
- 开展沟通
- 实施批准变更
- 管理供应商与团队协作
- 输出交付成果

### 3. 容易丢分的点

很多案例题会把以下行为写成问题：

- 没有经批准计划就开始干
- 工作执行和计划脱节
- 变更未审批先实施
- 项目经理只顾推进进度，不更新文件和记录

## 7. 管理项目知识

这部分容易被忽视，但它既能出选择题，也能出案例分析表述点。

### 1. 管理项目知识做什么

- 利用现有经验支持当前项目
- 把当前项目中形成的经验沉淀下来
- 避免重复犯错

### 2. 常见输出

- 经验教训登记
- 知识沉淀材料
- 对组织过程资产的补充

### 3. 注意点

管理项目知识不是只在结项时做，而是贯穿项目全过程。

## 8. 监控项目工作

### 1. 核心目的

监控项目工作是把实际情况和计划进行比较，发现偏差、风险和问题，为管理决策提供依据。

### 2. 常见内容

- 采集工作执行数据
- 对比计划与实际
- 识别偏差和趋势
- 形成工作绩效信息或报告
- 为变更控制提供输入

### 3. 相关易混概念

| 概念 | 理解方式 |
| --- | --- |
| 工作绩效数据 | 原始观察结果，例如实际完成工时、实际成本 |
| 工作绩效信息 | 对数据加工后形成的偏差分析、趋势分析 |
| 工作绩效报告 | 面向汇报和管理使用的正式输出 |

## 9. 实施整体变更控制

### 1. 为什么整体变更控制很重要

项目里变化不可避免，但项目失败往往不是因为有变化，而是因为变化没有被控制。

### 2. 变更控制的核心目标

- 接收变更请求
- 分析变更影响
- 决定是否批准
- 跟踪批准后变更的落实情况
- 保证计划、基准和文件保持一致

### 3. 变更请求从哪来

变更请求可能来自：

- 业主或用户
- 项目经理
- 项目团队
- 供应商
- 质量检查或测试结果
- 风险应对需要

### 4. 常见变更类型

- 纠正措施：为纠正已经发生的偏差
- 预防措施：为了防止潜在问题发生
- 缺陷补救：修复已经发现的缺陷
- 更新：对计划或文件进行必要更新

### 5. 变更处理基本步骤

建议你记住下面这条通用主线：

1. 提出变更
2. 记录变更
3. 分析影响
4. 提交评审
5. 批准或拒绝
6. 实施批准变更
7. 更新计划、文件和基准
8. 跟踪验证结果

### 6. 变更控制委员会

变更控制委员会通常负责：

- 审查重要变更
- 评估影响
- 作出批准、否决或延期决策

项目经理不能因为需求方着急，就绕过正式变更流程直接改范围、改进度、改成本基准。

## 10. 结束项目或阶段

### 1. 为什么要正式收尾

项目做完不等于项目收尾完成。正式收尾要解决的是：

- 交付成果是否被正式接受
- 合同和采购事项是否处理完毕
- 资料是否归档
- 经验教训是否沉淀
- 团队和资源是否正式释放

### 2. 常见收尾输出

- 最终验收或正式确认
- 项目收尾文件
- 最终报告
- 经验教训
- 归档资料
- 资源释放记录

### 3. 容易忽略的点

- 项目提前终止也要正式收尾
- 阶段结束也要做阶段收尾
- 收尾不只是“交付上线”，还包括文档、合同、经验和资源关闭

## 11. 今天最容易混的概念

| 易混概念 | 正确区分 |
| --- | --- |
| 项目章程 vs 项目管理计划 | 一个负责授权启动，一个负责指导执行 |
| 监控项目工作 vs 实施整体变更控制 | 一个发现偏差和趋势，一个处理变更请求 |
| 工作绩效数据 vs 信息 vs 报告 | 原始数据、分析结果、正式汇报 |
| 纠正措施 vs 预防措施 vs 缺陷补救 | 已发生偏差、未发生问题、已发现缺陷 |
| 项目收尾 vs 阶段收尾 | 一个针对整个项目，一个针对阶段性成果 |
| 可行性研究 vs 项目章程 | 一个做决策论证，一个做正式批准 |

## 八、第 3 天案例题

### 1. 案例背景

某市高新区计划建设“智慧园区综合管理平台”，用于整合访客管理、门禁联动、停车管理、能耗监测、视频安防和领导驾驶舱等功能。项目总预算 980 万元，建设周期 14 个月，涉及建设单位、园区运营部门、物业公司、第三方硬件供应商、网络服务商和实施团队等多类干系人。

项目启动初期，建设单位领导在会议上口头表示“尽快推进”，项目团队随即开始需求调研和部分开发工作，但未形成正式项目章程。随后，项目经理组织各小组分别编写了需求、开发、测试、采购等工作安排，但没有形成统一的项目管理计划，也没有明确里程碑、基准和变更流程。

项目实施 3 个月后，建设单位提出新增“园区招商分析看板”功能，园区运营部门又提出需要增加“物业巡检移动端模块”，项目经理担心影响关系，要求开发团队先做，未进行正式影响分析和审批。与此同时，第三方硬件供应商通知门禁设备接口交付延期 1 个月，项目团队虽口头讨论过风险，但未更新计划，也未形成正式变更记录。

项目后期，平台主体功能虽然完成并上线试运行，但项目组仅组织了简单口头汇报，未形成正式验收资料、经验教训总结和项目关闭文件，部分关键文档也没有归档。

### 2. 问题

#### 问题 1

请指出该项目在整体管理与立项管理方面至少存在 5 个问题。

#### 问题 2

请说明该项目在项目启动和计划阶段至少缺少哪些关键成果或管理动作。

#### 问题 3

对于新增功能需求和外部接口延期，项目经理应如何按照整体变更控制思路进行处理？

#### 问题 4

项目上线试运行后，规范的项目收尾至少应包括哪些工作？

### 3. 参考答案要点

#### 问题 1 参考要点

可从以下角度作答，写出其中 5 点以上即可：

1. 未形成正式项目章程，项目缺乏正式授权依据。
2. 在没有正式授权和统一计划的情况下即启动实质性工作。
3. 各小组各写各的安排，未整合形成统一项目管理计划。
4. 未明确里程碑和基准，后续缺乏控制依据。
5. 未建立正式变更管理流程。
6. 新增需求未进行影响分析和审批即直接实施。
7. 外部接口延期未形成正式风险或变更记录。
8. 监控和变更结果没有回写到计划和文件中。
9. 上线后未进行正式验收和项目关闭。
10. 未沉淀经验教训、未归档关键文档。

#### 问题 2 参考要点

至少应提到以下内容中的若干项：

1. 形成并批准项目章程。
2. 明确项目目标、范围概述、主要里程碑、初步预算和高层级风险。
3. 明确项目经理角色及权限。
4. 整合形成经批准的项目管理计划。
5. 明确范围基准、进度基准、成本基准。
6. 明确变更管理机制和审批路径。
7. 明确沟通机制、风险管理安排和阶段控制点。

#### 问题 3 参考要点

答题建议按“提出、分析、审批、实施、更新、跟踪”六步来写：

1. 对新增功能需求形成正式变更请求并记录。
2. 分析其对范围、进度、成本、资源、质量和风险的影响。
3. 对接口延期形成风险和变更事项记录。
4. 提交给有权限的评审机制或变更控制委员会审查。
5. 对批准的变更更新项目管理计划、相关基准和项目文件。
6. 将批准结果通知相关干系人并组织实施。
7. 跟踪变更实施效果和接口延期应对结果。

#### 问题 4 参考要点

规范收尾至少应包括：

1. 对交付成果进行正式确认或验收。
2. 完成合同、采购、供应商相关收尾事项。
3. 形成项目最终报告。
4. 整理经验教训和知识沉淀。
5. 归档关键文档和管理记录。
6. 完成成果移交和运维交接。
7. 正式释放资源和关闭项目或阶段。

### 4. 案例题答题提醒

这类案例题不要写成散文，建议按下面方式表达：

- 问题一条一条写
- 原因一条一条写
- 措施按流程顺序写

如果题目问“存在什么问题”，就不要把大量篇幅写成概念定义；如果题目问“如何处理”，就要把动作词写出来，例如：记录、分析、评审、批准、更新、跟踪、归档。
