# 防病毒告警智能分析初步方案

## 一、场景定义与约束

你当前面对的问题，本质上不是“如何让大模型回答安全问题”，而是“如何在严格离线、算力受限、知识不足的前提下，把病毒告警研判流程做成可落地的智能分诊系统”。

这个场景有四个非常强的约束。

- 第一，模型运行全程必须在离线内网，不能依赖任何外部 API、云推理或在线知识查询。
- 第二，当前缺少足够成熟的“误报 / 真报”判定知识与标注数据，直接微调很容易把噪声学进去。
- 第三，内网环境可能主要依赖 CPU 推理，因此不能把方案建立在“大模型长上下文慢推理”上。
- 第四，防病毒告警存在明显的高风险漏报成本，因此系统设计不能只追求自动化率，更要重视保守性、可解释性和人工兜底。

因此，这个项目更适合定义为：

**构建一个离线的病毒告警智能分诊与研判助手，而不是一开始就做一个完全替代人工的自动处置系统。**

这个定位很重要。

如果一上来就要求模型直接给出最终处置结论，一旦出现漏报，风险会很高。

如果先把它定义为“高质量预筛选 + 结构化解释 + 风险升级建议”，项目更容易落地，也更容易在内网取得信任。

---

## 二、核心结论

针对你的约束，我的建议不是“只靠一个离线大模型微调后直接上生产”，而是采用下面这条路线。

### 1. 总体路线

推荐采用“规则与特征引擎 + 本地知识库检索 + 小型离线模型 + 人工反馈闭环”的混合架构。

这比纯大模型方案更适合你的原因有三点。

- 对知识不足更稳，因为可以先把显性经验固化成规则、模板和知识库，而不是全部押注在微调数据上。
- 对 CPU 更友好，因为只有少量疑难告警才进入 LLM 研判链路。
- 对安全场景更可靠，因为可以把低风险误报、高风险可疑、信息不足分开处理，而不是做激进的二分类。

### 2. 目标不要只做二分类

不建议模型直接只输出“误报 / 真报”。

更稳妥的输出应该至少分成四类。

- 明显误报
- 倾向误报但建议抽查
- 可疑待人工复核
- 高风险疑似真实威胁

如果条件允许，再增加一类。

- 信息不足，自动补采上下文

这样做的好处是，系统可以把“模型不确定”显式表达出来，避免为了追求自动化而强行给错结论。

### 3. 微调不是第一步，知识建设才是第一步

你目前最大的短板不是模型，而是领域知识资产不足。

所以第一阶段最重要的事情不是训一个模型，而是把下面这些东西沉淀出来。

- 什么样的告警经常是误报。
- 哪些字段对判定最关键。
- 分析员平时是按什么思路判断的。
- 哪些业务软件、脚本、驱动、运维工具经常触发误报。
- 哪些行为一旦出现，即便看起来像误报也不能自动放过。

换句话说，先做“知识蒸馏与知识扩展”，再做“参数微调”。

## 三、建议的总体架构

建议把整个系统拆成五层。

```text
告警输入层
	-> 日志归一化与特征提取层
	-> 快速规则 / 白名单 / 特征打分层
	-> 本地知识检索 + 小模型研判层
	-> 工单输出 / 人工复核 / 反馈回流层
```

### 1. 告警输入层

这一层负责接入现有防病毒系统产生的原始告警，并尽量补齐关联上下文。

建议采集的字段包括。

- 病毒名称、引擎名称、特征编号
- 文件路径、文件名、哈希值、文件大小
- 文件签名信息、发布者信息、证书是否可信
- 父进程、子进程、命令行参数
- 终端主机名、资产分级、所属部门、操作系统版本
- 首次出现时间、近 7 天出现频次、是否大范围爆发
- 是否来自统一软件发布、补丁升级、批量脚本执行
- 是否命中历史白名单、历史误报记录、历史处置工单
- 如有条件，再补充沙箱静态特征、动态行为摘要、网络连接摘要

这里有一个非常重要的工程原则。

**不要让 LLM 直接吃原始全量日志，而是先把日志转成结构化告警画像。**

原因很简单。

- 原始日志噪声大，模型容易被无关文本干扰。
- 结构化输入更节省上下文长度。
- 结构化字段更适合后续规则判断、特征建模和审计追溯。

### 2. 日志归一化与特征提取层

这一层是整个系统的地基。

如果这一层做不好，后面的知识库、模型、微调都只能建立在脏数据上。

建议优先做下面几类特征。

- 文件类特征：路径位置、扩展名、签名、哈希、是否压缩壳、是否临时目录
- 进程类特征：父子进程关系、是否由办公软件或脚本解释器拉起
- 资产类特征：是否关键服务器、是否生产终端、是否运维跳板机
- 时间类特征：是否批量同时触发、是否与软件发布窗口重合
- 历史类特征：相同哈希、相同路径、相同主机、相同签名是否多次被认定为误报
- 行为类特征：是否存在启动项、计划任务、注册表修改、异常外联、提权痕迹

建议同时生成一份标准化摘要，供后面的 LLM 使用，例如。

- 告警基本事实
- 关键证据
- 风险信号
- 误报信号
- 缺失上下文

### 3. 快速规则 / 白名单 / 特征打分层

这一层的目标不是取代人工，而是尽量用低成本方法先把明显场景筛掉。

可以分成三类能力。

#### 3.1 静态白名单

- 企业自研软件哈希白名单
- 合法签名发布者白名单
- 已批准运维工具白名单
- 已知安全更新包白名单

#### 3.2 专家规则

- 特定毒名在特定目录反复出现且无行为异常，倾向误报
- 来自批量软件分发系统且签名可信，倾向误报
- 出现在系统关键目录、脚本解释器拉起、伴随持久化行为，必须升级研判

#### 3.3 轻量模型或打分卡

在这一步，不一定非要上 LLM。

完全可以先训练一个轻量级分类器，例如基于结构化特征的逻辑回归、XGBoost 或 LightGBM，用来做第一道分流。

这是我对你原始思路的一个重要补充。

**在 CPU 场景下，最划算的往往不是让 LLM 处理所有告警，而是让轻量模型先处理 60% 到 80% 的高频重复场景，LLM 只接手边界样本。**

### 4. 本地知识检索 + 小模型研判层

当告警经过前面的规则层后，剩下的通常是更复杂的样本。

这一层才需要引入 LLM。

建议采用“检索增强生成”而不是“只靠模型参数记忆”。

研判链路可以设计成。

1. 根据告警画像检索本地知识库。
2. 取回相似历史案例、处置 SOP、误报模式、恶意行为特征、业务白名单解释。
3. 将检索结果和结构化告警摘要一起输入小模型。
4. 输出结构化结论、证据链、置信度和建议动作。

建议模型输出必须结构化，例如。

- 判定类别
- 置信度
- 主要证据
- 缺失证据
- 推荐动作
- 是否需要人工介入

### 5. 工单输出 / 人工复核 / 反馈回流层

这是决定系统能不能越做越好的关键层。

每一条模型输出，后面都要能接住人工反馈。

建议至少记录下面这些回流字段。

- 人工最终判定
- 人工修正原因
- 哪条知识命中有效
- 哪条规则误导了模型
- 哪类样本模型总是拿不准

这部分数据后面会直接变成你的蒸馏样本、微调样本和评测集。

## 四、离线环境下的部署原则

离线，不只是“模型文件放在内网”，而是整个生命周期都要按离线方式设计。

### 1. 推理、存储、训练都在内网闭环

建议做到下面几点。

- 推理服务部署在内网服务器，不调用外部 API。
- 向量库、知识库、日志库、工单库全部部署在本地。
- 模型下载、升级、知识包更新采用离线介质或单向导入机制。
- 对外部引入的公开安全知识做统一审批、脱毒、校验和版本管理。

### 2. 将“更新能力”设计成离线数据包机制

因为没有在线能力，所以知识更新和模型升级不能依赖随时联网。

建议做成固定节奏的数据包导入。

- 月度知识包：误报案例、恶意样本家族摘要、MITRE ATT&CK 映射、常见 IOC 模式
- 月度白名单包：企业软件新版本哈希、签名变更、发布系统更新记录
- 季度模型包：LoRA 权重、蒸馏后学生模型、提示模板升级

### 3. 模型服务与恶意样本环境隔离

如果需要对可疑文件做静态特征提取或沙箱分析，建议和 LLM 推理环境隔离。

原因在于。

- 防止恶意样本处理链影响模型服务稳定性。
- 便于做权限隔离和审计。
- 便于后续扩容和运维。

实践上可以把原始样本保存在受控分析区，而传递给 LLM 的只是“提炼后的摘要特征”，而不是整个恶意文件本身。

## 五、知识不足时，应该如何做知识蒸馏与扩展

你明确提到一个现实问题：目前没有充足的知识来判断一个病毒日志是不是误报。

这非常关键。

我建议把“知识蒸馏”理解成三件事，而不只是“大模型蒸馏小模型”。

### 1. 第一条线：从人工经验蒸馏到结构化知识

先把现有分析员脑中的隐性经验挖出来。

可以用非常务实的方式做。

- 访谈 3 到 5 名一线分析员，记录他们判断误报时最看重的字段。
- 复盘过去 3 到 6 个月的典型误报工单，抽取共性模式。
- 把口头经验整理成判定清单、决策树、问答模板和反例集。

例如，分析员经常会隐含使用下面这类经验。

- 这个毒名在补丁发布后大量出现，多半是误报。
- 这个文件路径位于企业软件安装目录，且签名可信，风险较低。
- 这个样本虽然毒名可疑，但全网大范围同时出现，更像引擎误判。
- 这个文件名看起来正常，但由脚本解释器拉起并伴随计划任务创建，必须升级处理。

这些经验先不要急着丢给模型训练。

先把它们变成。

- 规则
- 知识条目
- 标注规范
- 提示模板

这样价值最高。

### 2. 第二条线：从历史数据蒸馏到案例知识库

建议把历史告警和工单整理成“案例单元”。

每个案例单元最好包含下面这些字段。

- 原始告警摘要
- 结构化特征
- 人工最终结论
- 证据链
- 采取的动作
- 为什么不是另一种结论

这里尤其要强调“反例信息”。

如果只存“最后判定是误报”，而不存“为什么不是恶意”，后续模型学到的知识会很浅。

### 3. 第三条线：从离线教师模型蒸馏到小模型

如果后续你有条件在内网准备一台更强的训练节点，可以部署一个较大的开源教师模型，仅用于离线数据标注增强和蒸馏。

具体做法可以是。

1. 用较大教师模型阅读结构化告警和知识检索结果。
2. 让教师模型输出结构化解释、关键证据、反证、建议动作。
3. 由人工抽样审核高价值样本。
4. 将审核后的样本用于训练更小的学生模型。

这里要注意一个原则。

**教师模型也必须在离线环境运行。**

不要为了做蒸馏而把敏感告警数据拿去外部平台。

### 4. 第四条线：从公开安全知识做受控扩展

虽然运行必须离线，但知识来源不必只局限于你当前手头已有的内部材料。

你完全可以通过受控导入方式，把公开安全知识引入内网并进行二次加工。

建议优先导入下面几类资料。

- 主流杀软厂商公开的误报说明与样本说明
- MITRE ATT&CK、CAPEC、常见恶意行为模式说明
- Sigma 规则、YARA 规则的行为描述部分
- Windows 事件、PowerShell、脚本、计划任务等系统行为说明文档
- 常见打包器、压缩壳、运维工具、扫描工具、远控组件的特征说明

导入后不要原样堆进知识库。

应该转成适合本场景的知识形态。

- 告警模式说明
- 常见误报模式
- 常见真实威胁模式
- 字段解释字典
- 研判步骤 SOP
- 对比型样本对

### 5. 最关键的建议：先建“领域判定框架”，再建数据集

很多团队一上来就收集日志，然后急着标注。

这往往效率不高。

更好的顺序是先明确判定框架。

例如你可以先定义一套研判维度。

- 来源可信度
- 文件可信度
- 行为危险度
- 环境匹配度
- 历史重复度
- 横向扩散性
- 业务解释合理性

有了这套维度后，再去标注数据，后面的规则、知识库、模型和评测才会一致。

## 六、模型路线建议

### 1. 不建议一开始就做“大模型直接微调上生产”

原因主要有四个。

- 误报 / 真报知识不足，标签噪声会很大。
- 单纯监督微调容易学到表面模式，例如只记住某些毒名或路径。
- CPU 推理成本高，难以支撑全量告警直接研判。
- 安全场景需要可解释性和可审计性，单纯参数记忆不够透明。

### 2. 推荐采用三阶段模型路线

#### 阶段 A：无微调版本

先不上微调，只做下面几件事。

- 结构化特征提取
- 规则和轻量分类器
- 本地知识库检索
- 小模型按固定模板输出结构化结论

这是最适合快速验证价值的 MVP。

#### 阶段 B：轻量 LoRA 微调

当你积累了第一批高质量样本后，再做小规模领域微调。

推荐只做下面这类任务型微调。

- 结构化告警归因
- 误报模式解释
- 研判结论标准化输出
- 缺失证据识别

也就是说，让模型更会“按你的业务格式分析”，而不是妄图让它自己凭空记住全部安全知识。

#### 阶段 C：教师-学生蒸馏

当案例足够多、流程相对稳定后，再把大教师模型的分析能力蒸馏到更小的学生模型上，目标是提升 CPU 上的推理速度和稳定性。

### 3. 模型选择原则

因为你要考虑中文、安全术语和 CPU 部署，所以建议遵循下面几个原则。

- 优先选中文和结构化输出能力较好的开源指令模型。
- 参数规模优先考虑 3B 到 7B 量级，而不是更大的模型。
- 推理格式优先使用 GGUF 等 CPU 友好格式，并做 4-bit 或 5-bit 量化。
- 只在少量复杂样本上调用 LLM，不要让 LLM 成为全量入口。

可以参考的组件方向如下。

| 组件 | 建议方向 | 说明 |
| --- | --- | --- |
| 主研判 LLM | 3B 到 7B 级中文指令模型 | 兼顾中文理解与 CPU 推理 |
| Embedding 模型 | 小型中文向量模型 | 用于案例检索和知识召回 |
| 轻量分类器 | XGBoost / LightGBM / 逻辑回归 | 用于第一层分流 |
| 推理引擎 | llama.cpp 类 CPU 推理框架 | 适合量化模型离线部署 |
| 向量库 | FAISS / Qdrant 本地部署 | 根据运维复杂度选择 |

这里要强调一点。

**真正吃 CPU 的不是“有没有模型”，而是“每条告警送进去多少文本、调用多少次模型、有没有做前置分流”。**

## 七、CPU 场景下的性能优化建议

这是你的方案里必须重点考虑的一部分。

如果不做架构优化，只是简单把模型搬到 CPU 上，体验通常会很差。

### 1. 采用级联处理，而不是全量大模型直推

推荐处理顺序如下。

1. 去重与聚合。
2. 白名单和规则快速过滤。
3. 轻量模型初判。
4. 只有边界样本才进入 LLM。

这个顺序能大幅降低 LLM 的调用量。

### 2. 强制使用结构化短上下文

不要把全量原始日志、沙箱原文、工单原文一次性全部丢给模型。

建议把输入压缩成固定模板，例如。

- 基本事实 10 行以内
- 关键证据 5 条以内
- 检索知识 3 条以内
- 历史相似案例 2 条以内
- 明确问题 1 条

这样往往比长上下文堆料更有效，也更快。

### 3. 预计算和缓存

下面这些能力都可以预计算。

- 高频样本的向量
- 常见毒名的知识召回结果
- 已知误报模式的解释模板
- 重复告警的历史结论

如果同一哈希、同一路径、同一签名样本重复出现，尽量复用历史研判结果，而不是重新推理。

### 4. 把“分类”和“解释”拆开

如果性能很紧张，可以把任务拆成两段。

- 小模型或轻量分类器负责先给出风险层级。
- LLM 只对需要人工查看的样本生成解释。

这样做非常实用。

因为很多时候，真正需要长文本能力的是“解释”，不是“初步分类”。

### 5. 输出格式要受控

建议强制模型输出 JSON 或固定字段，避免自由生成过长文本。

这样有三个好处。

- 更快
- 更容易接系统
- 更方便评测和审计

### 6. 如果 CPU 特别弱，优先做“检索增强 + 分类器”，推迟微调

这是一个务实建议。

在很多内网场景里，真正最先落地的，不是“专家大模型”，而是下面这种组合。

- 一套高质量字段解析器
- 一套误报 / 真报规则库
- 一个轻量分类器
- 一个本地案例检索系统
- 一个仅在必要时触发的小模型解释器

这条路线更容易先把价值做出来。

---

## 八、推荐的系统输出格式

不建议输出一段自由文本就结束。

建议系统统一输出标准化结果，例如。

```json
{
	"judgement": "likely_false_positive",
	"confidence": 0.82,
	"risk_level": "medium",
	"key_evidence": [
		"文件位于企业软件安装目录",
		"数字签名可信且与已登记发布者一致",
		"同一哈希在过去30天内多次出现且均为误报"
	],
	"counter_evidence": [
		"父进程为powershell.exe，需要抽查脚本来源"
	],
	"missing_context": [
		"缺少该终端近24小时行为摘要"
	],
	"recommended_action": "加入人工抽查队列，不建议立即隔离主机",
	"need_human_review": true,
	"referenced_knowledge": [
		"历史案例#FP-0231",
		"误报规则#WHITELIST-SIGN-07"
	]
}
```

有了这样的输出格式，后面无论是接工单系统、看板统计还是人工复核，都会轻松很多。

## 九、评测体系怎么设计

安全场景不能只看一个准确率。

建议至少同时看下面几类指标。

### 1. 业务指标

- 人工处理量下降比例
- 高频误报自动归并比例
- 平均研判时长下降比例

### 2. 安全指标

- 高风险真实威胁漏报率
- 可疑样本升级人工的召回率
- 高置信误报结论的错误率

### 3. 模型指标

- 四分类准确率
- 置信度校准情况
- 结构化输出完整率
- 检索命中率

### 4. 系统指标

- 单条告警平均延迟
- CPU 占用
- 高频告警缓存命中率
- 每日可处理上限

在安全场景里，建议优先保证下面这个原则。

**宁可把拿不准的样本升级给人工，也不要为了追求自动化率而压低可疑样本召回。**

## 十、建议的实施路线图

### 阶段 0：两到四周，做数据盘点和判定框架

目标不是上模型，而是把基础打稳。

输出物建议包括。

- 告警字段字典
- 研判维度定义
- 误报 / 真报 / 待复核分类标准
- 典型误报模式清单
- 历史工单案例样本库

### 阶段 1：四到八周，做 MVP

这阶段建议只做最小可用闭环。

- 接入告警数据
- 完成归一化与特征提取
- 上线规则库和白名单
- 上线案例检索
- 上线小模型解释器
- 输出结构化结论到工单系统

这时候即便还没有微调，也已经能验证实际价值。

### 阶段 2：六到十周，做知识蒸馏和轻量微调

这阶段开始做样本沉淀和能力固化。

- 选取高价值样本做人工精标
- 构建教师样本和反例样本
- 进行 LoRA 微调
- 对边界样本做蒸馏和对比优化

### 阶段 3：持续迭代，做稳定化和规模化

- 引入月度知识包更新机制
- 引入模型版本管理和回滚机制
- 引入专家规则审批流程
- 引入告警聚类和家族级复用能力

---

## 十一、风险与应对

### 1. 最大风险不是模型不聪明，而是数据脏和标签乱

应对方式。

- 先统一判定标准
- 先做结构化字段清洗
- 先做小规模高质量样本，而不是一开始就追求海量数据

### 2. 最大安全风险是漏报

应对方式。

- 不做激进自动放行
- 高风险特征命中时强制升级人工
- 把“不确定”设计成合法输出类别

### 3. 最大性能风险是把所有样本都送给 LLM

应对方式。

- 先聚合、再过滤、后研判
- 优先结构化短输入
- 做缓存和历史复用

### 4. 最大知识风险是只依赖模型，不沉淀规则和案例

应对方式。

- 同步建设规则库、知识库、案例库
- 人工反馈必须可回流
- 每次误判都要定位是规则问题、知识问题还是模型问题

---

## 十二、我对这个项目的最终建议

如果把你的目标压缩成一句话，我建议这样落地。

**先做一个离线、CPU 友好、以规则和检索为主、以小模型解释为辅、以人工闭环为核心的病毒告警智能分诊系统；等知识资产和样本质量起来之后，再做专业化微调和蒸馏。**

这条路线有几个明显优势。

- 更符合你当前“离线 + 知识不足 + CPU 受限”的现实约束。
- 更容易先做出业务价值，而不是陷入长期训模却看不到效果。
- 更利于后续扩展，从误报研判逐步走向威胁归因、处置建议和自动化编排。

如果要给一个最务实的起步顺序，我建议是。

1. 先定义判定框架和输出格式。
2. 再建设案例库、规则库和知识库。
3. 先做检索增强和轻量分流。
4. 再引入小模型解释器。
5. 最后再做微调和蒸馏。

对你这个场景来说，这样的技术顺序，比“先找一个离线大模型直接微调”更稳，也更容易成功。
